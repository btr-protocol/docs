# 6. BTR Swap -Integration

> Two consumer profiles: **front-end (user mode)** and **vault keeper (keeper mode)**. Both hit the same back service; mode resolution is the only difference.

---

## 1. Front-End (User Mode)

The front-end (`front/`) calls the swap service via the wagmi/viem-friendly fetch wrapper. Mode defaults to `user` -no explicit declaration needed.

```ts
const res = await fetch(`${SWAP_URL}/quote`, {
  method: 'POST',
  headers: { 'content-type': 'application/json' },
  body: JSON.stringify({
    fromChain: 1,
    toChain:   10,
    fromToken: '0xa0b86991c6218b36c1d19d4a2e9eb0ce3606eb48', // USDC
    toToken:   '0x0b2c639c533813f4aa9d7837caf62653d097ff85', // USDC.e
    fromAmount: '1000000000',  // 1000 USDC (6 decimals)
    fromAddress: account.address,
    slippageBps: 50,
  }),
});
const { routes, best } = await res.json();
```

Then `/build` to lock in calldata and submit:

```ts
const buildRes = await fetch(`${SWAP_URL}/build`, {
  method: 'POST',
  headers: { 'content-type': 'application/json' },
  body: JSON.stringify({ ...quoteReq, preferAggId: best.aggId }),
});
const route = await buildRes.json();
const tx = await walletClient.sendTransaction({
  to:    route.to,
  data:  route.data,
  value: BigInt(route.value ?? '0'),
});
```

Rate-limit hint: cap UI quote refresh at **one quote per ≥ 5 seconds** to stay well below `RATIO_SOFT = 8` over a 5-minute window even if the user never executes.

## 2. Vault Keeper (Keeper Mode)

Keepers (vault rebalancer bots, e.g. `back/services/agents/`) call with `mode: 'keeper'` or via the `x-btr-mode: keeper` header. The service forces `KEEPER_MODE_AGGS = [LIFI]` (`back/services/swap/src/lib/config.ts:38`) and validates the returned `to` against the per-chain allowlist.

```ts
const res = await fetch(`${SWAP_URL}/build`, {
  method: 'POST',
  headers: {
    'content-type': 'application/json',
    'x-btr-mode':   'keeper',
  },
  body: JSON.stringify({
    fromChain: 42161,
    toChain:   42161,
    fromToken: '0xaf88d065e77c8cc2239327c5edb3a432268e5831', // USDC arb
    toToken:   '0xfd086bc7cd5c481dcc9c85ebe478a1c0b69fcbb9', // USDT arb
    fromAmount: '1000000000',
    fromAddress: VAULT_ADDR,
    slippageBps: 30,
  }),
});

if (!res.ok) {
  // 422 keeper calldata rejected → upstream routed to non-allowlisted contract
  // 429 rate-limited → retry after Retry-After seconds
  // 404 no route → no LiFi route within 8s budget; back off and retry
  throw new Error(await res.text());
}

const route = await res.json();
// route.to is guaranteed ∈ KEEPER_TO_ALLOWLIST[fromChain]
await vault.swapAtomic(route.to, route.data, route.value);
```

The vault contract (`alm/evm/src/Vault.sol`) is responsible for the on-chain side of the allowlist -the keeper-mode validator is defense-in-depth, not the only gate.

## 3. Direct SDK Use (Bypass Service)

For batch / offline workloads where running an HTTP service is overhead, depend directly on `@btr-supply/swap`:

```ts
import { getAllTimedTr, AggId } from '@btr-supply/swap';

const trs = await getAllTimedTr({
  aggIds: [AggId.LIFI, AggId.SOCKET, AggId.ONE_INCH],
  input:  { chainId: 1, address: '0x...', symbol: 'USDC', decimals: 6, name: 'USD Coin' },
  output: { chainId: 10, address: '0x...', symbol: 'USDC', decimals: 6, name: 'USD Coin' },
  inputAmountWei: 1_000_000_000n,
  payer:        '0x...',
  receiver:     '0x...',
  maxSlippage:  50,
  expiryMs:     8000,
});
```

This skips rate-limiting, mode resolution, and keeper validation -apply them yourself if needed.

## 4. CLI

`@btr-supply/swap-cli` (`swap/packages/cli/`) wraps the SDK for terminal use:

```bash
btr-swap quote \
  --input  1:0xa0b86991c6218b36c1d19d4a2e9eb0ce3606eb48:USDC:6 \
  --output 10:0x0b2c639c533813f4aa9d7837caf62653d097ff85:USDC.e:6 \
  --input-amount 1e9 \
  --payer 0xYOUR_ADDR \
  --aggregators LIFI,ONE_INCH \
  --serialization TABLE \
  --display RANK
```

Useful for: pre-trade price discovery, smoke-testing aggregator availability, comparing routes during integration.

## 5. Errors

| Status | Body `error` | Meaning |
|--------|--------------|---------|
| 400 | `body must be a JSON object` / `missing field: <name>` | Validation failure (`server.ts:49-61`). |
| 404 | `not found` | Unknown path. |
| 404 | `no route` | `/build` ran the fan-out but no aggregator returned a usable route within the budget. |
| 405 | `method not allowed` | Wrong HTTP verb. |
| 422 | `keeper calldata rejected` | Keeper-mode route landed at a non-allowlisted `to`. The `detail` field carries the violated reason. |
| 429 | `rate-limited` | See [05. Rate Limit](/docs/swap/05-Rate-Limit). |
| 500 | `internal error` | Unhandled. The service logs the underlying exception. |

## 6. Deployment

The service runs as `@btr-protocol/swap-service` under the back monorepo:

```bash
cd /Users/derpa/Work/btr/back
bun install
bun run --filter @btr-protocol/swap-service dev   # local
bun run --filter @btr-protocol/swap-service build # bundle
```

Container expectations: stateless, horizontally scalable (modulo the rate-limiter caveat in [05. Rate Limit](/docs/swap/05-Rate-Limit) §6). One pod per region is sufficient until an HA limiter is implemented.

Required env: aggregator API keys for any AggId in `USER_MODE_AGGS` that requires one. Missing keys silently disable the corresponding aggregator at runtime.
