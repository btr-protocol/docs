# 7. BTR Swap -LiFi Intents (Catalyst)

> Intent-based cross-chain swaps via the LiFi Intents (Catalyst / OIF) network. BTR uses the **Simple Escrow** path, opened on-chain through the ALM `Swapper.openIntent(...)` adapter. Solvers detect on-chain orders automatically; the order server is consulted only for quote discovery and lifecycle polling.

## 1. Why Intents

For large swaps ($1k+) atomic routing through a meta-aggregator pays a visible price impact + bridge fee. Intent solvers compete off-chain on fill price; the winning solver advances funds on the destination chain and reclaims the locked input only after an oracle verifies delivery. Net effect: tighter execution and refund-on-timeout safety.

The front gates the Intent toggle behind a `$1,000` floor (`INTENT_USD_FLOOR` in `useSwap.ts`).

## 3. On-Chain Escrow Flow (canonical for BTR)

```
front                    back (swap-service)        order.li.fi              ALM Swapper          settler / solver
 │ POST /api/swap/quote   │                            │                          │                       │
 │ {kind:"intent", ...}   ├── POST /quote/request ─►   │                          │                       │
 │                        ◄── quotes[] ────────────────┤                          │                       │
 ◄── best, quoteId ──────┤                            │                          │                       │
 │                        │                            │                          │                       │
 │ POST /api/swap/intent/prepare {quoteId, ...}        │                          │                       │
 │                        ├── (cache) /chains/supported│                          │                       │
 │                        │   → InputSettlerEscrow, oracles                       │                       │
 │                        │  build StandardOrder + abi-encode `open(bytes)`       │                       │
 ◄── {order, inputSettlerEscrow, abiEncodedCalldata, tokenIn, ...} ─┤             │                       │
 │                                                                                │                       │
 │ wallet.signTx → Swapper.openIntent(tokenIn,tokenOut,amountIn,minOut,deadline,orderData) ─►            │
 │                                                                                ├── settler.call(...) ─►│
 │ ← txHash                                                                       │                       │
 │                                                                                                        │ solver fills on dest
 │ GET /api/swap/status?onChainOrderId=...                                                                │
 │                        ├── GET /orders/status ───────────────────────────────►│                       │
 ◄── {meta:{orderStatus:"Signed"→"Delivered"→"Settled"}}                                                  │
```

Key facts:
- The user signs **the EVM transaction** to `Swapper.openIntent`, not an EIP-712 order. The Simple Escrow on-chain path doesn't require a client-side typed-data signature.
- `Swapper.openIntent` is the existing ALM adapter (`alm/evm/src/adapters/Swapper.sol`). It pulls `amountIn`, approves the pinned ERC-7683 settler, then forwards `orderData` verbatim via `settler.call(orderData)`.
- `orderData` = `open(bytes)` selector + ABI-encoded `StandardOrder` payload (built server-side in `back/services/swap/src/lib/intent.ts`).

## 4. `StandardOrder` Shape (spec, verbatim)

```solidity
struct StandardOrder {
    address user;
    uint256 nonce;
    uint256 originChainId;
    uint32 expires;          // BTR default: now + 1800s
    uint32 fillDeadline;     // BTR default: expires - 60s
    address inputOracle;     // from /chains/supported
    uint256[2][] inputs;     // [[uint256(asset), amount]]
    MandateOutput[] outputs;
}

struct MandateOutput {
    bytes32 oracle;
    bytes32 settler;
    uint256 chainId;
    bytes32 token;
    uint256 amount;
    bytes32 recipient;
    bytes call;              // wire JSON: callbackData
    bytes context;
}
```

JSON wire serialization (handled in `swap/packages/core/src/LiFi/intent.ts`):
- All `uint256/uint32` fields → decimal strings.
- The on-chain `call` field is renamed to `callbackData` on the wire.

Addresses on quote requests use **EIP-7930 interoperable addresses** (`encodeInteropAddress` helper). Layout:

```
0x | Version=01 | ChainType=00 (EVM) | ChainRefLen | ChainRef (chainId BE) | AddressLen=14 | Address (20B)
```

## 5. Lifecycle

`GET /api/swap/status?onChainOrderId=<id>` (or `catalystOrderId=<id>`) proxies LiFi's `/orders/status` and returns `meta.orderStatus` ∈:

| State | Meaning |
|---|---|
| `Submitted` | order received by the order server (transitional) |
| `Open`      | order registered on-chain (transitional) |
| `Signed`    | order available for solver pickup |
| `Delivered` | solver has delivered output on destination chain |
| `Settled`   | proof verified, escrow released to solver — order complete |
| `Refunded`  | (BTR-surfaced) post-`expires` reclaim |
| `Failed`    | (BTR-surfaced) terminal error |

The front polls every 4s for ≤ 6 min (`useSwap.ts`). Status pane surfaces the current state.

---

## 6. Why NOT `/orders/submit`

Per spec, `POST /orders/submit` is only used for the **off-chain gasless Compact** flow (sponsor-signed claims). On-chain Escrow orders are detected by solvers automatically through ERC-7683 events. BTR's `/api/swap/intent/submit` returns **HTTP 410 Gone** to make the contract loud:

```json
{
  "error": "gone",
  "message": "use Swapper.openIntent on-chain — Li.Fi Intents Escrow flow does not require /orders/submit (solvers auto-detect on-chain orders)",
  "hint": "POST /api/swap/intent/prepare → call Swapper.openIntent on ALM"
}
```
