# 3. BTR Swap -Aggregators

> Per-aggregator surface. Three classes: cross-chain meta-aggregators (route across chains via bridges), passive liquidity aggregators (single-chain DEX routing), and intent / RFQ (off-chain settlement).

## 2. Adapter Contract

Every aggregator implements the abstract interface in `swap/packages/core/src/abstract.ts`:

| Method | Purpose |
|--------|---------|
| `getQuote(params)` | Returns a price-only quote (output estimate, fee breakdown) without calldata. |
| `getTransactionRequest(params)` | Returns an `ITransactionRequestWithEstimate` -calldata + estimates, ready for signing. |
| `parseTransactionRequest(raw)` | Maps the upstream provider response into the shared `ITransactionRequestWithEstimate` shape. |
| `getStatus?(stepId, txHash)` | Optional. Polls the upstream status endpoint and returns `OK` / `PENDING` / `FAILED` / `UNKNOWN`. |

`getAllTimedTr` (re-exported by the SDK index) iterates over the supplied `aggIds`, calls `getTransactionRequest` in parallel with an 8-second `expiryMs`, attaches `latencyMs`, and returns the array sorted by **net output** (output − gasCostUsd − feeCostUsd).

## 3. Atomic vs Intent

- **Atomic**: caller signs and broadcasts a single tx whose calldata is returned in `tr.data`. Settlement on the same block as inclusion.
- **Intent**: caller signs an off-chain order (EIP-712 typed data); a third-party solver / filler submits the on-chain tx. `tr.data` is `undefined`; the route's `kind === 'intent'` and an out-of-band order id is returned in `tr.steps[0].metadata`.

For now the active set (`USER_MODE_AGGS`) is **all-atomic**. Intent adapters remain in source but are not wired into `config.ts`.

### 3.1. LiFi Intents (Catalyst) -dedicated surface

Cross-chain intent routing via the LiFi Catalyst / OIF order server lives outside `USER_MODE_AGGS` and is exposed through dedicated endpoints (`/api/swap/intent/prepare`, `/api/swap/status?onChainOrderId=...`). The user signs an EVM tx to `Swapper.openIntent(...)` on the ALM contract; solvers detect on-chain orders automatically. See [`07. LiFi Intents`](./07.%20LiFi%20Intents.md) for the full flow, lifecycle, and configuration.

## 4. Authentication

Auth is per-adapter, read from `process.env`. The SDK never prompts; missing keys cause that aggregator's `getTransactionRequest` to throw, which `getAllTimedTr` catches and excludes from the returned array. Net effect: a missing key silently disables the aggregator.

For multi-tenant deployments, set the keys in the swap-service environment, **not** in the SDK consumer's environment.

## 5. Status Endpoint (Forward-Looking)

`GET /status/:hash` currently returns `NOT_FOUND` (`server.ts:75-78`). Wiring requires:

1. Persist the `(aggId, txHash)` tuple at `/build` time (currently the service is stateless).
2. On `/status/:hash`, look up the tuple and dispatch to the correct adapter's `getStatus`.
3. Normalise upstream status codes to `{ id, status: 'OK' | 'PENDING' | 'FAILED' | 'UNKNOWN', steps?: [...] }`.

The tuple persistence is the open piece -a Redis or local KV is sufficient.

## 6. Disabled Adapters

`UNIZEN`, `SQUID`, `RANGO` are commented out in `USER_MODE_AGGS` (`config.ts:23-27`) pending API-key procurement. To re-enable: set the relevant env var, uncomment the AggId, restart. No code changes required.

`AIRSWAP` is permanently disabled (`disabled` tag in `swap/README.md:70`); the source remains for reference but is excluded from publishing.
