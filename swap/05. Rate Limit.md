# 5. BTR Swap — Rate Limit

> Per-IP rolling 5-minute window; `quote/exec` ratio thresholds gate abusive callers. Three tiers: 60s cooldown, 1h ban, 24h ban. In-memory only — resets on process restart.

---

## 1. Goal

Prevent quote-spamming without throttling legitimate flows where a user fetches several quotes before executing one. The key observation: a normal session yields roughly `quotes ≤ 5 × executions`. Anything materially above that is either reconnaissance (price discovery) or an abuse vector (probing for arbitrage, scraping, etc.).

## 2. Algorithm

Implemented in `back/services/swap/src/lib/ratelimit.ts`. Both `/quote` and `/build` invoke `checkAndRecord(limiter, ip, kind)`. `kind` is `'quote'` for `/quote` and `'build'` for `/build` (`/build` increments the **execution** counter — the calldata return is treated as proof of execution intent).

Per IP, the limiter maintains:

```ts
type Bucket = {
  quotes:     number[];   // ms timestamps within last 5 min
  executions: number[];
};
```

On each call:

1. Prune timestamps older than `WINDOW_MS = 5 × 60 × 1000`.
2. Append the current timestamp to the appropriate array.
3. Compute `ratio = quotes.length / max(1, executions.length)`.
4. Tier-escalate if the ratio crosses a threshold.

## 3. Tiers

`ratelimit.ts:11-13`:

| Constant | Value | Action |
|----------|-------|--------|
| `RATIO_SOFT` | `8`  | 60 s cooldown |
| `RATIO_HARD` | `24` | 1 hour ban |
| `RATIO_PERM` | `64` | 24 hour ban |

Escalation is **monotone** — once banned at 24h, a re-trigger of the soft tier within the active ban window does not de-escalate (`ratelimit.ts:108-111`). A more severe new tier upgrades the existing ban; a less severe new tier is ignored until the ban expires.

A banned caller continues to consume `quote` slots (`ratelimit.ts:91-95`), so spamming during a 60s cooldown can still escalate to the 1h or 24h tier.

## 4. Response

Rate-limited callers receive HTTP `429` with both a JSON body and a `Retry-After` header:

```http
HTTP/1.1 429 Too Many Requests
Retry-After: 60
Content-Type: application/json

{ "error": "rate-limited", "retryAfter": 60 }
```

Service-side log line: `rate-limited` with `{ ip, kind, retryAfter }` (`server.ts:34-39`).

## 5. IP Extraction

`extractIp` (`ratelimit.ts:73-76`) reads, in order:

1. `x-forwarded-for` (first hop).
2. `cf-connecting-ip` (Cloudflare).
3. `x-real-ip` (nginx / k8s ingress).
4. Fallback: `'unknown'` — meaning all anonymous direct callers share one bucket. **Always deploy behind a reverse proxy that sets one of the above headers.**

## 6. State & HA

State is process-local (`Map<ip, Bucket>` + `Map<ip, Ban>`). Implications:

- A process restart resets all buckets and bans.
- Horizontal scaling is **not** linear: each replica enforces its own counters. Two replicas double the effective per-IP budget.

For HA, swap the in-memory `Limiter` for a Redis-backed implementation. The `Limiter` interface (`ratelimit.ts:25-29`) is intentionally minimal so this is a one-file replacement.

## 7. Tuning

To increase quote tolerance (e.g. for an analytics dashboard scenario), raise `RATIO_SOFT`. To disable entirely (e.g. behind an authenticated keeper-only endpoint), wrap `checkAndRecord` to short-circuit `{ allowed: true }` based on a header / mode check before the limiter is consulted.

The current tiers are conservative — calibrated for a public, unauthenticated front-end. Internal keeper endpoints should run on a separate instance or with the limiter bypassed for trusted source IPs.
