# 7.2. Adapters & Pools

> What external DEX types BTR Vaults can manage liquidity on, how ranges are chosen, and how multiple holders share one adapter.

---

## 1. Supported DEX Types

`Dex.sol:16` defines the canonical enum:

```solidity
enum DexType { UniV3, UniV4, PancakeV3, PancakeInfinity, Algebra, Ramses }
```

| Type | Slot0 selector | Notes |
|------|----------------|-------|
| `UniV3` | `slot0()` (`0x3850c7bd`) | Canonical CLMM (Uniswap V3, Aerodrome Slipstream — V3-compatible). |
| `UniV4` | `StateView.getSlot0(PoolId)` (`0x76d0bdb1`) | PoolId-keyed state; uses dedicated `pm` (PositionManager) + `sv` (StateView). |
| `PancakeV3` | `slot0()` | V3-fork, identical layout. |
| `PancakeInfinity` | `StateView.getSlot0(PoolId)` | V4-fork; same V4 init path. |
| `Algebra` | `globalState()` (`0xc8e5e498`) | Algebra fork (e.g., QuickSwap, Camelot, ThenaFi). |
| `Ramses` | `globalState()` | Algebra-derived. |

Branching lives in `Dex._slot0Branched()` (`Dex.sol:440`). Two init flavours exist: `initPool` for V3-style pools and `initPoolV4` for V4 / PancakeInfinity, where the caller supplies pre-computed `pid`, `pm`, `sv`, `hooks`, `fee`, and `tickSpacing`.

## 2. Tick Ranges and Range Categorization

Operators select `(tickLower, tickUpper)` per `rebalance` (`Dex.sol:304`). Range categorization is operator-driven and pool-suitability-driven:

| Category | Asset class | Range width | Rebalance cadence |
|----------|-------------|-------------|-------------------|
| **Stable** | USD/USD, EUR/USD pegs | Tight (a few bips around peg) | Rare; widen on de-peg events. |
| **LSD-peg** | wstETH/ETH, sfrxETH/ETH | Narrow band tracking peg drift | Periodic; on-chain APY accrual moves the peg. |
| **Volatile** | ETH/USDC, blue-chip/USD | Wide; centred on oracle mid | Frequent; bounded by `minIntervalSecs`. |

The contract is range-agnostic. The `tickSpacing` is fixed at init and the only runtime constraint is `tickLower < tickUpper` (`Dex.sol:307`). Range policy is governed off-chain by the keeper, gated on-chain by:

- `minIntervalSecs` — minimum dwell between rebalances; keeper may only ratchet up by `KEEPER_INTERVAL_DELTA = 1 day` per call (`Dex.sol:42–44`, `:184–199`).
- `frozen()` checked **twice** per rebalance (pre-burn and post-swap), so atomic routes that move the pool tick cannot bait LP into a deviating market (`Dex.sol:309`, `:338`).
- `maxSlipBps` — post-mint TVL slippage envelope, enforced only when **all** routes are atomic (`Dex.sol:343–347`). Intent routes have invisible in-flight value, so per-route `minOut` + Swapper enforcement carry slippage protection.

## 3. Multi-Holder Model

`Dex.sol:92–93`:

```solidity
mapping(address => bool)    public   isHolder;
mapping(address => uint128) internal principalOf;
```

A single Dex adapter can be shared by multiple holders — a vault, an EOA, or a multisig — each with their own internal LP-share balance.

- `setHolder(h, true)` — owner adds a holder.
- `setHolder(h, false)` — owner removes a holder; reverts if `principalOf[h] != 0` (`Dex.sol:175`).
- `pull(amount)` — holder deposits `token0`; mints shares pro-rata to current marked NAV (`Dex.sol:258`).
- `withdraw(to, amount)` — holder withdraws up to idle `token0`; burns shares rounded **up** so dust cannot drain residuals (`Dex.sol:288`).

The share-mint paths are gated by `_checkLive()` (`Dex.sol:421`), which fails closed if the adapter is frozen or has an open intent — preventing in-flight intent NAV drift from inflating new shares.

## 4. Aerodrome Slipstream

Aerodrome's Slipstream is V3-compatible (`slot0`, `mint`, `burn`, `collect`); use `DexType.UniV3` and the adapter operates without modification. CL gauge / emissions are out-of-scope for the adapter and handled off-chain.

## 5. Pool Init Flow

```text
owner → Factory.deployDex()                     // returns dex
owner → Dex.initPool(pool, dexType, pp, ts)     // V3-family
       OR
owner → Dex.initPoolV4(pid, t0, t1, pm, sv, hooks, fee, ts, pp)
owner → Dex.setHolder(vault, true)
vault → Vault.addAdapter(dex)                   // calls Dex.vaultAssets via _sync
```

`Vault.addAdapter` requires `IDex(a).asset() == _asset` (`Vault.sol:472`) — adapter `token0` is the vault asset. `_sync` runs strict (no try/catch) on add, so a misconfigured adapter fails loud.

## 6. Worth-Documenting Invariants (per adapter)

- `principalOf[h] == 0` ⇒ holder may be deregistered, otherwise revert (`Dex.sol:175`).
- `assetValue(token)` reverts on `killed`, `BadDenom`, `StalePool`, or `StaleOracle` (`Dex.sol:357–365`).
- `liquidity == 0` ⇒ amounts are `(0,0)`; idle-only NAV (`Dex.sol:365`).
- After `rebalance` with **any** intent route, the post-mint slip check is **skipped** (`Dex.sol:343`).
- Pool callback is gated by `msg.sender == pool` and only honoured for the active position (`Dex.sol:511`).
