# Open Questions & Known Limitations

> Surfaced by the Phase 42W multi-cohort math-review pass (extraction, adversarial cross-validation, fixer-rewrite, and final-review stages). Each item below is a deliberate **deferral, gap, or design choice** worth tracking openly.

---

## 1. Code↔doc mismatches (active)

These are issues where the source code and the design intent disagree. They were found during the formal review, not in production.

### 1.1 Latent σ-floor trade-rejection - currently inert

**Issue.** The design specifies that trades whose risk-budgeted stop-loss falls below `sl_min_sigma · σ_P · √H` should be **rejected** (the SL would be tighter than the σ-floor allows). The intent: protect from leverage settings that force unrealistically tight stops.

**Reality in master.** `trainer.rs:1193` clamps `sl_pct ≥ k_sl_min · σ_H` before the rejection check at `trainer.rs:1215` can fire. The check is unreachable; the rejection path is dead code. Effectively, the σ-floor acts as a **silent clamp**, not a rejection gate.

**Consequence.** At high volatility (≥200 bps/bar) with leverage 3×, the *effective* per-trade risk silently inflates to ≈ 2× the configured $\rho = 2\%$, reaching ≈ 4% in the worst case. Backtests stay honest (the simulator follows the same clamp), but **live deployment carries more tail risk than the nominal configuration suggests**.

**Remediation options for the Prime team:**

- **(a) Remove the clamp at line 1193.** Restore the original rejection semantics - high-leverage trades in vol spikes simply do not open.
- **(b) Move the check upstream of the clamp.** Same effect as (a) but more surgical.
- **(c) Re-label as a soft floor.** Acknowledge in docs that the σ-floor is by design a clamp, not a rejection - adjust risk-budget reporting accordingly.

**Recommendation:** **(b)** - minimal code change, preserves the design intent, and makes the rejection branch reachable. See [§07 Directional Trading §7.4](./07.%20Directional%20Trading.md).

### 1.2 Stage-2 meta-classifier - plumbed but inactive

**Issue.** The trainer struct declares `meta_model: Option<GBMModel>` (`trainer.rs:204-208`) and the methodology is documented as a two-stage López de Prado meta-labeling pipeline. The Stage-2 classifier is meant to filter Stage-1 false positives and provide a calibrated bet-size signal.

**Reality in master.** `meta_model: None` everywhere in master (`trainer.rs:587`, `:888`). Confidence is currently `pred.abs()` in regression mode (`trainer.rs:1222-1227`), not a calibrated probability.

**Consequence.** Bet sizing is by signal magnitude only, not a separate meta-probability. The pipeline is operating as Stage-1-only.

**Activation prerequisites:** nested cross-fit walk-forward inside each training fold to generate clean meta-labels, then train Stage-2 on the out-of-fold predictions. Out of scope for current cadence; tracked here.

See [§05 Perpetual Boosting §5.4](./05.%20Perpetual%20Boosting.md).

### 1.3 Intra-fold seed-diverse ensembling - reserved

**Issue.** The methodology describes an ensemble with $N_{\text{eff}}$ for variance reduction via seed diversity. The struct supports it.

**Reality.** `primary_models: vec![model]` - single seed per fold (`trainer.rs`). The decay-weighted inter-fold ensemble is **active** ($E = 4, \delta = 0.5$); the intra-fold multi-seed branch is **dormant**.

**Consequence.** Variance reduction comes from the inter-fold ensemble only; the bias-variance-covariance gains documented in §5.5 of the methodology are theoretical until intra-fold seed diversification is enabled.

See [§05 Perpetual Boosting §5.5](./05.%20Perpetual%20Boosting.md).

## 2. Missing cost-model components

The walk-forward backtest uses a simple linear fee model. Several real-world costs are not yet modelled.

| Cost | Status | Impact |
|---|---|---|
| **Fee per side** | ✓ modelled (3 bps round-trip) | Already in `fitness.rs`. |
| **Slippage** | ✗ not modelled | Backtest PnL overstates fillable execution; severity = $\Theta(\sqrt{\text{size}})$ for typical impact curves. |
| **Funding rate** (perp shorts) | ✗ not modelled | Backtested SO strategies in funding-positive regimes are PnL-optimistic. |
| **Borrow rate** (margin shorts) | ✗ not modelled | Same direction as funding for CEX margin. |
| **Block-time gas / MEV** | n/a - off-chain only | Relevant only for the on-chain ALM consumer (CL range output). |

**Roadmap:** introduce a parameterised slippage curve (sqrt-law: $\text{slip}(q) = k \cdot \sqrt{q / \text{ADV}}$, calibrated per asset) and a funding-rate term driven by realised funding history from the NXR collector. ETA: pre-mainnet.

## 3. Missing mathematical depth

Items flagged during adversarial review as desirable theoretical extensions but **not** active blockers. Each is an opportunity for a future research note.

### 3.1 Predictive distribution / uncertainty quantification

Currently the model emits a **point** prediction $\hat y$. No native predictive variance or confidence interval. The single-seed primary model has no internal source of epistemic uncertainty. Stage-2 (when activated) provides a probability-of-profit estimate but not a quantile distribution.

**Candidates:** quantile boosting (Friedman 2001 §6, or LightGBM quantile loss); conformal prediction wrapper; intra-fold seed ensemble to obtain a sampling distribution.

### 3.2 Volatility forecasting

Parkinson is a *spot* estimator, not a *forecast*. Prime currently uses $\hat\sigma_P(t)$ as if it were $\hat\sigma_P(t+H)$. GARCH/EGARCH overlays are unexplored but would tighten the σ-normalisation.

**Candidate:** light EWMA-σ or GARCH(1,1) post-Parkinson, feeding `entry_threshold_sigma`, SL/TP, and CL-range widths.

### 3.3 Regime detection

No explicit regime-switch feature. The MTF blend in the bar-generator partially absorbs regime drift via inverse-variance weighting, but the model is not informed when "regime $r_t$" is in flux.

**Candidate:** HMM or Bayesian online change-point detector emitting a soft-regime feature into the GBM matrix.

### 3.4 Higher-order moment corrections

Parkinson assumes near-mesokurtic standardised increments under GBM. Empirically the kurtosis falls from ≈ 12 (1-second bars) to ≈ 0.4 (adaptive Renko bars) - sub-Gaussian, ✓. But residual skewness in directional regimes is not modelled. Rogers-Satchell would add drift robustness; Yang-Zhang would add overnight-gap robustness (n/a for crypto). Not currently a bottleneck.

### 3.5 Hyperparameter sensitivity

Of the parameters in [§09 Hyperparameters](./09.%20Hyperparameters.md), only `leverage` and (CL-specific) range-width multipliers are grid-searched. Notably absent:

- `budget = 0.3` - library default; **no sensitivity sweep run in production**. Adversarial review flagged this; a sweep $b \in \lbrace 0.1, 0.2, 0.3, 0.5, 1.0\rbrace$ on a held-out fold is the obvious next experiment.
- `parkinson_lookback = 50` - picked on calibration heuristics.
- Renko multiplier $k$ - per-ticker, calibrated to target bars/day (upstream NXR `scale_to_target_k`, rolling 365d median), not bottom-up optimised by Prime.
- Ensemble decay $\delta = 0.5$ - picked on prior intuition.

**Roadmap:** sequential one-dimensional sweeps with the production fitness as the objective. Estimated effort: 1-2 weeks for the full battery.

### 3.6 Sample-size theory

No formal lower bound on the number of bars required for stable boosting. Empirically `train_bars = 5000` works; the theoretical foundation (PAC-style bound with $\log(1/\delta) / \epsilon^2$ scaling) is left as documentation TODO.

### 3.7 Stationarity argument for features

Several features ([§03 Features](./03.%20Features.md)) - Hurst exponent, RSI, z-scores - implicitly assume locally stationary data. The information-time mapping (§01) makes this approximately true at the bar level, but a formal stationarity claim is not derived. Phase-2 work.

### 3.8 Class imbalance / sample weighting

`compute_sample_weights` is not implemented in the ML crate (see [§04 Labels §4.6](./04.%20Labels.md)). Class imbalance handling and label-concurrency weighting (López de Prado 2018, ch. 4) are deferred. Currently the perpetual booster's internal regularisation absorbs mild imbalances; severe regimes (e.g., one-sided trends labelling 80% positive) are untested.

## 4. CL-specific limitations

### 4.1 No minimum-improvement gate on range changes

The rebalance trigger (`raw_divergence`) re-rebalances whenever predicted optimum moves away from current range. There is **no** check that the predicted improvement exceeds the rebalance gas + slippage cost. ALM Vaults rebalance more often than is economically justified in tight-vol regimes.

**Remediation:** add an `min_alpha_improvement_bps` threshold gated against estimated rebalance cost. Estimated effort: half a day in `cl.rs`.

### 4.2 LVR formula uses adapted form

[§08 CL Range Management §8.4](./08.%20CL%20Range%20Management.md) - the LVR formula in production (`cl.rs:163`) is `0.75 · σ² · 0.125 · √p / L_ref`, an adapted form of Milionis et al. 2022's classical $(1/8)\sigma^2 p L$ with a Parkinson-bias dampening factor 0.75 and a √p / L_ref scaling for unit-liquidity CL positions. The adapted form has been formally re-derived (Phase 42W) and matches Milionis et al. Thm 1 with $\mathcal{P}''(p) = -(1/2) p^{-3/2}$ substituted. **Not** a bug - but worth a follow-up validation against a Monte-Carlo LP simulator before mainnet.

### 4.3 σ-normalised multi-output range projection - planned, not in master

The current code emits a single scalar signal mapped to a width multiplier. The full $(\hat w_l, \hat w_h, \hat\mu)$ multi-output formulation in [§08 §8.7](./08.%20CL%20Range%20Management.md) is **planned but not active**. Asymmetric ranges are not predicted; midpoint offset is implicit.

## 6. Phase-42X math review (new items)

Surfaced by the Phase 42X review pass. Each is a deferred fix, not a production blocker.

### 6.1 Renko annualisation rate

**Issue.** `bars_per_day_for_annualization` defaults to $1440$ (1-min bars). For adaptive-Renko streams the realised bars/day is *event-driven* (typically 50–400/day for crypto majors) and must be supplied via the override path. Without it, annualised Sharpe is inflated by $\sqrt{1440 / B_{\text{actual}}}$.

**Status.** Override exists but is silent on misuse. Roadmap: hard-fail when (a) the bar stream is Renko-tagged and (b) the override matches the 1-minute default. See [§06 Walk-Forward Validation §6.3](./06.%20Walk-Forward%20Validation.md).

### 6.2 σ → 0 staleness watchdog

**Issue.** When $\hat\sigma_P \le 10^{-10}$ the trainer silently disables TP (`trainer.rs:1211`) and bypasses the σ-floor rejection check (`trainer.rs:1215`). A stale or frozen feed therefore yields TP-less, floor-less trades instead of an explicit "stale data" rejection.

**Status.** Healthy-market unreachable in practice; pathological-input safety net missing. Roadmap: add an upstream staleness guard that rejects bars with $\hat\sigma_P < \epsilon$. See [§02 Parkinson §2.6](./02.%20Parkinson%20Volatility.md).

### 6.3 Feature-selection look-ahead

**Issue.** `precompute_data` builds the feature matrix once over the *full* series. Per-row causality (I1) holds, but the *feature selection* step (variance pruning, NaN screening, cross-asset alignment) sees the full sample. This is a mild in-sample step relative to strict López-de-Prado purged-CV.

**Status.** Leakage magnitude is empirically small for stationary feature families. Roadmap: re-run selection per fold using only $\mathcal{T}_k$. See [§06 Walk-Forward Validation §6.2](./06.%20Walk-Forward%20Validation.md).

### 6.4 Single-bar Parkinson in CL labels

**Issue.** The CL labelling routine (`cl.rs:145`) substitutes a single-bar Parkinson realisation into the LVR rent integral rather than the window-averaged $\hat\sigma_P$ used elsewhere. This produces high-variance per-bar LVR estimates that the booster must then average away through tree depth.

**Status.** Empirically the booster denoises this; theoretically a window-averaged σ would be more efficient. Roadmap: A/B test windowed vs single-bar σ in the LVR term on a held-out fold. See [§08 CL Range Management §8.4](./08.%20CL%20Range%20Management.md).

### 6.6 Feature dedup vs sample budget (Phase 42Y audit → 42Z deployed)

**Status (Phase 42Z, 2026-05-17): RESOLVED.** Production now runs the **32-feature compressed set** per the 4-expert panel sign-off. The 8 hard gates in [§03 §3.7.2](./03.%20Features.md) are now **MANDATORY CI promotion gates** run before each model deploy (per-fold permutation importance LO/SO-split, tail-event subset CV at 5%, max-loss-month delta $\leq 25$ bps, $\Theta(w;\,H)$ stability $\pm 15\%$, drawdown-KS parity, sensor-fault and stale-feed sims, per-fold FS plumbing). The 61-baseline is retained for ablation via `--baseline-features`. §6.3 (per-fold FS) remains OPEN as a MUST-CLOSE blocker for the gate-3.7.2 §8 sub-requirement; the rest of the gates run in CI today against the global selector, with the per-fold refactor scheduled next.

**Historical context (kept for audit trail):**



**Issue.** The production feature matrix at the canonical 1-alt configuration is **61 features** (29 MTF + 12 Renko-tracker + 8 cross-asset + 12 microstructure; see [§03 §3.1](./03.%20Features.md) and [§03 §3.9](./03.%20Features.md)). The training window is $N_{\text{train}} = 5000$ bars with horizon $H = 20$, giving an effective independent-sample count $N_{\text{eff}} \approx N_{\text{train}} / H = 250$ (Politis non-overlap rule; even tighter under López-de-Prado triple-barrier uniqueness, $N_{\text{uniq}} \approx 150 - 200$).

Under Harrell's "$\geq 10$ samples per feature" rule and the perpetual booster's `budget = 0.3` complexity regularisation, the honest fit-able feature ceiling is $\sim 25 - 30$. **The current 61 is approximately $2\times$ over budget**. The Phase 42Y adversarial-redundancy audit identified a compressed shortlist of **28 features** ([§03 §3.7](./03.%20Features.md)) that should preserve most of the signal: drop ADX×3, `rsi_s`, M-scale for slow indicators, 10 of 12 renko-tracker, 5 of 8 cross-asset, and 10 of 12 microstructure features.

**Status.** Production runs all 61 features; the booster's internal regularisation absorbs the over-budget regime in practice. The audit verdicts are research hypotheses, **not deployed cuts**. Empirical CV gating is required before any drop is moved to master:

1. **Permutation importance per fold** on the full 61-feature set; bottom-half-in-every-fold features become drop candidates.
2. **Pairwise Spearman $|\rho| > 0.85$** on the train fold; drop the lower-importance member.
3. **VIF $> 10$** flag for linear-stability sanity.
4. **Re-train at 28 features**; ship if CV Sharpe is within 10% of the 61-feature baseline.

**Coupling to §6.3.** Per-fold feature selection (§6.3 above) and per-fold pruning are the same plumbing change. The cleanest path is to (a) move feature selection inside the walk-forward loop, then (b) compare the in-loop selector's output to the hand-curated shortlist. If they substantially agree, ship the static shortlist (simpler, less variance); if they diverge, the in-loop selector wins.

**Expert-panel outcome (Phase 42Y).** A 4-of-4 **APPROVE-WITH-MODS** verdict was returned by a four-reviewer panel (statistics, microstructure, GBM-ML, risk-adversarial). The consensus shortlist is **$\sim 32$ features** (range 30-35), **not the original audit's 28** — the floor was raised via *residualisation* of higher-$\rho$ partners (e.g. `jump_var_z = z(\max(rv-bipower,\,0))`, `drift_resid_z = drift` residualised against `ret_s`) rather than strict drop. Sample-size also revised: Politis-Romano block-bootstrap with $\ell^\star \approx 46$ gives $N_{\text{eff}} \approx 110 - 150$ (anchor 150), tighter than the audit's 250. Composition details and the consensus drop/rescue table live in [§03 Features §3.7](./03.%20Features.md), with per-expert verdicts tabulated in [§03 §3.7.1](./03.%20Features.md).

**Eight hard gates ([§03 §3.7.2](./03.%20Features.md)) are non-negotiable.** Aggregate Sharpe-within-10% is necessary but NOT sufficient. The gates require per-fold permutation importance split LO/SO, tail-event subset CV at 5% tolerance, max-loss-month delta $\leq 25$ bps, $\Theta(w;\,H)$ stability within $\pm 15\%$, drawdown-distribution KS parity, sensor-fault and stale-feed simulations, and per-fold feature-selection plumbing.

**Blocker dependency (historical, now superseded by Phase 42Z deploy).** §6.3 (per-fold feature selection) was the stated blocker. It remains OPEN as a MUST-CLOSE item for the gate-3.7.2 §8 sub-requirement; the other 7 gates run in CI today.

**ETA.** §6.3 plumbing: 2-3 days (still scheduled). 8-gate suite operational in CI as of 2026-05-17.

### 6.7 Drift residualisation method (Phase 42Z) — FIXED 2026-05-17

**Issue (historical).** `micro_drift_resid_z` ([§03 §3.10.2](./03.%20Features.md)) previously used a **series-wide** OLS coefficient $\hat{\beta} = \mathrm{Cov}(\text{drift},\,\text{ret\_s}) / \mathrm{Var}(\text{ret\_s})$, fit once on the full series — a look-ahead leak flagged as a BLOCKER by the Phase 42Z validator panel.

**Status: RESOLVED.** `crates/bin/src/ml.rs` now computes $\hat{\beta}_i$ via an **expanding-window OLS** with Welford-style running accumulators $(\Sigma X,\,\Sigma Y,\,\Sigma XY,\,\Sigma X^2)$. At each bar $i$, $\hat{\beta}_i$ uses only data in $[0..=i]$; warmup $n \ge 30$ before any nonzero β is emitted. The residual series is then z-scored with the existing causal rolling window ($S = 20$). Strictly causal end-to-end.

**Future work.** Move to a rolling 200-bar $\hat{\beta}$ estimate (refit per walk-forward fold) once the per-fold FS plumbing at §6.3 ships; the two changes are co-located. **Priority: LOW** (expanding β is causal and adequate for current panel sign-off).

### 6.8 Asymmetric-vol channel (Phase 42Z panel, MED)

**Issue.** Panel validators flagged that the current Parkinson-only volatility channel symmetrises high-low range and loses the asymmetric component (intraday up-vol vs down-vol drift). For directional strategies (LO/SO) this asymmetry should carry predictive content for next-bar return sign.

**Proposed.** Add a semi-variance ratio feature `sigma_asym = σ_up / σ_down` over the medium scale $M$, where $\sigma_{\text{up}}^2 = \frac{1}{n}\sum_{r_i > 0} r_i^2$ and $\sigma_{\text{down}}^2 = \frac{1}{n}\sum_{r_i < 0} r_i^2$, clamped to $[0.5,\,2.0]$. One feature, no parameter tuning beyond window. Sits adjacent to `vol_regime` in the MTF block (§3.8.1).

**Status.** Not implemented. Specification deferred to Phase 42AA. **Priority: MED.** No coupling to current 8-gate CI suite; gate inclusion deferred until empirical lift is measured against the 32-feature compressed baseline.

**Remediation.** Add `sigma_asym` to `features/mtf.rs` behind the runtime `compressed` flag (active by default once shipped); add `Y` row to §3.8.1 manifest.

### 6.5 Drawdown-duration penalty: e-fold time, not half-life

**Issue.** Doc symbol `T_{1/2} = 16` and config field `dd_duration_halflife_days` imply a half-life, but the formula `exp(-T̄ / 16)` is an *e-fold* time. Actual half-life is $16 \ln 2 \approx 11.1$ d.

**Status.** Doc renamed to $T_e$ in [§06 Walk-Forward Validation §6.4](./06.%20Walk-Forward%20Validation.md); config field name retained for backward compat. Roadmap: rename field to `dd_duration_efold_days` at next breaking-config bump.
