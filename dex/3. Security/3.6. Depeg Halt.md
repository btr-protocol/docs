# Base-Token Depeg Halt

> Hybrid-oracle circuit breaker on the base (numéraire) token. Internal TWAP drives all quoting; an external Chainlink feed gates swaps when the base token visibly depegs from its nominal 1e18 reference.

---

## 1. Motivation

AIMM's internal TWAP is the **primary** price source for every non-base asset (Earth Finance / Trader Joe LB lineage — internal price discovery from pool state, see [Oracles §2](/docs/dex/3.5-Oracles)). This is robust against single-block manipulation and avoids LP value leakage to Chainlink heartbeat lag (estimated $0.25M-$3M/yr at $10M TVL on external-only quoting per the Phase 42M T4 oracle review).

But internal TWAP cannot detect when the **base token itself** depegs. If the pool's numéraire (e.g. a stablecoin) silently moves from $1.00 to $0.95, every quote in the pool is implicitly mispriced by 5% — the internal oracle has no way to know the unit of account has shifted. Internal-only pricing is therefore vulnerable on monopoly-sequencer L2s (Base, Arbitrum today) where a sequencer-aware adversary can pin the base-token price across multiple blocks.

The solution: a hybrid mode where the **base token alone** is gated by an external Chainlink feed, while every other asset continues to quote off the internal TWAP. The external feed acts as a circuit-breaker, not a price source.

---

## 2. Mechanism (R44-2, R44-2b)

### 2.1. Configuration

Per-pool optional base oracle, set by owner via `PoolAux.setBaseTokenOracle(pool, oracle)` (`PoolAux.sol:141`, `Admin.sol:134`, `PoolAdminWrite.sol:137`):

- `oracle == address(0)` — **legacy mode**: base price hard-pinned to `1e18`. No halt. Use for fully-stable bases or pools where the operator accepts depeg risk.
- `oracle != address(0)` — **hybrid mode**: base price read live from the external feed and checked against the halt threshold on every quote.

Untimelocked because it is a *halt* knob (fail-safe direction): turning on the gate only ever **adds** restrictions, never relaxes them. Owner can disable (set to `address(0)`) to restore legacy quoting.

### 2.2. Halt Threshold

`Constants.BASE_DEPEG_HALT_BPS = 500` (5%) — `Constants.sol:50-56`.

Note: the supply-side ALM uses a tighter `BASE_DEPEG_HALT_BPS = 200` (2%) circuit breaker because the ALM is a value-preservation product (depositors expect NAV stability). The DEX-side 5% is wider because the DEX is a *quoting* venue — a tighter halt would false-positive on routine Chainlink heartbeat noise and break swap UX without LP benefit. The two thresholds are intentionally distinct.

On every swap quote, `Pricing.sol` reads the external base price and reverts `BaseDepegged(basePrice, devBps)` if:

```
devBps = |basePrice - 1e18| * BPS / 1e18 > BASE_DEPEG_HALT_BPS
```

References:
- `Pricing.sol:337` — primary check on quote entry
- `Pricing.sol:345-358` — `_checkBaseDepeg` helper
- `Pricing.sol:384` — R44-2b defensive recheck inside `_executeLeg` for the base-token edge-hop (cannot hardcode 1e18 when a real oracle is pinned)

### 2.3. Failure Modes

| Scenario | Behavior |
|---|---|
| Base oracle stale | Revert via Chainlink staleness check (inherited from external oracle TTL) |
| Base oracle sequencer-down (L2) | Revert via sequencer-up feed gating |
| Base oracle reports within 500 bps band | Quote proceeds, internal TWAP drives non-base prices |
| Base oracle reports > 500 bps deviation | All swaps revert `BaseDepegged` until depeg resolves OR owner unsets the oracle |

Test coverage: `dex/evm/test/PoolBaseDepeg.t.sol` — exercises legacy mode (no halt), within-band (swap succeeds), out-of-band (swap reverts), and the `_executeLeg` defensive branch (R44-2b).

## 3. Watcher / Pause Composition

The depeg halt composes with the existing pause / kill-switch surface (see [Access Control](/docs/dex/3.3-Access-Control) and [Flow Guards](/docs/dex/3.4-Flow-Guards)):

- **R44-2 is automatic and on-chain** — no off-chain watcher required. Every quote re-reads the base price.
- **Pause / kill switches remain available** as the operator-driven escape hatch for everything *other than* base depeg (smart-contract bugs, oracle compromise, governance emergencies).
- **No watcher is required to keep the pool safe** against base depeg, but an off-chain watcher *should* still monitor the event log for `BaseDepegged` reverts (signal that the gate has fired) so the operator can decide whether to switch oracles, raise the threshold via governance, or migrate base tokens.

## 4. Comparison vs Other Designs

### 4.1. vs OrbSwap (CCMM / Orbital)

OrbSwap's sphere invariant gives **intrinsic geometric depeg isolation**: a depegged asset drains asymmetrically through pure curve geometry, no oracle is involved. Within OrbSwap's domain (pegged-only baskets) this is strictly better than R44-2 — the curve does the work an oracle would do, with zero external trust assumption.

AIMM cannot match this for two reasons:
1. AIMM serves **mixed-volatility** baskets (stables + LSTs + majors). Volatile assets have no natural peg; the sphere invariant is undefined.
2. AIMM's price source is decoupled from reserves (TWAP-driven, not invariant-driven). The reserve geometry that gives OrbSwap its isolation does not exist in AIMM by design — that decoupling is what gives AIMM regime-adaptive quoting in the first place.

**Honest framing**: for a stables-only deployment, OrbSwap's intrinsic isolation is the better tool. AIMM's R44-2 is the next-best alternative when mixed-volatility makes the sphere unavailable.

### 4.2. vs Curve V1 Amplification Penalty

Curve V1's stableswap invariant uses an amplification coefficient `A` that flattens the curve near the equal-balance point and reverts to constant-product geometry as imbalance grows. A depegged asset organically drains the pool at progressively worse prices, similar in spirit to OrbSwap but with a 1D rather than n-sphere geometry.

Curve V1's mechanism is **gradual** (the price impact rises smoothly with imbalance) where R44-2 is **binary** (quote-or-halt at the 500 bps line). Curve loses LPs to depeg arb continuously up to the amplification cliff; AIMM stops the bleeding hard at 500 bps but cannot price the band-edge regime gracefully (it just halts).

**Tradeoff**: Curve V1's design preserves availability at the cost of LP losses; AIMM's R44-2 preserves LP value at the cost of availability. For a curated venue where halts are operationally acceptable and LP retention is paramount, the binary gate is the right call.

### 4.3. vs External-Only Quoting (Chainlink-Primary AMMs)

External-only AMMs (some DODO and Swaap variants, am-AMM) quote off Chainlink for *every* price, on every asset. The Phase 42M T4 review estimated this leaks $0.25M-$3M/yr at $10M TVL via heartbeat lag — Chainlink updates on a deviation/heartbeat schedule, so arbitrageurs predictably extract value in the lag windows.

R44-2 sidesteps this: Chainlink is consulted *only* for the base token, *only* as a halt gate, never as a quote source. The heartbeat-lag arb surface shrinks to the single base-token feed, and even that is bounded by the 500 bps band before it becomes harmful.

## 5. Tech Debt and Mainnet Prerequisites

Per the Phase 42M T4 oracle review, before mainnet:

- `ExternalOracle.MAX_DEV_THRESHOLD` is currently **event-only** — must become an **on-chain cap** with revert-on-breach.
- External oracle pushes should require **2-of-N** signer approval, not single-key.
- Both are external-oracle concerns; R44-2 itself ships as documented above.
