# Governance & Access Control

## 1. Overview

The protocol implements a comprehensive governance system with role-based access control, timelocked operations, and emergency procedures. Governance uses a three-multisig architecture (Treasury, Admin/Council, Guardian) with transparent on-chain operations and community oversight via Snapshot voting.

## 2. Governance Entities

### 2.1. Treasury Multisig (Governance Multisig)

**Contract**: `Treasury.sol` at `contracts/src/Treasury.sol`

**Initial Setup**: 3-of-5 multisig (core team)

**Long-term**: 3-of-5 elected DAO council

**Powers**:

- **Mint BTR tokens** to any address (immediate)
  ```solidity
  treasury.mintGovToken(recipient, amount);
  ```

- **Grant vested allocations** during TGE initialization
  ```solidity
  treasury.initializeTGE(
      treasuryAmount,
      seedingAmount,
      baseEmissionRate,
      halvingInterval,
      beneficiaries,
      allocations
  );
  ```

- **Allocate incentives** to pool distributors
  - Mint BTR to distributor contracts
  - Distributors handle per-pool distribution

- **Special allocations** for events, grants, partnerships
  - Direct minting to recipients
  - No vesting cliffs for special cases

- **Collect protocol fees** from pools
  ```solidity
  treasury.collectProtocolFees(admin, pool, token);
  ```

- **Transfer ownership** (Solady pull-pattern handover)
  ```solidity
  // newOwner: opens handover (Solady)
  requestOwnershipHandover();
  // current owner: finalize (zero-target rejected)
  completeOwnershipHandover(pendingOwner);
  ```
  `transferOwnership(address)` and `renounceOwnership()` are **disabled** (revert `NotAuth`) - handover is the only path.

**Cannot Do**:
- Burn tokens from other addresses
- Modify vesting schedules post-TGE
- Access pool governance functions
- Control bridge operations

### 2.2. Bridge Owner (DAO Council)

**Contract**: `Bridge.sol` at `contracts/src/Bridge.sol`

**Initial Setup**: 3-of-5 multisig (core team)

**Long-term**: 3-of-5 elected DAO council

**Powers**:

- **Configure bridgeable tokens** (2-day timelock)
  ```solidity
  // Initial setup (no timelock)
  bridge.setTokenConfig(token, limit, decimals, inRatio, unlimited);

  // Updates (2-day timelock)
  bridge.requestConfigChange(token, newLimit, decimals, newRatio, updateLimit, updateRatio);
  bridge.executeConfigChange(token);
  ```

- **Set cross-chain peers** (2-day timelock)
  ```solidity
  bridge.requestSetPeer(uint32 eid, bytes32 peer);
  bridge.executeSetPeer(eid);
  ```

- **Pause/unpause tokens** (immediate)
  ```solidity
  bridge.pauseToken(token, true);   // Emergency pause
  bridge.pauseToken(token, false);  // Unpause
  ```

- **Set daily bridge limits**
  - Outbound: Absolute amount per token
  - Inbound: Percentage of outbound (e.g., 120%)
  - B64-encoded for gas efficiency

- **Migrate to new bridges**
  - Deploy new bridge contract
  - Update token mint/burn permissions
  - Transition over grace period

- **Salvage stuck tokens** (immediate, owner-only sweep)
  ```solidity
  bridge.salvage(token, recipient, amount);
  treasury.salvage(token, recipient, amount);
  ```

**Cannot Do**:
- Mint tokens directly (only via ERC-7802 callbacks)
- Bypass rate limits
- Access pool or treasury functions

### 2.3. Pool Owner (Curator)

**Contract**: Any `Pool` clone (EIP-1167 minimal proxy deployed by `PoolFactory`)

**Initial Setup**:
- **Core Protocol Pools**: 3-of-5 DAO council multisig
- **Permissionless Pools**: Deployer's choice (any address)

**Long-term**:
- **Official Pools**: Elected DAO council
- **Permissionless Pools**: Pool creator's discretion

**Powers**:

#### 2.3.1. Asset Management (1-day timelock)

- **Add assets to pool**
  ```solidity
  pool.requestAddAsset(token, oracleConfig, riskConfig, liquidityProfile, minFeeBps, decimals);
  // ... wait 1 day
  pool.executeAddAsset(token);
  ```

- **Update risk configurations** (1-day timelock)
  ```solidity
  pool.requestUpdateRiskConfig(token, newRiskConfig);
  pool.executeUpdateRiskConfig(token);
  ```

  **RiskConfig Parameters**:
  - `flags`: FROZEN | SWAP_ENABLED | LIABILITY_SWAP_ENABLED | DECAY_ENABLED | FLASH_ENABLED | FEE_ON_TRANSFER | BRIDGEABLE  (bit 6 RESERVED)
  - `maxDepth`: Maximum liquidity profile depth
  - `decayStartRatioBps`: Liability decay threshold
  - `decaySlope`: Decay rate parameter
  - `maxDailyVolatility`: Circuit breaker threshold

- **Update liquidity profiles** (depth curve shapes)
  ```solidity
  pool.requestUpdateProfile(token, newProfile);
  pool.executeUpdateProfile(token);
  ```

  **LiquidityProfile Structure**:
  - `knots[17]`: Price offset knots (int8, weights.length + 1 points)
  - `weights[16]`: Liquidity weights per segment (uint8)
  - Total weights must sum to 200
  - Knots must be monotonically increasing
  - max(knots) - min(knots) must equal 100

#### 2.3.2. Oracle Management (2-day timelock)

- **Configure price oracles**
  ```solidity
  pool.requestOracleUpdate(token, oracleConfig);
  pool.executeOracleUpdate(token);
  ```

  **OracleConfig Parameters**:
  - `primary`: Primary oracle address (or `address(this)` for internal)
  - `secondary`: Fallback oracle address
  - `feedId`: Price feed identifier
  - `mode`: USE_INTERNAL | USE_EXTERNAL | ALLOW_FALLBACK

- **Update anchor relationships** (immediate)
  ```solidity
  pool.setAnchor(token, anchorToken);
  ```

  Controls pricing topology in anchor tree architecture.

#### 2.3.3. Fee Management (1-day timelock)

- **Update fee parameters**
  ```solidity
  pool.requestUpdateFeeParams(IPool.FeeParams({
      protoShare: 20,      // % to protocol
      flashFeeBps: 500     // Flash loan fee (0.05%)
  }));
  pool.executeUpdateFeeParams();
  ```

#### 2.3.4. Staking Parameters (2-day timelock)

- **Update staking lock durations**
  ```solidity
  pool.requestStakingConfigUpdate(
      stakeLockDuration,   // Lock after stake
      transferCooldown     // Lock after transfer
  );
  pool.executeStakingConfigUpdate();
  ```

- **Configure LP staking**
  ```solidity
  pool.updateStakingConfig(lpToken, salt);
  ```

  Deploys deterministic staked LP (sLP) token via CREATE3.

- **Pause/unpause staking** (immediate)
  ```solidity
  pool.pause();    // Emergency pause
  pool.unpause();  // Resume
  ```

#### 2.3.5. Distribution Management (2-day timelock)

- **Add reward tokens**
  ```solidity
  pool.addRewardToken(lpToken, rewardToken);
  ```

- **Set emission rates**
  ```solidity
  pool.updateDistributionRate(lpToken, rewardToken, ratePerSecond);
  ```

- **Update claim parameters** (2-day timelock)
  ```solidity
  pool.requestDistributionParamsUpdate(minClaimPower, matchingRatio);
  pool.executeDistributionParamsUpdate();
  ```

  **Parameters**:
  - `minClaimPower`: Minimum reward multiplier without sBTR (basis points)
  - `matchingRatio`: sBTR value needed to match LP value for full rewards

#### 2.3.6. Critical Operations (3-7 day timelock)

- **Transfer pool ownership** (3-day timelock)
  ```solidity
  pool.requestOwnershipTransfer(newOwner);
  pool.executeOwnershipTransfer();
  ```

- **Update modules** (3-day timelock)
  ```solidity
  pool.requestModuleUpdate(implAddress, selectors, initData);
  pool.executeModuleUpdate();
  ```

- **Update bridge address** (3-day timelock)
  ```solidity
  pool.requestBridgeUpdate(newBridge);
  pool.executeBridgeUpdate();
  ```

- **Update treasury address** (3-day timelock)
  ```solidity
  pool.requestTreasuryUpdate(newTreasury);
  pool.executeTreasuryUpdate();
  ```

- **Migrate base token** (7-day timelock)
  ```solidity
  pool.requestBaseMigration(newBaseToken);
  pool.executeBaseMigration();
  ```

#### 2.3.7. Emergency Powers (Immediate)

- **Freeze assets** (no timelock)
  ```solidity
  pool.freezeAsset(token);   // Disable all operations
  pool.unfreezeAsset(token); // Re-enable
  ```

- **Pause staking** (no timelock)
  ```solidity
  pool.pause();    // Disable stake/unstake
  pool.unpause();  // Re-enable
  ```

**Cannot Do**:
- Mint BTR tokens
- Control bridge operations
- Access other pools' governance
- Bypass timelocks (except emergency freezes)

### 2.4. External Oracle Keeper

**Role**: Off-chain EOA or multisig

**Contracts**: `ExternalOracle.sol` instances

**Initial Setup**: Keeper EOA (can be upgraded to multisig)

**Long-term**: Decentralized keeper network or DAO-controlled multisig

**Powers**:

- **Push price feeds on-chain**
  ```solidity
  oracle.updatePrice(token, price, volatility, timestamp);
  ```

- **Update multiple feeds** (batch operation)
  ```solidity
  oracle.updatePriceBatch(tokens, prices, volatilities, timestamp);
  ```

**Responsibilities**:

- Fetch prices from reliable off-chain sources
- Calculate TWAP/EMA to prevent manipulation
- Update at regular intervals (e.g., every 5 minutes)
- Monitor for price anomalies
- Maintain high uptime

**Security Requirements**:

- **Use TWAP prices**: Not spot prices (flash loan protection)
- **Multi-source aggregation**: Combine multiple exchanges
- **Outlier detection**: Reject suspicious price movements
- **Transparency**: Public monitoring of price updates
- **Decentralization**: Multiple keepers with rotation

**Critical Importance**:

> External oracles are **mission-critical** for pools using them. Stale or manipulated prices can lead to:
> - Incorrect swap rates
> - Arbitrage exploitation
> - Insolvency risk
> - User losses

**Recommendation for Permissionless Pools**:

- **Prefer internal oracles** (Curve-style TWAPs)
- Use external oracles only if:
  - Highly reputable (Chainlink, Pyth, etc.)
  - Multiple fallbacks configured
  - Circuit breakers enabled
  - You trust the keeper

## 3. Timelock Overview

All timelocks follow this pattern:

1. **Request**: Owner initiates operation
   - Emits `TimelockRequested(id, opType, executeAfter)`
   - Stores operation data on-chain

2. **Wait**: Time passes (1, 2, 3, or 7 days)
   - Users can review pending operations
   - Community can discuss in forum
   - Exit window for concerned users

3. **Execute**: Owner executes after timelock expires
   - Must be within grace period
   - Emits operation-specific event
   - Changes take effect

4. **Cancel** (optional): Owner cancels before execution
   - Emits `TimelockCancelled(id, opType)`
   - Clears pending operation

### 3.1. Timelock Summary Table

| Risk Level | Delay | Operations |
|------------|-------|------------|
| **Emergency** | None | Freeze asset, pause staking, pause bridge token |
| **Low** | 1 day | Add asset, update risk config, update fees |
| **Medium** | 2 days | Oracle config, staking params, distribution params, bridge config |
| **High** | 3 days | Ownership transfer, module update, bridge/treasury update |
| **Critical** | 7 days | Base token migration, treasury ownership |

**Grace Periods**:
- Pool operations: 7 days
- Bridge operations: 3 days
- Treasury operations: 7 days

## 4. Multi-Signature Requirements

All governance entities should use multisig wallets:

### 4.1. Recommended Setup (Launch)

**Governance Multisig (Treasury)**:
- **Threshold**: 3-of-5
- **Signers**: Core team members
- **Key Storage**: Hardware wallets (Ledger, Trezor)
- **Geographic Distribution**: Different jurisdictions
- **Backup Plans**: Key recovery procedures

**DAO Council Multisig (Bridge + Official Pools)**:
- **Threshold**: 3-of-5
- **Signers**: Core team initially, elected members later
- **Key Storage**: Hardware wallets
- **Geographic Distribution**: Different jurisdictions
- **Meeting Cadence**: Weekly sync calls

### 4.2. Long-Term Setup (Post-DAO)

**Elected Council**:
- **Selection**: Token-weighted voting
- **Term**: 6-12 months
- **Accountability**: On-chain voting records
- **Compensation**: Protocol emissions
- **Removal**: Governance vote if malicious

## 5. Emergency Procedures

### 5.1. Asset Exploitation Detected

1. **Immediate**: Freeze affected asset
   ```solidity
   pool.freezeAsset(exploitedToken);
   ```

2. **Within 1 hour**: Assess impact
   - Check reserves vs liabilities
   - Identify affected users
   - Estimate losses

3. **Within 24 hours**: Public disclosure
   - Post-mortem to governance forum
   - Explanation of freeze reason
   - Proposed remediation plan

4. **Timelock**: Update risk config or remove asset
   ```solidity
   pool.requestUpdateRiskConfig(token, saferConfig);
   // or
   pool.requestRemoveAsset(token);
   ```

5. **After resolution**: Unfreeze if safe
   ```solidity
   pool.unfreezeAsset(token);
   ```

### 5.2. Bridge Exploit Detected

1. **Immediate**: Pause affected token
   ```solidity
   bridge.pauseToken(token, true);
   ```

2. **Coordinate**: Contact other chain councils
   - Pause on all chains
   - Investigate root cause

3. **If compromise is severe**: Deploy new bridge
   - New contract deployment
   - Update token permissions
   - Migrate liquidity

### 5.3. Oracle Manipulation Detected

1. **Immediate**: Freeze assets using bad oracle
   ```solidity
   pool.freezeAsset(affectedToken);
   ```

2. **Switch to fallback**: If available
   ```solidity
   pool.requestOracleUpdate(token, fallbackConfig);
   ```

3. **Investigate**: Oracle keeper
   - Check keeper signing keys
   - Review recent price updates
   - Identify manipulation method

4. **Long-term**: Replace keeper or switch to internal oracle

## 6. Governance Architecture

### 6.1. Off-Chain Voting with On-Chain Execution

AIMM uses **Snapshot-based governance** rather than fully on-chain voting. This design choice is intentional:

**Why not on-chain governance?**
- **Cross-chain compatibility** -Voting power spans multiple chains (Ethereum, Arbitrum, Base, etc.)
- **Liquid staking** -sBTR/sLP are transferable; on-chain voting would require lock mechanisms
- **Gas efficiency** -Off-chain voting is free; on-chain proposals cost gas
- **UX simplicity** -Snapshot provides a familiar, user-friendly interface

**Transparency guarantees:**
- All voting power derived from on-chain state at snapshot block
- Multisig execution of approved proposals is publicly visible
- Timelocks provide community exit opportunity
- All parameters and operations logged on-chain

> For complete governance mechanics, see [DAO Voting](../../protocol/07.%20Governance.md).

## 7. Access Control Architecture

### 7.1. Role Hierarchy

```
Treasury Owner (Governance Multisig)
├── Mint BTR tokens
├── Initialize TGE
├── Collect protocol fees
└── Transfer ownership (7-day timelock)

Bridge Owner (DAO Council)
├── Configure token bridging (2-day timelock)
├── Set cross-chain peers (2-day timelock)
├── Pause tokens (immediate)
└── Salvage stuck tokens (immediate, owner-only sweep)

Pool Owner (Curator)
├── Asset Management
│   ├── Add assets (1-day timelock)
│   ├── Update risk configs (1-day timelock)
│   └── Freeze assets (immediate)
├── Oracle Management (2-day timelock)
├── Fee Management (1-day timelock)
├── Staking Management (2-day timelock)
├── Distribution Management (2-day timelock)
├── Critical Operations (3-7 day timelock)
│   ├── Module upgrades (3-day)
│   ├── Ownership transfer (3-day)
│   └── Base token migration (7-day)
└── Emergency Powers (immediate)

External Oracle Keeper
├── Update price feeds
└── Monitor oracle health
```

### 7.2. Permission Verification

All privileged functions use the `onlyOwner` modifier:

```solidity
modifier onlyOwner() {
    if (msg.sender != $.owner) revert OnlyOwner();
    _;
}
```

Treasury uses Solady's `Ownable` pattern for consistency.

## 8. Monitoring & Transparency

### 8.1. On-Chain Events

All governance actions emit events for transparency:

```solidity
// Timelock operations
event TimelockRequested(bytes32 indexed id, uint8 opType, uint48 executeAfter);
event TimelockCancelled(bytes32 indexed id, uint8 opType);

// Asset management
event AssetAdded(address indexed token, uint8 decimals, uint128 minLiquidity);
event RiskConfigUpdated(address indexed token, uint16 flags, uint128 maxVolume);
event EmergencyFreeze(address indexed token);

// Module updates
event ModulesUpdated(address indexed impl, bytes4[] selectors);

// Ownership changes
event OwnershipTransferred(address indexed oldOwner, address indexed newOwner);
```

### 8.2. Recommended Monitoring

**For Users**:
- Subscribe to `TimelockRequested` events
- Review pending operations via governance dashboard
- Exit positions if suspicious changes proposed

**For Council Members**:
- Monitor oracle price updates
- Track daily bridge volumes
- Review protocol fee accumulation
- Watch for emergency freezes

**For Developers**:
- Track module upgrades
- Monitor storage layout changes
- Test against staging deployments

## 9. Best Practices

### 9.1. For Governance Multisig Signers

1. **Use hardware wallets**: Never sign from hot wallets
2. **Verify transactions**: Double-check addresses and amounts
3. **Communicate clearly**: Discuss proposals before signing
4. **Document decisions**: Keep records of reasoning
5. **Rotate keys**: Periodically update signer set

### 9.2. For Pool Operators

1. **Announce changes**: Post to forum before timelock
2. **Explain rationale**: Why is change needed?
3. **Test on testnet**: Verify expected behavior
4. **Monitor post-change**: Watch for unexpected effects
5. **Have rollback plan**: Be ready to revert if needed

### 9.3. For Oracle Keepers

1. **Use TWAP prices**: Never push spot prices
2. **Multi-source data**: Aggregate multiple exchanges
3. **Outlier detection**: Reject suspicious prices
4. **High uptime**: Keep feed fresh (<5 min delay)
5. **Transparency**: Public monitoring dashboards
