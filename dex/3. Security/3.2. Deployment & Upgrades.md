# Deployment & Upgradeability

## 1. Overview

The protocol uses a modular diamond-lite proxy pattern for upgradeable pools and standalone contracts for treasury and bridge. This architecture balances upgradeability with security through timelocked governance.

## 2. Architecture

### 2.1. Core Components

1. **PoolProxy** (`contracts/src/PoolProxy.sol`)
   - Lightweight EIP-2535-inspired proxy
   - Routes function calls to registered modules
   - ERC-7201 storage pattern for collision resistance
   - Immutable deployer for initialization security

2. **Modules** (Implementation contracts)
   - `Admin.sol`: Administrative functions
   - `Core.sol`: Swap, deposit, withdraw logic
   - `Flash.sol`: Flash loan operations
   - `Staking.sol`: BTR and LP staking
   - `Distributor.sol`: Reward distribution
   - `Oracle.sol`: Price feed management

3. **Standalone Contracts**
   - `Treasury.sol`: BTR token management, TGE, vesting
   - `Bridge.sol`: Cross-chain token bridging

## 3. Deployment Process

### 3.1. Phase 1: Deploy Core Infrastructure

```solidity
// 1. Deploy Treasury
Treasury treasury = new Treasury(
    governanceMultisig,  // owner
    btrToken             // BTR address
);

// 2. Deploy Bridge
Bridge bridge = new Bridge(
    daoCouncilMultisig,  // owner
    lzEndpoint           // LayerZero V2 endpoint
);

// 3. Deploy Module Implementations
Admin adminImpl = new Admin();
Core coreImpl = new Core();
Flash flashImpl = new Flash();
Staking stakingImpl = new Staking();
Distributor distributorImpl = new Distributor();
Oracle oracleImpl = new Oracle();
```

### 3.2. Phase 2: Deploy Pool via Factory Pattern

```solidity
// 1. Deploy PoolProxy
PoolProxy pool = new PoolProxy();

// 2. Initialize pool (must be done by deployer in same transaction)
pool.initialize(
    curatorMultisig,     // owner/curator
    USDC,                // base token
    WETH,                // wrapped native
    IPool.FeeParams({
        protoShare: 20,    // 20% to protocol
        flashFeeBps: 500   // 0.05% flash loan fee
    })
);

// 3. Register modules (creates function selector → implementation mapping)
bytes4[] memory adminSelectors = [
    IAdmin.freezeAsset.selector,
    IAdmin.unfreezeAsset.selector,
    IAdmin.requestAddAsset.selector,
    IAdmin.executeAddAsset.selector,
    // ... all admin function selectors
];

bytes4[] memory coreSelectors = [
    ICore.swap.selector,
    ICore.deposit.selector,
    ICore.withdraw.selector,
    // ... all core function selectors
];

// Initial module setup (no timelock for first-time setup)
Admin(address(pool)).executeModuleUpdate(
    address(adminImpl),
    adminSelectors,
    "" // no init data
);

Admin(address(pool)).executeModuleUpdate(
    address(coreImpl),
    coreSelectors,
    "" // no init data
);

// ... repeat for all modules
```

### 3.3. Phase 3: Connect Dependencies

```solidity
// Set treasury address in pool (HIGH_TIMELOCK = 3 days)
Admin(address(pool)).requestTreasuryUpdate(address(treasury));
// ... wait 3 days
Admin(address(pool)).executeTreasuryUpdate();

// Set bridge address in pool (HIGH_TIMELOCK = 3 days)
Admin(address(pool)).requestBridgeUpdate(address(bridge));
// ... wait 3 days
Admin(address(pool)).executeBridgeUpdate();
```

## 4. Upgradeability Mechanisms

### 4.1. Module Upgrades

Modules can be upgraded or replaced via **HIGH_TIMELOCK (3 days)**:

```solidity
// 1. Deploy new module implementation
CoreV2 newCore = new CoreV2();

// 2. Request module update
Admin(pool).requestModuleUpdate(
    address(newCore),
    coreSelectors,
    initData  // optional initialization call
);

// 3. Wait 3 days + execute
Admin(pool).executeModuleUpdate();
```

**Key Properties**:
- Selectors can point to different implementations
- Multiple selectors can share one implementation
- Init data allows constructor-like setup
- Previous implementation remains unchanged

### 4.2. Storage Persistence

All upgrades preserve storage via **ERC-7201 namespaced storage**:

```solidity
// Storage location is deterministic and collision-resistant
bytes32 constant CORE_STORAGE_LOC = keccak256(
    abi.encode(uint256(keccak256("pool.storage.base.v1")) - 1)
) & ~bytes32(uint256(0xff));

function _s() internal pure returns (PoolStorage storage $) {
    assembly { $.slot := CORE_STORAGE_LOC }
}
```

**Guarantees**:
- Storage layout preserved across upgrades
- No storage collision between modules
- Standard ERC-7201 pattern

### 4.3. Adding New Functions

New functionality can be added without affecting existing functions:

```solidity
// Deploy module with new feature
NewFeature newFeatureImpl = new NewFeature();

// Register new selectors
bytes4[] memory newSelectors = [
    INewFeature.newFunction.selector
];

Admin(pool).requestModuleUpdate(
    address(newFeatureImpl),
    newSelectors,
    initData
);
```

### 4.4. Removing Functions

Selectors can be unregistered by pointing them to `address(0)`:

```solidity
bytes4[] memory deprecatedSelectors = [
    IDeprecated.oldFunction.selector
];

Admin(pool).requestModuleUpdate(
    address(0),         // Zero address = remove
    deprecatedSelectors,
    ""
);
```

## 5. Non-Upgradeable Components

The following components are **immutable by design**:

1. **PoolProxy Fallback Logic**
   - Selector routing is hardcoded
   - Only module addresses can change
   - Prevents rug-pull attacks

2. **Treasury Contract**
   - Cannot be upgraded
   - Ownership transferrable only
   - Protects vesting schedules

3. **Bridge Contract**
   - Cannot be upgraded
   - Migration requires new deployment
   - Rate limits protect against exploits

4. **ERC-7201 Storage Locations**
   - Hardcoded constants
   - Ensures storage safety

## 6. Security Through Timelocks

All dangerous operations require time delays:

**CRITICAL**

| | | | |
|---|---|---|---|
| Base token migration | 7 days | 7 days | No |

**HIGH RISK**

| | | | |
|---|---|---|---|
| Ownership transfer | 3 days | 7 days | No |
| Module update | 3 days | 7 days | No |
| Bridge address update | 3 days | 7 days | No |
| Treasury address update | 3 days | 7 days | No |

**MEDIUM RISK**

| | | | |
|---|---|---|---|
| Oracle config change | 2 days | 7 days | No |
| Staking params update | 2 days | 7 days | No |
| Distribution params | 2 days | 7 days | No |
| Bridge peer config | 2 days | 3 days | No |
| Bridge token config | 2 days | 3 days | No |

**LOW RISK**

| | | | |
|---|---|---|---|
| Add asset | 1 day | 7 days | No |
| Update risk config | 1 day | 7 days | No |
| Update fee params | 1 day | 7 days | No |

**EMERGENCY**

| | | | |
|---|---|---|---|
| Freeze asset | None | N/A | N/A |
| Pause staking | None | N/A | N/A |
| Pause bridge token | None | N/A | N/A |

### 6.1. Timelock Mechanics

```solidity
// Request operation
requestOperation(...);
// State: PENDING
// Emits: TimelockRequested(id, opType, executeAfter)

// Wait for timelock period
// ...

// Execute within grace period
executeOperation(...);
// State: EXECUTED
// Emits: OperationExecuted(id, opType)

// OR cancel before execution
cancelOperation(...);
// State: CANCELLED
// Emits: TimelockCancelled(id, opType)
```

**Grace Period Protection**:
- Operations expire after `timelock + gracePeriod`
- Prevents indefinite pending operations
- Requires re-initiation if expired

## 7. Deployment Configurations

### 7.1. Mainnet Deployment

**Treasury Owner**: 3-of-5 Governance Multisig
- Core team members initially
- Transition to elected DAO council

**Bridge Owner**: 3-of-5 DAO Council Multisig
- Same signers as governance initially
- Separate keys long-term

**Pool Owner** (Official Pools): 3-of-5 DAO Council Multisig
- Controls core protocol pools
- Sets asset listings, risk params

**Pool Owner** (Permissionless): Any address
- Users can deploy their own pools
- Full control over their pool
- No protocol control

### 7.2. Testnet Deployment

Shorter timelocks for testing:
- CRITICAL: 1 hour
- HIGH: 30 minutes
- MEDIUM: 15 minutes
- LOW: 5 minutes

**Not recommended for production value**.

## 8. Permissionless Pool Deployment

Anyone can deploy a pool via factory pattern:

```solidity
// 1. Deploy PoolProxy
PoolProxy myPool = new PoolProxy();

// 2. Initialize with yourself as owner
myPool.initialize(
    msg.sender,  // You are the curator
    baseToken,
    wnative,
    feeParams
);

// 3. Register modules
// ... (same as Phase 2 above)

// 4. Configure your pool
Admin(address(myPool)).requestAddAsset(...);
```

**Properties**:
- Full control over your pool
- No protocol governance
- Timelocks still enforced (protects users)
- Can use own oracles
- Can set own fee parameters

**Recommended**:
- Use **internal oracles only** (Curve-style TWAPs)
- Avoid external oracle dependencies
- Set conservative risk parameters
- Test thoroughly on testnet

## 9. Best Practices

### 9.1. For Module Developers

1. **Never modify storage layout**
   - Only append new storage variables
   - Use ERC-7201 namespaced slots
   - Document storage in comments

2. **Use initialization functions**
   - Not constructors (delegatecall context)
   - Include `initData` in module updates
   - Check initialization state

3. **Maintain interface compatibility**
   - Don't change function signatures
   - Add new functions, don't modify existing
   - Use versioned interfaces

4. **Test with actual proxy**
   - Don't test implementation directly
   - Use full deployment setup
   - Test upgrade paths

### 9.2. For Pool Operators

1. **Use multisig for ownership**
   - Minimum 3-of-5 signers
   - Distributed key custody
   - Hardware wallet signing

2. **Announce changes in advance**
   - Post to governance forum
   - Allow community review
   - Explain rationale

3. **Test on staging first**
   - Deploy to testnet
   - Run full scenarios
   - Monitor for issues

4. **Monitor post-upgrade**
   - Watch events
   - Check state consistency
   - Have rollback plan

### 9.3. For Users

1. **Verify module addresses**
   - Check on-chain via `getModule(selector)`
   - Compare against official deployments
   - Use block explorers

2. **Monitor timelock events**
   - Subscribe to `TimelockRequested`
   - Review pending operations
   - Exit if suspicious

3. **Understand risks**
   - Upgradeable pools have governance risk
   - Timelocks provide exit window
   - Emergency freezes possible

## 10. Emergency Procedures

### 10.1. Asset Freeze (No Timelock)

```solidity
// Instantly disable all operations on an asset
Admin(pool).freezeAsset(vulnerableToken);
```

**Effect**: Disables swap, deposit, withdraw, flash loans immediately.

### 10.2. Staking Pause (No Timelock)

```solidity
// Instantly pause all staking operations
Staking(pool).pause();
```

**Effect**: Disables stake/unstake while allowing reward claims.

### 10.3. Bridge Token Pause (No Timelock)

```solidity
// Instantly pause bridging for a token
Bridge(bridge).pauseToken(token, true);
```

**Effect**: Prevents inbound and outbound bridging immediately.

## 11. Upgrade Checklist

Before deploying an upgrade:

- [ ] Storage layout unchanged or safely extended
- [ ] All selectors mapped correctly
- [ ] Initialization data prepared
- [ ] Timelock announced to community
- [ ] Testnet upgrade successful
- [ ] Security audit completed (for major changes)
- [ ] Emergency freeze procedures documented
- [ ] Monitoring alerts configured
- [ ] Rollback plan documented
