# Deployment & Upgradeability

## 1. Overview

Phase 42H removed the Diamond proxy. The protocol now deploys as:

1. A small set of **standalone singleton contracts** (`Admin`, `Staking`, `Distributor`, `Flash`, `PoolFactory`) plus the shared `AccessControl`, `Treasury`, and `Bridge`.
2. **Per-pool EIP-1167 minimal-proxy clones** of a reference `Pool` impl, deployed via `PoolFactory`. Each clone has its own storage; the cloned bytecode is immutable.

Upgradeability is achieved via:
- **Reference-impl swap** at `PoolFactory` (7-day timelock) -affects only future clones.
- **UUPS** on `Treasury`, `Bridge`, and the shared `AccessControl` -gated by owner + timelock.
- **Singleton replacement** by re-deployment (no in-place upgrade) -old singletons remain functional; the system migrates by updating the pool-side reference.

There is no `delegatecall` between protocol contracts. Cross-contract interactions are normal external calls.

## 2. Architecture

### 2.1. Core Components

1. **Pool reference impl** (`dex/evm/src/Pool.sol`)
   - Standalone AIMM contract. Inherits internal TWAP oracle.
   - Deployed once per protocol version; never called directly by users.
   - Each clone is an EIP-1167 minimal proxy of this impl, initialised via `initialize(...)`.

2. **PoolFactory** (`dex/evm/src/PoolFactory.sol`)
   - Deploys `Pool` clones (`createPool(...)`).
   - Holds the reference-impl swap timelock (`pendingReferencePool`, `upgradeTimelock`).
   - Maintains `allPools` + `officialPools` + `isPool` registries.

3. **Singletons** -each holds an immutable `AC` (AccessControl) ref and a `mapping(pool, ...)` keying for per-pool state:
   - `Admin.sol` -per-pool timelock queue + restricted setters
   - `Staking.sol` -governance + LP staking
   - `Distributor.sol` -token-only campaign Merkle distribution
   - `Flash.sol` -ERC-3156 flash-loan provider (stateless)

4. **Shared singletons** (live under `shared/`)
   - `AccessControl.sol` -single owner source of truth
   - `Treasury.sol` -BTR token TGE + vesting + fee collection (UUPS)
   - `Bridge.sol` -LayerZero v2 OApp (UUPS) with cancel paths for queued config/peer changes

## 3. Deployment Process

### 3.1. Phase 1: Deploy Shared + Singletons

```solidity
// 1. Deploy AccessControl (UUPS, init owner)
AccessControl ac = deployUUPS(new AccessControl(), ownerInit);

// 2. Deploy Treasury (UUPS)
Treasury treasury = deployUUPS(new Treasury(), treasuryInit);

// 3. Deploy Bridge (UUPS, LayerZero v2 endpoint)
Bridge bridge = deployUUPS(new Bridge(), bridgeInit(lzEndpoint));

// 4. Deploy singletons -each wires the shared AC immutably.
Admin admin = new Admin(address(ac));
Staking staking = new Staking(address(ac));
Distributor distributor = new Distributor(address(ac));
Flash flash = new Flash();
```

### 3.2. Phase 2: Deploy Pool Reference + Factory

```solidity
// 1. Deploy the reference Pool impl (the impl bytecode; never called by users).
Pool poolImpl = new Pool();

// 2. Deploy the factory; wires the AC + initial reference impl.
PoolFactory factory = new PoolFactory(address(ac), address(poolImpl));
```

### 3.3. Phase 3: Deploy a Pool Clone

```solidity
// Permissionless. Caller-supplied config.
address pool = factory.createPool(
    owner,              // pool curator (must be the AC-owned authority or self for permissionless)
    baseToken,          // anchor token (e.g. USDC)
    wnative,            // wrapped native (WETH)
    feeParams           // FeeParams(protoShare, flashFeeBps)
);
// Pool is now an initialised EIP-1167 clone of poolImpl.
```

### 3.4. Phase 4: Wire Per-Pool Config

```solidity
// All sensitive config is timelocked at the Admin singleton; opIds are namespaced per-pool.
admin.requestAddAsset(pool, token, riskCfg);
// ... wait 1 day
admin.executeAddAsset(pool, token);
```

## 4. Upgrade Mechanisms

### 4.1. Pool Reference-Impl Swap (PoolFactory, 7-day timelock)

```solidity
// 1. Deploy new reference impl.
Pool poolImplV2 = new Pool();

// 2. Queue swap.
factory.requestReferenceUpgrade(address(poolImplV2));

// 3. Wait 7 days, then execute.
factory.executeReferenceUpgrade();

// OR cancel before exec.
factory.cancelReferenceUpgrade();
```

**Key Properties**:
- Affects **future** clones only. Existing clones run their original impl bytecode forever.
- No grace window: governance must explicitly cancel; the queue does not silently expire.
- `PoolStorage` field order/types are **append-only** across versions to keep clone storage layouts forward-compatible.

### 4.2. UUPS Upgrades (Treasury / Bridge / AccessControl)

```solidity
treasury.upgradeToAndCall(newImpl, initData);
```

- Authorisation enforced via `AccessControl.owner()`.
- Upgrade-pending flag held in EIP-1153 transient storage (Phase 42H ﾂｷ G18) so it cannot persist across unrelated calls.

### 4.3. Singleton Replacement

`Admin` / `Staking` / `Distributor` / `Flash` are **not** upgradeable in place. To replace one:

1. Deploy a new version pointing at the same `AC`.
2. Migrate per-pool state (typically by re-enacting config on the new singleton).
3. Update pool-side references where applicable.
4. Old singleton remains deployed but is no longer the canonical entry point.

This is intentional -replacing rather than upgrading keeps the audit surface small and makes every change visible at a fresh address.

## 5. Non-Upgradeable Components

- All AIMM library contracts (`libraries/Pricing.sol`, `PoolOracle.sol`, `Spline.sol`, `Maths.sol`, `AnchorTree.sol`, `PoolDecay.sol`, `PoolAdmin.sol`, `Constants.sol`).
- Each deployed `Pool` clone (immutable per-instance).
- `PoolFactory` itself.
- Singleton contracts (replaced rather than upgraded).
- Shared `Constants.sol`, `Errors.sol`, `Timelock.sol`.

## 6. Security Through Timelocks

All dangerous operations require time delays. See [ADR-002 Timelock Encodings](/docs/shared/43-ADR-002-Timelock-Encodings) for the four encoding variants and rationale.

**CRITICAL (7 days)**
- Pool reference-impl swap (factory)
- Base token migration

**HIGH (3 days)**
- Ownership transfer (Treasury, AccessControl)
- Bridge / Treasury address update at Pool
- Bridge upgrade (UUPS)

**BASE (2 days)**
- Oracle config change
- Staking param update
- Bridge peer set / config change (with cancel path)

**LOW (1 day)**
- Add asset
- Update risk config
- Update fee params

**EMERGENCY (no timelock)**
- `Admin.freezeAsset(pool, token)`
- `Staking.pause(pool)`
- `Bridge.pauseToken(token, true)`

### 6.1. Timelock Mechanics (per `shared/Timelock.sol`)

```solidity
// Most singletons key requests by (pool, opId).
requestOperation(...);    // emits TimelockRequested(id, opId, eta)
// wait...
executeOperation(...);    // emits OperationExecuted(id, opId)
// or
cancelOperation(...);     // emits TimelockCancelled(id, opId)
```

**Grace Period**: most ops auto-expire after `eta + grace`. `PoolFactory.referenceUpgrade` is the exception -see ﾂｧ4.1.

## 7. Deployment Configurations

### 7.1. Mainnet

- **AccessControl owner**: 3-of-5 governance multisig (transitions to elected DAO council).
- **Treasury owner**: same as governance multisig.
- **Bridge owner**: 3-of-5 DAO council multisig (separate keys long-term).
- **Pool curator (official pools)**: 3-of-5 DAO council multisig.
- **Pool curator (permissionless pools)**: any address; full control, but timelocks still enforced for user protection.

### 7.2. Testnet

Shortened timelocks for testing: CRITICAL 1h / HIGH 30m / BASE 15m / LOW 5m. Not for production value.

## 8. Permissionless Pool Deployment

```solidity
address myPool = factory.createPool(msg.sender, baseToken, wnative, feeParams);
admin.requestAddAsset(myPool, token, riskCfg);
// ... wait timelock, then execute.
```

- Pool curator has full control over their pool's assets and risk parameters.
- Timelocks are enforced even on permissionless pools (protects pool users).
- Curators can use internal oracle only or wire external feeds; conservative defaults recommended.

## 9. Best Practices

### 9.1. For Pool Reference-Impl Developers

1. **Storage layout is append-only.** `PoolStorage` field order and types are frozen across reference-impl versions. New fields appended only.
2. **No constructors with side effects.** `Pool` is cloned; only `initialize(...)` runs per-instance.
3. **No new external dependencies without an explicit migration story.** Existing clones cannot adopt new dependencies, so any new dep must be deployable per-impl.
4. **Test the reference impl AND a clone.** A clone is what users hit; impl in isolation will skip `delegatecall`-style edge cases (and confirm there should be none).

### 9.2. For Pool Operators

1. Use a multisig (竕･3-of-5) for curator authority.
2. Announce config changes in advance; the timelock window is the community's review period.
3. Test on testnet first.
4. Monitor `TimelockRequested` / `OperationExecuted` / `TimelockCancelled` events for your pool.

### 9.3. For Users

1. **Verify pool address and curator.** Query `factory.isOfficialPool(pool)` for protocol pools; for permissionless pools, inspect the curator directly.
2. **Monitor timelock events.** Subscribe to `TimelockRequested` for your pool; review pending ops; exit if suspicious.
3. **Understand the upgrade model.**
   - Your specific `Pool` clone is immutable.
   - The factory can change the reference impl, but it will not retroactively affect existing clones.
   - Singletons (`Admin`, `Staking`, etc.) can be replaced by the AC owner; track those addresses.

## 10. Emergency Procedures

### 10.1. Asset Freeze (No Timelock)

```solidity
admin.freezeAsset(pool, vulnerableToken);
```
Disables swap, deposit, withdraw, flash loans on that asset immediately.

### 10.2. Staking Pause (No Timelock)

```solidity
staking.pause(pool);
```
Disables stake/unstake; reward claims continue.

### 10.3. Bridge Token Pause (No Timelock)

```solidity
bridge.pauseToken(token, true);
```
Prevents inbound and outbound bridging immediately.

### 10.4. Cancel Queued Bridge Change

```solidity
bridge.cancelSetPeer(eid);
bridge.cancelConfigChange(configType);
```
Retires an in-flight queued upgrade without waiting for its grace window (Phase 42H ﾂｷ G21).

## 11. Upgrade Checklist

- [ ] `PoolStorage` layout unchanged or safely extended (append-only).
- [ ] No constructor side effects in the reference impl.
- [ ] Reference-impl test suite green against a deployed clone (`forge test` 281/281).
- [ ] Timelock requested + announced to community.
- [ ] Testnet swap executed and observed.
- [ ] ABI regen run in `~/Work/btr/sdk` and `abi-freshness` test green.
- [ ] Emergency freeze paths verified.
- [ ] Monitoring + alerting wired to the new impl + factory.
- [ ] Cancel path tested (so the swap can be aborted if needed).
