# White Labelling

> Deploying branded BTR Protocol instances

---

## 1. Overview

White labelling enables protocols and exchanges to deploy their own branded pool instances while using the BTR Protocol infrastructure.

**Key Principle**: Permissionless deployment -anyone can deploy a pool, no approval required.

> For AMM-specific pool deployment details (tokens, hooks, modules), see [Curation & Pool Deployment](../1.%20AIMM/1.3.%20Integration/1.3.3.%20Curation%20&%20Pool%20Deployment.md).

---

## 2. Deployment Model

### 2.1. Pool Clones via Factory

Each deployment is an EIP-1167 minimal-proxy clone of the canonical `Pool` reference impl, deployed atomically by `PoolFactory`:

```solidity
address pool = factory.createPool(
    owner,           // Your protocol address (curator authority)
    baseToken,       // Anchor token (e.g., USDC)
    wnative,         // Wrapped native (WETH, etc.)
    FeeParams(25, 5) // Protocol fee config
);
// `factory.createPool` atomically deploys the clone and calls `initialize`.
```

**Characteristics**:
- One factory deploys + registers every pool (`factory.allPools` / `factory.isPool`).
- Per-instance immutable bytecode (no in-place upgrade).
- Initialisation cannot be front-run -`createPool` wires deployer + owner in a single call.
- Reference-impl swaps at the factory affect only **future** clones; existing clones are untouched.

### 2.2. Ownership

| Action | Timelock |
|--------|----------|
| Freeze/unfreeze asset | None (emergency) |
| Add asset | 1 day |
| Update risk config | 1 day |
| Register module | 3 days |
| Transfer ownership | 3 days |
| Migrate base token | 7 days |

## 3. Branding Integration

### 3.1. Frontend

```typescript
import { Pool } from '@btr-protocol/sdk';

// Your branded pool client
const pool = new Pool(yourPoolAddress);

// Swap
const quote = await pool.getSwapQuote(tokenIn, tokenOut, amount);
const tx = await pool.swap(tokenIn, tokenOut, amount, minOut);

// Provide liquidity
const depositTx = await pool.deposit(token, amount);
```

### 3.2. UI Customization

- Design your own interface
- Point to your pool address
- Apply your branding (colors, logo, theme)
- Keep "Powered by BTR Protocol" attribution

## 4. Customization Options

### 4.1. Token Selection

Curate your own token list via asset management:

```solidity
pool.requestAddAsset(
    token,
    OracleConfig(address(pool), address(0), 0, 0, 6), // Internal TWAP
    RiskConfig(9800, 5000, 31709791, 10000, SWAP_ENABLED),
    LiquidityProfile(...),
    10,  // 0.1% min fee
    6,   // Decimals
    priceB64,
    volFast,
    volSlow
);
```

### 4.2. Hooks

Deploy custom hooks for specialized behavior:

```solidity
pool.setHook(token, address(customHook));
```

**Hook Use Cases**:
- Deposit bonuses
- Withdrawal fees
- Dynamic fee adjustment
- KYC/whitelist gates

> See [Hooks](../1.%20AIMM/1.3.%20Integration/1.3.2.%20Hooks.md) for complete hook development guide.

### 4.3. Modules

Register custom modules via diamond-lite pattern:

```solidity
bytes4[] memory selectors = [ /* your function selectors */ ];
pool.requestModuleUpdate(address(customModule), selectors, initData);
```

## 5. Attribution Requirements

When white-labelling, we ask that you:

1. **Credit the protocol**: "Powered by BTR Protocol" or similar
2. **Link to docs**: Direct interested users to official documentation
3. **Maintain security**: Don't modify pool logic (use hooks/modules)
4. **Keep open**: Avoid artificial barriers to token lists
