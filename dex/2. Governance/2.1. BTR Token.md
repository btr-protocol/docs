# BTR Token

> Protocol governance token with fixed supply, liquid staking, and quadratic voting

---

## 1. Overview

BTR (Bayesian True Range) is the protocol's governance and utility token.

| Property | Value |
|----------|-------|
| **Max Supply** | 100,000,000 BTR (hard-coded) |
| **Type** | ERC-20 |
| **Minting** | Treasury-controlled only |
| **Standard** | Solidity 0.8.28+ |

## 3. Token Utility

### 3.1. Governance Voting

Staked BTR (sBTR) provides voting power for:
- Protocol parameter changes
- Treasury spending proposals
- Module upgrades
- Asset listings

> For complete voting formula including quadratic damping, see [DAO Voting §2](/docs/2.2-DAO-Voting#2-voting-power-formula).

### 3.2. Reward Boosting

sBTR holders receive amplified LP rewards through the matching mechanism:

```
claimPower = min(1.0, sBTR_value × matchingRatio / sLP_value)
effectiveStake = lpStake × claimPower
```

**Example**:
- 1000 sLP staked ($1000 value)
- 100 sBTR staked ($100 value)
- Matching ratio: 5:1

```
coverage = $100 × 5 / $1000 = 50%
claimPower = 50% + (50% × minClaimPower) = 75%
effectiveStake = $1000 × 0.75 = $750
```

### 3.3. Protocol Fee Sharing

Treasury collects protocol fees and can use them for:
- BTR buybacks
- LP incentives
- Development funding

## 4. Staking (sBTR)

### 4.1. Liquid Staked Tokens

**Key differentiator**: sBTR tokens are **fully transferable** and liquid.

| Feature | BTR/sBTR | veCRV | veAERO |
|---------|----------|-------|--------|
| **Transferable** | ✅ Yes (ERC-20) | ❌ No | ⚠️ NFT (non-fungible) |
| **Fungible** | ✅ Yes | ❌ No | ❌ No |
| **Lock Period** | 14 days (default) | Up to 4 years | Up to 4 years |
| **Lock Decay** | None | Linear | Linear |

This enables:
- **Secondary markets**: sBTR can be traded on DEXs
- **Composability**: sBTR can be used in other DeFi protocols
- **Simpler UX**: No lock management required
- **No lock decay**: Constant voting power during lock period

### 4.2. Stake BTR

```solidity
function stakeGov(uint256 amount) external
```

**Effects**:
- Transfer BTR to pool
- Mint sBTR 1:1
- Set unlock time (default: 14 days)

### 4.3. Unstake BTR

```solidity
function unstakeGov(uint256 amount) external
```

**Requirements**:
- Lock period expired (`block.timestamp >= unlockTime`)
- Sufficient sBTR balance

### 4.4. Lock Period

- **Default**: 14 days (configurable by governance via `stakeLockDuration`)
- **Purpose**: Prevent vote-and-dump attacks
- **Extending**: New stakes extend lock to `block.timestamp + lockDuration`

## 5. Voting Power Formula

### 5.1. Linear Component

$$V_{linear} = (L * lpBase/10000) + (boosted * lpBoosted/10000) + (G * govMult/10000)$$

where:
- $L$ = total LP stake value ($)
- $G$ = governance stake value ($)
- $boosted$ = min(L, boostCap * G)

### 5.2. Quadratic Damping

> For complete formula including damping parameters, see [DAO Voting §2.2-2.3](/docs/2.2-DAO-Voting#22-quadratic-damping).

The linear voting power is then dampened to prevent concentration:

```
V_effective = V_linear × (1 - scale × V_linear^exponent)
```

Where `scale ≈ 0.00548` and `exponent ≈ 0.265` create a concave curve that reduces influence of extremely large holders.

### 5.3. Default Parameters

| Parameter | Value | Effect |
|-----------|-------|--------|
| `lpBaseMultiplier` | 5000 | 0.5x voting per LP dollar |
| `lpBoostedMultiplier` | 5000 | 0.5x bonus for boosted LP |
| `govMultiplier` | 10000 | 1x voting per BTR dollar |
| `boostCap` | 50000 | Max 5x LP boost per BTR |

## 6. Delegation

### 6.1. Delegate Votes

```solidity
function delegateVoting(address to) external
```

**Effects**:
- Voting power transferred to delegate
- Original holder keeps tokens
- Can self-delegate to reclaim (call with `to = msg.sender`)

### 6.2. Use Cases

- **Passive holders** -Delegate to active governance participants
- **Multi-sig optimization** -Aggregate votes to single voter
- **Protocol politicians** -Build voting coalitions

## 7. Comparison with Peers

| Feature | BTR | CRV | UNI | AERO |
|---------|-----|-----|-----|------|
| Max Supply | Fixed (100M) | Inflationary | Fixed | Inflationary |
| Staking | sBTR (liquid) | veCRV (locked) | None | veAERO (NFT) |
| Lock Period | 14 days default | Up to 4 years | N/A | Up to 4 years |
| Transferable Stake | ✅ Yes | ❌ No | N/A | ⚠️ NFT only |
| Boost Mechanism | sBTR matching | veCRV boost | None | veAERO boost |
| Fee Sharing | Via treasury | Direct | Via governance | Direct |

### 7.1. Key Differences

**vs veCRV/veAERO**:
- **Liquid staking**: sBTR is transferable; veCRV/veAERO are locked
- Shorter lock periods (14 days vs up to 4 years)
- No lock decay (constant power during lock)
- Simpler UX (no lock management)

**vs UNI**:
- Active staking incentives
- Reward boosting mechanism
- Protocol-level fee capture

## 8. Security Considerations

### 8.1. Supply Cap Enforcement

```solidity
// In Treasury
function mint(address to, uint256 amount) external onlyOwner {
    if (totalSupply + amount > MAX_SUPPLY) revert ExceedsMaxSupply();
    _mint(to, amount);
}
```

### 8.2. Minting Authorization

Only Treasury contract can mint BTR:
- Treasury address is immutable
- Minting requires governance approval
- Emergency pause available

### 8.3. Transfer Cooldown

sBTR has transfer cooldown after claiming rewards to prevent:
- Vote renting
- Reward farming
- Sybil attacks
