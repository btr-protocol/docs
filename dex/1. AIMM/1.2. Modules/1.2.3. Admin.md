# Admin Contract

> Pool configuration, timelock governance, and emergency controls

---

## 1. Overview

`Admin` (`dex/evm/src/Admin.sol`) is a standalone singleton governance contract serving every pool. Each public function takes `address pool` as its first arg. Owner authority routes through the shared singleton `AccessControl`. There is no `delegatecall` -Admin calls `Pool`'s restricted setters via standard external calls, and timelock state is keyed by `(pool, opId)` locally in Admin (not in `PoolStorage`).

Admin manages pool configuration with security-first design:
- **Timelocked operations** for sensitive changes
- **Emergency freeze** capability without delay
- **Two-phase execution** (request  wait  execute)
- **Grace period enforcement** to prevent indefinitely pending operations

---

## 2. Emergency Functions (No Timelock)

### 2.1. Freeze Asset

Immediately halts all operations on an asset:

```solidity
function freezeAsset(address token) external onlyOwner
```

**Effects:**
- Sets `FROZEN_BIT` in risk config flags
- Blocks swaps, deposits, withdrawals, flash loans
- Immediate effect (no timelock)

**Use cases:**
- Security incident response
- Oracle failure
- Suspected exploit

### 2.2. Unfreeze Asset

```solidity
function unfreezeAsset(address token) external onlyOwner
```

Clears `FROZEN_BIT` flag, restoring normal operations.

## 3. Timelock Delays

| Operation | Delay | Risk Level | ID |
|-----------|-------|------------|----|
| Add Asset | 1 day | LOW | `keccak256("ADD_ASSET", token)` |
| Update Risk Config | 1 day | LOW | `keccak256("UPDATE_RISK", token)` |
| Update Fee Params | 1 day | LOW | `TIMELOCK_ID_FEE_PARAMS` |
| Update Oracle | 2 days | BASE | `keccak256("ORACLE_UPDATE", token)` |
| Update Staking | 2 days | BASE | `TIMELOCK_ID_STAKING` |
| Update Bridge | 3 days | HIGH | `TIMELOCK_ID_BRIDGE` |
| Update Treasury | 3 days | HIGH | `TIMELOCK_ID_TREASURY` |
| Update Module | 3 days | HIGH | `TIMELOCK_ID_MODULE` |
| Transfer Ownership | 3 days | HIGH | `TIMELOCK_ID_OWNERSHIP` |
| Migrate Base Token | 7 days | CRITICAL | `TIMELOCK_ID_BASE_MIGRATION` |

**Grace Period:** 7 days after timelock expires to execute. Operations expire if not executed within grace period.

## 4. Two-Phase Execution Pattern

### 4.1. Step 1: Request

```solidity
function requestAddAsset(
    address token,
    OracleConfig calldata oracleCfg,
    RiskConfig calldata riskCfg,
    LiquidityProfile calldata profile,
    uint16 minFeeBps,
    uint8 decimals,
    uint64 initialPrice,
    uint32 initialFastVolEMA,
    uint32 initialSlowVolEMA
) external onlyOwner
```

**Actions:**
1. Compute operation ID
2. Pack timelock: `[executeAt:48][grace:48]`
3. Store pending data
4. Emit `TimelockRequested` event

### 4.2. Step 2: Wait

- Minimum delay must pass
- Within grace period

### 4.3. Step 3: Execute

```solidity
function executeAddAsset(address token) external onlyOwner
```

**Actions:**
1. Validate timelock (delay passed, within grace)
2. Decode and validate pending data
3. Apply changes
4. Clear pending state
5. Emit completion event

### 4.4. Step 4: Cancel (Optional)

```solidity
function cancelTimelock(uint8 opType) external onlyOwner
```

Owner can cancel any pending operation before execution.

## 5. Asset Management

### 5.1. Add Asset

Adds new token to pool with full configuration:

```solidity
function requestAddAsset(
    address token,
    OracleConfig calldata oracleCfg,
    RiskConfig calldata riskCfg,
    LiquidityProfile calldata profile,
    uint16 minFeeBps,
    uint8 decimals,
    uint64 initialPrice,
    uint32 initialFastVolEMA,
    uint32 initialSlowVolEMA
) external onlyOwner
```

**Default values set on execution:**
- `maxFeeBps`: 10000 (1%)
- `minDispersion`: 1000 (0.1%)
- `maxDispersion`: 100000 (10%)
- `gamma`: 10000 (1.0x)
- `vega`: 10000 (1.0x)
- `lambda`: 10000 (1.0x)
- `haircutSuppressor`: 10000 (1.0x)
- `anchor`: baseToken (default hub-and-spoke)
- `anchorDepth`: 1

**Validation:**
- Token not already configured
- Profile weights sum to 200
- Profile offsets monotonically increasing
- Oracle config valid
- Initial price non-zero
- Initial volatility values non-zero

### 5.2. Update Risk Config

```solidity
function requestUpdateRiskConfig(
    address token,
    RiskConfig calldata cfg
) external onlyOwner
```

Changes decay thresholds, feature flags, depth amplifier.

### 5.3. Update Fee Params

```solidity
function requestUpdateFeeParams(
    FeeParams calldata params
) external onlyOwner
```

Updates protocol share and flash loan fee.

## 6. Anchor Tree Management

### 6.1. Set Anchor (No Timelock)

```solidity
function setAnchor(
    address token,
    address anchor
) external onlyOwner
```

**Validation:**
- Asset must exist
- Anchor must exist or be address(0) for root
- No cycles created
- Depth within MAX_DEPTH (4)

**Effects:**
- Updates `asset.anchor`
- Updates `asset.anchorDepth`
- Route paths recomputed on next swap

## 7. Oracle Configuration

### 7.1. Update Oracle

```solidity
function requestOracleUpdate(
    address token,
    OracleConfig calldata cfg
) external onlyOwner
```

Changes price feed sources:

```solidity
struct OracleConfig {
    address primary;        // Primary feed address
    address secondary;      // Fallback feed address
    bytes32 feedId;         // Feed identifier
    uint16 modeFlags;       // USE_INTERNAL | USE_EXTERNAL | ALLOW_FALLBACK

```

**Validation:**
- Primary oracle callable
- Secondary oracle callable (if configured)

## 9. Governance Operations

### 9.1. Ownership Transfer

```solidity
function requestOwnershipTransfer(address newOwner) external onlyOwner
function executeOwnershipTransfer() external
```

**Note:** Execute callable by anyone (allows self-claim).

### 9.2. Pool Reference-Impl Swap

Pool upgrades live at the `PoolFactory`, not at `Admin`. Existing pool clones are immutable; new clones use the new reference impl.

```solidity
factory.requestReferenceUpgrade(address newImpl);
factory.executeReferenceUpgrade();
factory.cancelReferenceUpgrade();
```

See [3.2 Deployment & Upgrades §4.1](/docs/3.2-Deployment-&-Upgrades) for the full flow.

### 9.3. Base Token Migration

```solidity
function requestBaseMigration(address newBase) external onlyOwner
function executeBaseMigration() external onlyOwner
```

Changes root anchor of entire pool. **CRITICAL** operation - longest timelock.

### 9.4. Bridge Update

```solidity
function requestBridgeUpdate(address newBridge) external onlyOwner
function executeBridgeUpdate() external onlyOwner
```

Changes authorized cross-chain bridge contract.

### 9.5. Treasury Update

```solidity
function requestTreasuryUpdate(address newTreasury) external onlyOwner
function executeTreasuryUpdate() external onlyOwner
```

Changes protocol fee collection address.

## 10. Timelock Storage

### 10.1. Packed Format

```solidity
// Single uint96 per operation
mapping(bytes32 => uint96) pendingOps;   // [executeAt:48][grace:48]
mapping(bytes32 => bytes) pendingData;    // Operation parameters
```

### 10.2. Validation

```solidity
function validate(uint96 packed) internal view {
    (uint48 executeAt, uint48 grace) = unpack(packed);

    if (packed == 0) revert InvalidState();
    if (block.timestamp < executeAt) revert TooEarly();
    if (block.timestamp > executeAt + grace) revert Expired();
}
```

## 12. Security Considerations

### 12.1. Grace Period Protection

Grace period prevents:
- Indefinitely pending operations
- "Governance bomb" attacks
- Forgotten timelocks executing years later

### 12.2. Multi-sig Recommendation

Owner should be a multi-signature wallet:
- 3-of-5 or 4-of-7 recommended
- Geographically distributed signers
- Hardware wallet usage

### 12.3. Emergency Response

For security incidents:
1. **Immediate:** `freezeAsset()` on affected tokens
2. **Assess:** Evaluate damage and root cause
3. **Plan:** Prepare fix with timelock
4. **Execute:** Apply fix after delay
5. **Monitor:** Unfreeze and watch closely
