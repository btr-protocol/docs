# Rescue Module

> Emergency asset recovery and vulnerability response procedures

---

## 1. Overview

**Rescue Module** (`RescueV1`) provides **emergency asset recovery** when critical vulnerabilities or exploits compromise a pool. Rather than losing assets, the pool owner can:

- **Freeze assets instantly** to prevent further trading
- **Recover stuck assets** (ERC-20, ERC-721, ERC-1155) via timelock
- **Pause operations** across all pools if critical infrastructure fails
- **Update core parameters** (oracles, modules) to patch vulnerabilities

**Key principle**: Speed vs. safety tradeoff:
- **Immediate freezes** (seconds): Stop the bleeding
- **Timelocked recovery** (2-7 days): Allow community exit before drastic action

---

## 2. Emergency Asset Freezing

For freeze mechanism details, see [Admin Module §2](/docs/1.2.3-admin#2-emergency-functions-no-timelock).

**In rescue context**, freezing is the first response to suspected exploits:
1. `freezeAsset(token)` — Immediate, blocks all operations
2. Investigate root cause (1-2 hours)
3. If false alarm: `unfreezeAsset(token)`
4. If real issue: proceed to `requestRescue()`

## 3. Asset Recovery (Timelock Protected)

### 3.1. Recovery Flow

**Two-phase process** (prevents recovery mistakes):

**Phase 1: Request** (Owner-only)
```solidity
pool.requestRescue(
    TokenType rescueType,           // ERC20, NATIVE, ERC721, ERC1155
    address token,                  // Token to recover
    uint256[] calldata tokenIds,    // For NFTs: which IDs to recover
    address receiver                // Where to send recovered assets
);
```

- Snapshots current balance of asset
- Stores request with all parameters
- Timelock starts: **4 days delay**
- Emits `RescueRequested` event

**Phase 2: Execute** (Anyone can execute after timelock)
```solidity
// Wait 4 days (RESCUE_TIMELOCK)
pool.executeRescue(
    address receiver,
    TokenType rescueType
);
```

- Validates timelock has passed
- Executes recovery per token type
- Cleans up pending data
- Emits `RescueExecuted` event

### 3.2. Token Type Support

| Type | Method | Example | Limits |
|------|--------|---------|--------|
| **ERC20** | `rescueToken()` | Recover stuck USDC | Safe transfer, amount capped |
| **Native ETH** | `rescueAll()` | Recover stuck ETH | Balance snapshot |
| **ERC721** | `rescueERC721Batch()` | Recover NFTs | Best-effort, skips unowned |
| **ERC1155** | `rescueERC1155Batch()` | Recover multi-tokens | Best-effort, only non-zero |

### 3.3. Safety Features

**Balance Snapshot Protection**:
- At request time: snapshots current balance
- At execute time: recovers only up to snapshot (prevents over-recovery)

**Best-Effort Batch Processing** (for NFTs):
- Iterates through requested token IDs
- Skips unowned tokens (doesn't revert)
- Transfers only pool-owned tokens

### 3.4. Complete Recovery Example

**Scenario**: Pool receives airdrop of random NFTs, one has vulnerability

```solidity
// 1. Identify problematic NFT (ID 12345 has reentrancy bug)
pool.freezeAsset(problematicNFT);  // Immediate freeze

// 2. Request recovery of all NFTs (except 12345)
uint256[] memory tokenIds = [1, 2, 3, ... 12344, 12346, ...];
pool.requestRescue(
    TokenType.ERC721,
    problematicNFT,
    tokenIds,
    recoveryAddress  // Multi-sig treasury
);

// 3. Wait 4 days + announce to community

// 4. Execute recovery
pool.executeRescue(recoveryAddress, TokenType.ERC721);

// 5. All safe NFTs transferred to treasury
// 6. Problematic NFT remains frozen until further action
```

## 4. Vulnerability Response Procedures

### 4.1. Quick Reference: What to Do

| Issue | Time | Action | Result |
|-------|------|--------|--------|
| **Oracle price exploit** | Seconds | `freezeAsset()` | Block trading, no loss |
| **Stuck tokens** | 4+ days | `requestRescue()` → execute | Recover assets |
| **Oracle misconfiguration** | 2+ days | `requestOracleUpdate()` → execute | Fix pricing |
| **Module bug** | 3+ days | `requestModuleUpdate()` → execute | Patch code |
| **Reentrancy in hook** | Seconds | `setHook(token, address(0))` | Remove hook |

### 4.2. Scenario: Pool Hacked via Flash Loan Reentrancy

**T=0: Attack Detected**
```solidity
// Attacker exploited flash loan hook callback
// Drained $2M from pool reserves

// Immediate action:
pool.freezeAsset(ETH);    // Stop further ETH trading
pool.freezeAsset(USDC);   // Stop stablecoin pair

// Emit emergency alert to community
```

**T=1 hour: Root Cause Analysis**
```
Vulnerability: Hook allows reentrancy into swap
Impact: Attacker called swap() from within hook callback
Solution: Upgrade hook implementation (or remove hook)
```

**T=2 hours: Patch Deployment**
```solidity
// Deploy patched hook with reentrancy guard:
PatchedHookV1 patchedHook = new PatchedHookV1();

// Register via module update timelock (3 days)
pool.requestModuleUpdate(
    address(patchedHook),
    [IHook.preSwap.selector, ...],  // Replace unsafe selectors
    initData
);
```

**T=3 days: Execute Patch + Recovery**
```solidity
// Execute patched module
pool.executeModuleUpdate();

// Now recover stuck assets
pool.requestRescue(TokenType.ERC20, USDC, recipient);
```

**T=7 days: Assets Recovered**
```solidity
// Execute recovery
pool.executeRescue(recipient, TokenType.ERC20);

// USDC transferred to multi-sig treasury
// Investigation can continue offline
// Pool reopens with patched modules
```

## 5. Timelocks in Emergency Scenarios

> For general timelock governance, see [Admin Module §3-4](/docs/1.2.3-admin#3-timelock-delays).

### 5.1. Rescue-Specific Timelocks

| Action | Timelock | Grace Period | Notes |
|--------|----------|--------------|-------|
| **freezeAsset()** | None | N/A | Immediate emergency response |
| **requestRescue()** | 4 days | 3 days | Allows community exit before recovery |

### 5.2. Why 4-Day Rescue Delay?

- Gives users time to withdraw unaffected assets
- Prevents hasty recovery mistakes
- Allows community review of recovery plan
- Balance: fast enough for emergencies, slow enough for transparency

## 6. Community Coordination

### 6.1. Emergency Communication

When freeze/rescue is activated:

1. **Emit on-chain event**:
   ```solidity
   event RescueRequested(
       TokenType indexed rescueType,
       address indexed token,
       address indexed receiver,
       uint256 snapshotBalance,
       uint256 executeAt
   );
   ```

2. **Announce to community**:
   - Twitter/Discord: Explain vulnerability
   - Technical summary: Root cause analysis
   - Recovery plan: Timeline and expected outcome

3. **Allow user exit**:
   - During 4-day rescue delay, users can still unstake LP
   - Frozen assets can be withdrawn via rescue
   - Governance voting continues

### 6.2. Transparency

**Available Info**:
- Current frozen assets (on-chain state)
- Pending rescues (query pending data)
- Historical freezes (event logs)
- Recovery execution (transaction receipts)

**No hidden operations** — all actions logged on-chain.

## 8. Security Guarantees

### 8.1. What Rescue Does NOT Do

❌ **Cannot steal user funds** — Only recovers pool-owned assets (stuck tokens)

❌ **Cannot modify LP balances** — Only moves frozen assets

❌ **Cannot be triggered by attacker** — Owner-only + 4-day timelock

❌ **Cannot be hidden** — All on-chain, all events logged

### 8.2. What Rescue DOES Do

✅ **Recovers stuck tokens** — Assets accidentally sent to pool

✅ **Recovers airdropped NFTs** — Unsolicited ERC-721/1155

✅ **Prevents exploitation** — Fast freeze + slow recovery

✅ **Maintains user trust** — Transparent multi-sig authority

## 11. Getting Help

If vulnerability is discovered:

1. **Do NOT publicly disclose** (gives attacker time to exploit)
2. **Contact team privately**: security@example.com (or equivalent)
3. **Provide**:
   - Vulnerability description
   - Attack vector
   - Potential impact
   - POC code (if safe)
4. **Expect**:
   - Immediate pool freeze (if active exploitation)
   - 24-48 hour patch development
   - Transparent communication timeline
   - Potential bug bounty
