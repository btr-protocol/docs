# Flash Loans

## 1. Overview

The protocol implements ERC-3156 compliant flash loans, allowing users to borrow any supported asset for the duration of a single transaction without collateral.

## 2. Architecture

### 2.1. Implementation

Flash loan functionality is implemented in the standalone singleton `Flash` contract (`dex/evm/src/Flash.sol`). It serves every pool — each public function takes `address pool` as its first arg. Flash calls `Pool`'s restricted entry points (`flashSend` for the token push, `flashAccount` for the protocol-fee ledger update) via standard external calls — no `delegatecall`, no module trust.

### 2.2. Key Features

- **ERC-3156 Compliance**: Standard interface for maximum compatibility
- **Multi-Asset Support**: Flash loans available on any enabled asset
- **Configurable Fees**: Pool-wide flash loan fee in basis points
- **Protocol Revenue**: Fees split between LPs and protocol treasury
- **Hook Integration**: Pre/post flash loan hooks for custom logic
- **Liquidity Constraints**: Respects minimum liquidity requirements

## 3. Technical Details

### 3.1. Flash Loan Flow

1. **Validation**
   - Check asset is flash-enabled (`FLASH_ENABLED_BIT`)
   - Check asset is not frozen (`FROZEN_BIT`)
   - Verify sufficient reserves after minimum liquidity

2. **Pre-Flash Hook** (Optional)
   ```solidity
   preFlashLoan(pool, borrower, token, amount, fee, data)
   ```

3. **Loan Execution**
   - Transfer `amount` to borrower
   - Call borrower's `postFlashLoan()` callback
   - Verify ERC-3156 magic value returned

4. **Repayment Verification**
   - Check balance increased by `amount + fee`
   - Update reserves (excluding protocol fee)
   - Accumulate protocol fees

5. **Post-Flash Hook** (Optional)
   ```solidity
   postFlashLoan(pool, borrower, token, amount, fee, data)
   ```

### 3.2. Fee Structure

```solidity
fee = (amount * flashFeeBps) / 1_000_000
protocolFee = (fee * protoShare) / 100
lpFee = fee - protocolFee
```

- **flashFeeBps**: Pool-wide parameter (e.g., 500 = 0.05%)
- **protoShare**: Percentage to protocol (e.g., 20 = 20%)
- LP fee increases reserves, protocol fee collected separately

### 3.3. Security Features

- **Reentrancy Protection**: `nonReentrant` modifier on `flashLoan()`
- **Balance Verification**: Strict check on repayment amount
- **Callback Validation**: ERC-3156 magic value must be returned
- **Decay Application**: Liability decay applied before loan
- **Atomic Execution**: All operations within single transaction

## 4. Usage Examples

### 4.1. Basic Flash Loan

```solidity
contract Arbitrageur is IERC3156FlashBorrower {
    bytes32 constant CALLBACK_SUCCESS = keccak256("ERC3156FlashBorrower.postFlashLoan");

    function executeArbitrage(address pool, address token, uint256 amount) external {
        bytes memory data = abi.encode(msg.sender);
        IERC3156FlashLender(pool).flashLoan(this, token, amount, data);
    }

    function postFlashLoan(
        address initiator,
        address token,
        uint256 amount,
        uint256 fee,
        bytes calldata data
    ) external returns (bytes32) {
        // Perform arbitrage logic
        // ...

        // Approve repayment
        IERC20(token).approve(msg.sender, amount + fee);

        return CALLBACK_SUCCESS;
    }
}
```

### 4.2. Query Available Liquidity

```solidity
// Check maximum flash loan available
uint256 maxLoan = pool.maxFlashLoan(USDC);

// Calculate fee for specific amount
uint256 fee = pool.flashFee(USDC, 1_000_000e6); // 1M USDC
```

## 5. Configuration

### 5.1. Enabling Flash Loans

Pool owner must enable flash loans per asset via risk configuration:

```solidity
// Request flash loan enable (1 day timelock)
pool.requestUpdateRiskConfig(token, riskConfig);

// After timelock expires
pool.executeUpdateRiskConfig(token);
```

**RiskConfig flags**:
```solidity
riskConfig.flags |= FLASH_ENABLED_BIT;  // Enable flash loans
```

### 5.2. Fee Parameters

Flash loan fees are pool-wide and subject to 1-day timelock:

```solidity
// Request fee update
pool.requestUpdateFeeParams(IPool.FeeParams({
    protoShare: 20,      // 20% to protocol
    flashFeeBps: 500     // 0.05% flash loan fee
}));

// Execute after timelock
pool.executeUpdateFeeParams();
```

## 6. Emergency Controls

### 6.1. Asset Freeze

Pool owner can instantly disable flash loans by freezing an asset (no timelock):

```solidity
pool.freezeAsset(token);  // Disables all operations including flash loans
```

### 6.2. Minimum Liquidity

Reserves below `minLiquidity` cannot be flash loaned, protecting against liquidity drain attacks.

## 7. Gas Optimization

- Single storage slot for reserves update
- Minimal balance checks (before/after)
- Efficient fee calculation with no divisions in hot path
- Hook calls only when configured

## 8. Events

```solidity
event FlashLoanExecuted(
    address indexed borrower,
    address indexed receiver,
    address indexed token,
    uint256 amount,
    uint256 fee
);
```

## 9. Error Handling

| Error | Trigger |
|-------|---------|
| `FlashDisabled(token)` | Flash loans not enabled for asset |
| `AssetFrozen(token)` | Asset is frozen |
| `ZeroAmount()` | Requested amount is 0 |
| `InsufficientReserves()` | Not enough liquidity after minLiquidity |
| `FlashLoanCallbackFailed()` | Callback didn't return magic value or insufficient repayment |

## 10. Integration Notes

### 10.1. For Borrowers

1. Implement `IERC3156FlashBorrower` interface
2. Return correct magic value from `postFlashLoan()`
3. Ensure sufficient approval for repayment
4. Account for fees in arbitrage/liquidation logic

### 10.2. For Pool Operators

1. Enable flash loans via risk config (requires timelock)
2. Set appropriate fee parameters
3. Monitor flash loan activity via events
4. Consider minimum liquidity requirements

## 11. Best Practices

- **Test thoroughly**: Flash loan failures can be expensive
- **Check maxFlashLoan()**: Before attempting large loans
- **Account for slippage**: In arbitrage calculations
- **Handle native assets**: Use WETH internally, NATIVE for user-facing
- **Consider gas costs**: Flash loan fee + gas must be profitable
