# Distributor Contract

> Campaign-based cumulative Merkle reward distribution (ERC-20 only)

---

## 1. Overview

`Distributor` (`dex/evm/src/Distributor.sol`) is a standalone singleton campaign-based cumulative Merkle distributor serving every pool. Each public function takes `address pool` as its first arg; state is keyed by `(pool, campaignId, ...)`. Reward-token escrow lives at this contract's address; managers MUST pre-fund it before claims (claims revert on shortfall). Owner authority routes through the shared singleton `AccessControl`.

> **Phase 42H update**: the Distributor previously supported a second campaign flavour (POINTS) backed by SoulboundTokens with on-chain `redeemPoints` conversion. That path has been **removed**. All distribution is now standard ERC-20 token campaigns. Off-chain points programs may still be operated externally and bridged to a `TOKEN` campaign for on-chain redemption when ready.

---

## 2. Campaign Type

### 2.1. TOKEN Campaign

**Purpose:** Direct ERC-20 token distribution
**Mechanism:** Pre-funded escrow at the Distributor address
**Requirement:** Manager MUST transfer `totalClaimable - alreadyClaimed` of the reward token to the Distributor before/at root update; claims revert if balance is short.

**Use cases:**
- LP incentives
- Governance token distribution
- Partner rewards
- Off-chain points programs settled on-chain

## 3. Campaign Lifecycle

### 3.1. Creation

```solidity
function createCampaign(
    address pool,
    address rewardToken,
    address manager
) external onlyOwner returns (uint256 campaignId)
```

`campaignId` is per-pool. The first id minted for a pool is `1` (never `0` -preserves the F-A2-R14-3 fix).

### 3.2. Root Updates

```solidity
function updateCampaignRoot(
    address pool,
    uint256 campaignId,
    bytes32 merkleRoot,
    uint32 updatedAt,
    uint256 totalClaimable
) external
```

**Authorization:** Campaign manager only.

**Validation:** Distributor balance MUST cover `totalClaimable - alreadyClaimed`.

### 3.3. Claims

```solidity
function claimCampaign(
    address pool,
    uint256 campaignId,
    uint256 index,
    address account,
    uint256 totalEarned,
    bytes32[] calldata merkleProof
) external nonReentrant
```

**Requirements:**
- `msg.sender == account`
- Campaign status == ACTIVE
- Valid Merkle proof (leaf is domain-separated -see §5.1)
- `totalEarned > claimed[account]`

**Cumulative Model:**
- User claims difference: `claimable = totalEarned - alreadyClaimed`
- Root represents total-ever-earned, not per-epoch

### 3.4. Finalization

```solidity
function finalizeCampaign(address pool, uint256 campaignId) external onlyOwner
```

Sets status to FINALIZED. No more claims allowed.

## 4. Campaign States

```solidity
enum CampaignStatus {
    ACTIVE,      // Claims enabled
    PAUSED,      // Claims paused
    FINALIZED    // Campaign ended (no further claims)
}
```

### 4.1. State Transitions

```
ACTIVE   pause()    > PAUSED
PAUSED   resume()   > ACTIVE
ACTIVE   finalize() > FINALIZED
```

## 5. Merkle Tree Structure

### 5.1. Leaf Format (Domain-Separated)

```solidity
bytes32 leaf = keccak256(
    abi.encode(
        keccak256("BTR_DISTRIBUTOR_v1"),
        pool,
        campaignId,
        index,
        account,
        totalEarned
    )
);
```

| Field | Type | Purpose |
|-------|------|---------|
| domain | bytes32 | Hard-coded tag -prevents cross-instance / cross-campaign proof replay |
| `pool` | address | Per-pool keying |
| `campaignId` | uint256 | Per-campaign keying |
| `index` | uint256 | Unique index for user in tree |
| `account` | address | User address |
| `totalEarned` | uint256 | Cumulative total (not delta) |

### 5.2. Cumulative vs Delta Model

**AIMM uses cumulative:**
- Tree contains total-ever-earned
- User claims difference from last claim
- Allows late joiners without missing epochs
- Simpler off-chain computation

**Example:**
```
Epoch 1: Alice earns 100
  totalEarned = 100
  Alice claims 100

Epoch 2: Alice earns 50 more
  totalEarned = 150 (cumulative)
  Alice claims 50 (150 - 100 already claimed)
```

## 9. Integration: Off-Chain Components

**Required Infrastructure:**
1. **Indexer** -Track eligible activity (swaps, LP, staking)
2. **Calculator** -Compute rewards per user
3. **Tree Builder** -Generate Merkle tree (use the domain-separated leaf layout above)
4. **Pusher** -Call `updateCampaignRoot()`

### Proof Generation

```typescript
const DOMAIN = keccak256('BTR_DISTRIBUTOR_v1');
const leaves = entries.map(({ idx, addr, total }) =>
  keccak256(encodeAbi(
    ['bytes32', 'address', 'uint256', 'uint256', 'address', 'uint256'],
    [DOMAIN, poolAddr, campaignId, idx, addr, total],
  )),
);
const tree = makeMerkleTree(leaves);
const proof = tree.getProof(myLeafIndex);

// On-chain claim
distributor.claimCampaign(poolAddr, campaignId, idx, myAddr, totalEarned, proof);
```

## 10. Security Considerations

### 10.1. Pre-Funding Requirement

**CRITICAL:** Token campaigns must be pre-funded before claims.

```solidity
// In updateCampaignRoot() / claim path:
//   require(balanceOf(rewardToken, distributor) >= totalClaimable - alreadyClaimed)
```

### 10.2. Domain Separation

Leaf format embeds a static domain tag plus `pool` + `campaignId` so proofs cannot be replayed across pools, campaigns, or future protocol variants.

### 10.3. Reentrancy Protection

All claim functions use the `nonReentrant` modifier.

### 10.4. Self-Claim Only

```solidity
if (msg.sender != account) revert Err.Unauthorized();
```

Prevents griefing where a third party claims on behalf of a user.

## 11. Gas Optimization

- Single storage slot per user per campaign (cumulative claimed total).
- Merkle proof verification: O(log n), ~15-20 elements for 1M users, ~30k gas per claim.

## 12. Example: Token Airdrop

```solidity
// 1. Create campaign
uint256 id = distributor.createCampaign(pool, BTR_TOKEN, manager);

// 2. Fund distributor
BTR_TOKEN.transfer(address(distributor), 1_000_000e18);

// 3. Update root (off-chain generates tree using domain-separated leaves)
distributor.updateCampaignRoot(pool, id, root, uint32(block.timestamp), 1_000_000e18);

// 4. Users claim
distributor.claimCampaign(pool, id, index, user, totalEarned, proof);
```
