# AIMM Spread & Fee Model

> Asymmetric spread with symmetrical fee distribution

---

## 1. Overview

AIMM implements a sophisticated fee model that separates **spread asymmetry** (pricing) from **fee distribution fairness** (LP earnings):

### 1.1. Spread Asymmetry (Pricing)

- **Volatility Band** $S_v$: Symmetric base spread around mid-price
- **Directional Surcharge** $U$: Asymmetric penalty for coverage-worsening trades
- **Bid-Ask Spread**: Different costs for coverage-improving vs coverage-worsening directions

> For coverage ratio definition, see [Inventory Management §2](/docs/1.1.1-inventory-management#2-coverage-ratio).

### 1.2. Fee Distribution Symmetry (LP Fairness)

- **50/50 Split**: Half of fee taken from input token, half from output token
- **Fair LP Earnings**: LPs earn fees in both tokens regardless of trade direction
- **No One-Sided Fee Problem**: Unlike traditional AMMs where fees concentrate in one token

## 2. Unit System

### 2.1. BPS Precision

All fees and spreads use **0.0001% precision** (not standard 0.01% bps):

$$B = 10^6$$

where:
- $B$ = BPS_PRECISION
- 1 unit = 0.0001%
- 10,000 units = 1%
- 100 units = 0.01% (1 standard bps)

[^0]: **Solidity constant**: `uint256 private constant BPS_PRECISION = 1_000_000;` in `LibPricing.sol`. This provides 100x finer precision than standard 10,000 basis (0.01%).

### 2.2. Fee Bounds

| Parameter | Type | Max Value | Max Fee |
|-----------|------|-----------|---------|
| `minFeeBps` | uint16 | 65,535 | ~6.5% |
| `maxFeeBps` | uint16 | 65,535 | ~6.5% |

### 2.3. Volatility Units

Volatility $\sigma$ uses 1e6 base:
- 1,000,000 = 1%
- 10,000,000 = 10%

## 3. Spread Calculation

### 3.1. Risk Signals

> Volatility and deviation are used here as **pricing risk signals**. For how these values are computed and updated, see [Internal Oracle §8-11](/docs/1.2.2-internal-oracle#8-volatility-tracking).

**Per-Hop Volatility** $\sigma$:

$$\sigma_i = (\sigma_{\text{fast},i} + \sigma_{\text{slow},i}) / 2$$

where:
- $\sigma_f$ = fast EMA volatility
- $\sigma_s$ = slow EMA volatility

**Per-Hop Deviation**:

$$\Delta_i = \max(|o_{\text{fast},i} - o_{\text{slow},i}|,\; |o_{\text{fast},i}|)$$

where:
- $o_{f,i}$ = fast TWAP offset at hop $i$
- $o_{s,i}$ = slow TWAP offset at hop $i$

**Path-Level Aggregation**:

$$\begin{aligned} \sigma_p &= \max_i(\sigma_i) \\ \Delta_p &= \max_i(\Delta_i) \end{aligned}$$

### 3.2. Symmetric Volatility Band

Baseline spread, direction-agnostic:

$$S_v = \overbrace{100}^{\text{base spread (0.01\%)}} + \overbrace{\sigma_p \cdot \nu}^{\text{volatility contribution}}$$

where:
- $S_v$ = volatility band (basis points)
- $\sigma_p$ = path volatility
- $\nu$ = vega sensitivity (endpoint-only)

[^1]: **Solidity implementation**: Computes `scaledVol = (volatility * vega) / (100 * MULT_BASE)` where `MULT_BASE = 10000`, then `spread = 100 + scaledVol` in 0.0001% units (`BPS_PRECISION = 1,000,000`). So `100` = 0.01% base spread. See `LibPricing._calculatePathSpreadCached()`.

### 3.3. Directional Deviation Surcharge

Asymmetric penalty for informed/toxic flow:

$$U = \overbrace{\Delta_p \cdot \lambda}^{\text{toxic flow penalty}}$$

where:
- $U$ = deviation surcharge
- $\Delta_p$ = path deviation
- $\lambda$ = lambda sensitivity (endpoint-only)

[^2]: **Solidity implementation**: Computes `surcharge = (deviation * lambda) / MULT_BASE` where `MULT_BASE = 10000`. Result is in 0.0001% units. See `LibPricing._calculatePathSpreadCached()`.

### 3.4. Coverage Classification

$$\begin{aligned} c &= R/L \\ I_0 &= |c_{i,0} - 1| + |c_{o,0} - 1| \\ I_1 &= |c_{i,1} - 1| + |c_{o,1} - 1| \end{aligned}$$

where:
- $c$ = coverage ratio
- $R, L$ = reserves, liabilities
- $I$ = sum of coverage deviations
- subscripts 0, 1 = before, after trade
- subscripts $i, o$ = input, output asset

Trade improves coverage when $I_1 < I_0$.

### 3.5. Final Spread

$$\begin{aligned} S &= \begin{cases} S_v & \text{if improves coverage} \\ S_v + U & \text{otherwise} \end{cases} \\ S_f &= \operatorname{clamp}(S, f_{\min}, f_{\max}) \end{aligned}$$

where:
- $S$ = spread before clamping
- $S_f$ = final spread (clamped)
- $f_{\min}, f_{\max}$ = fee bounds

## 4. 50/50 Fee Distribution

### 4.1. Motivation

**Problem with Traditional AMMs**:
- Uniswap V2/V3: Fee taken 100% from input -> LPs earn one-sided fees
- Curve V1: Fee taken 100% from output -> LPs earn one-sided fees
- **Issue**: Imbalanced LP earnings on uni-directional trends, rewarding one-side only

**AIMM Solution**:
- **50% from input token** -> LPs earn fees in token being sold
- **50% from output token** -> LPs earn fees in token being bought
- **Result**: Symmetrical fee distribution rewards two-sided utilization regardless of trade direction

### 4.2. Implementation

$$\begin{aligned} \phi_i &= \frac{x \cdot S}{2} \\ \phi_o &= \frac{y \cdot S}{2} \end{aligned}$$

where:
- $\phi_i$, $\phi_o$ = fees from input and output
- $x$ = input amount
- $y$ = output amount
- $S$ = spread (as fraction)

[^3]: **Solidity implementation**: `fee_in = (amountIn * spreadBps) / (2 * BPS_PRECISION)` and `fee_out = (amountOut * spreadBps) / (2 * BPS_PRECISION)` where `BPS_PRECISION = 1,000,000`. The spread is in 0.0001% units. See `LibPricing.splitFee()`.

$$\begin{aligned} x' &= x - \phi_i \\ y' &= y - \phi_o \end{aligned}$$

where:
- $x'$, $y'$ = net amounts after fee deduction

### 4.3. State Updates

**Input Asset**: Reserves increased by net amount (after fee), liabilities unchanged.

**Output Asset**: Reserves decreased by net amount (after fee), liabilities unchanged.

**Fee Retention**: Both `fee_in` and `fee_out` stay in reserves -> increases coverage.

### 4.4. Example: ETH -> USDC Swap

**Setup**:
- Trader swaps 1 ETH for USDC
- Mid-price: 1 ETH = 2000 USDC
- Spread: 5000 (0.5% in 0.0001% units)

**Traditional AMM (100% input fee)**:
$$\text{Fee} = 1\, \text{ETH} \times 0.50\% = 0.005\, \text{ETH}$$
$$\text{LPs earn: } 0.005\, \text{ETH only}$$

**AIMM (50/50 split)**:
$$\text{fee}_{\text{in}} = 1\, \text{ETH} \times 0.25\% = 0.0025\, \text{ETH}$$
$$\text{fee}_{\text{out}} = 2000\, \text{USDC} \times 0.25\% = 5\, \text{USDC}$$
$$\text{LPs earn: } 0.0025\, \text{ETH} + 5\, \text{USDC} \text{ (balanced!)}$$

## 5. Bid-Ask Asymmetry

### 5.1. Two-Sided Market

At any instant, effective bid/ask around mid-price $m$:

- **Bid** (coverage-improving): $m(1 - S_v/B)$
- **Ask** (coverage-worsening): $m\left(1 + \frac{S_v + U}{B}\right)$
- **Spread Width**: $2S_v + U$

### 5.2. Coverage-Dependent Asymmetry

When pool is **over-collateralized** ($c > 1$):
- Buying from pool (reduces reserves) -> improves coverage -> pays $S_v$ only
- Selling to pool (increases reserves) -> worsens coverage -> pays $S_v + U$

When pool is **under-collateralized** ($c < 1$):
- Buying from pool -> worsens coverage -> pays $S_v + U$
- Selling to pool -> improves coverage -> pays $S_v$ only

At **equilibrium** ($c \approx 1$):
- Both directions pay similar spreads unless deviation high

### 5.3. Economic Interpretation

**Volatility Band** $S_v$: Market maker spread covering:
- Expected adverse selection
- Inventory rebalancing costs
- Option value of providing liquidity

**Deviation Surcharge** $U$: Informed flow penalty:
- High $\Delta$ signals oracle disagreement
- Coverage-worsening trades during uncertainty pay extra
- Prevents exploitation during price manipulation

## 6. Protocol Fee Split

### 6.1. Revenue Sharing

After calculating total fees (50% from input + 50% from output):

$$\begin{aligned} \phi_p &= \phi \cdot r \\ \phi_l &= \phi - \phi_p \end{aligned}$$

where:
- $\phi_p$ = protocol fee portion
- $\phi_l$ = LP fee portion
- $\phi$ = total fees
- $r$ = protocol share fraction (0-1)

[^4]: **Solidity implementation**: `protocolFee = (totalFees * protoShare) / 100` where `protoShare` is a uint8 with base 100 (e.g., 25 = 25%). See `LibPricing.splitFee()`.

### 6.2. Proto Share Parameter

| Proto Share | Type | Distribution |
|-------------|------|--------------|
| **0** | uint8 | 100% to LPs |
| **25** | uint8 | 25% protocol, 75% LPs (default) |
| **50** | uint8 | 50/50 split |

`protoShare` is a uint8 with base 100 (0-100 range representing 0%-100%).

## 7. Fee Bounds

### 7.1. Per-Asset Configuration

Each asset defines fee-related parameters:
- `minFeeBps`, `maxFeeBps`: Floor/ceiling in 0.0001% units
- `vega`: Volatility sensitivity (basis 10000)
- `lambda`: Deviation sensitivity (basis 10000)

See: [Parametrization](/docs/1.1.7-parametrization)

### 7.2. Path-Level Bounds

For multi-hop swaps:

$$\begin{aligned} f_{\min,p} &= \max_i(f_{\min,i}) \\ f_{\max,p} &= \max_i(f_{\max,i}) \\ S &= \operatorname{clamp}(S_r, f_{\min,p}, f_{\max,p}) \end{aligned}$$

where:
- $f_{\min,p}$ = path-level minimum fee
- $f_{\max,p}$ = path-level maximum fee
- $S_r$ = unclamped spread
- $i$ = index over all hops in path

## 8. Comparison with Other AMMs

### 8.1. Fee Token Location

| Protocol | Fee Taken In | Distribution |
|----------|--------------|--------------|
| **Uniswap V2/V3** | 100% input | One-sided LP earnings |
| **Trader Joe V2** | 100% input | One-sided LP earnings |
| **Curve V1** | 100% output | One-sided LP earnings |
| **Wombat** | Input, redistributed | Single-asset LP friendly |

### 8.2. Spread Characteristics

| Aspect | Uniswap V3 | AIMM |
|--------|------------|------|
| **Fee structure** | Fixed tiers (0.01-1%) | Market-aware (bi-factor) |
| **Volatility response** | None | $\sigma$-scaled band |
| **Inventory response** | None | Skew + surcharge |
| **Coverage awareness** | No | Yes (directional) |
| **LVR mitigation** | Via hooks only | Built-in surcharge |
| **Fee distribution** | One-sided (input) | **Symmetrical (50/50)** |

## 9. Parameter Guidelines

### 9.1. Vega (Volatility Sensitivity)

- **Basis**: 10,000 = 1.0x sensitivity
- **Example**: $\nu = 10000$, $\sigma = 1,000,000$ (1%) $\to S_v \approx 200$ units (0.02%)
- **Tuning**: Higher vega -> wider spreads for volatile assets

### 9.2. Lambda (Deviation Sensitivity)

- **Basis**: 10,000 = 1.0x sensitivity
- **Example**: $\lambda = 10000$, $\Delta = 500,000$ $\to U \approx 50$ units (0.005%)
- **Tuning**: Higher lambda -> larger penalty during oracle disagreement

### 9.3. Fee Bounds

- **Units**: 0.0001% precision ($B = 1,000,000$)
- **Example**: $f_{\min} = 100$ (0.01%), $f_{\max} = 50000$ (5%)
- **Purpose**: Prevent extreme spreads (max) and free-option attacks (min)

## 10. Multi-Hop Fee Aggregation

### 10.1. Path Structure

For swap A -> B through intermediate hops:

`A -> X_1 -> X_2 -> ... -> B`

### 10.2. Signal Aggregation

| Signal | Aggregation | Rationale |
|--------|-------------|-----------|
| $\sigma_p$ | $\max(\sigma_i)$ across legs | Conservative uncertainty |
| $\Delta_p$ | $\max(\Delta_i)$ across legs | Conservative deviation |
| $\nu$ | $\max$ (endpoints only) | Risk at edges |
| $\lambda$ | $\max$ (endpoints only) | Risk at edges |
| $f_{\min}$ | $\max$ (all legs) | Strictest floor |
| $f_{\max}$ | $\max$ (all legs) | Strictest ceiling |

### 10.3. Edge vs Intermediate Pricing

- **Edge hops** (first/last): Full price impact via spline traversal
- **Intermediate hops**: Mid-price only (no impact accumulation)

## 12. Economic Benefits

### 12.1. For Liquidity Providers

**Symmetrical Fee Earnings**:
- Earn fees in both tokens proportionally
- No concentration risk from one-sided fees
- Better portfolio balance maintenance

**Coverage Protection**:
- Fees retained in reserves improve coverage
- Higher coverage -> better LP protection
- Lower withdrawal haircuts

### 12.2. For Traders

**Fair Pricing**:
- Spread reflects actual market conditions (volatility + deviation)
- Coverage-improving trades get better prices (arb-friendly)
- Coverage-worsening trades pay fair premium during uncertainty

**Predictable Costs**:
- Bounded fees prevent extreme spreads
- Transparent market-aware model
- No hidden LVR extraction

## 13. Flash Loan Fees

### 13.1. Base Fee

Fixed percentage of borrowed amount:

$$\phi_f = x \cdot f_f$$

where:
- $\phi_f$ = flash loan fee
- $x$ = borrowed amount
- $f_f$ = flash fee rate

[^5]: **Solidity implementation**: `flashFee = (amount * flashFeeBps) / BPS_PRECISION` where `BPS_PRECISION = 1,000,000` and `flashFeeBps` defaults to 90 (0.009%). See `Flash.flashLoan()`.

### 13.2. Parameters

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `flashFeeBps` | uint16 | 90 | 0.009% flash loan fee |
