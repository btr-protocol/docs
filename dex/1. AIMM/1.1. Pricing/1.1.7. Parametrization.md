### 2.1. Sensitivity Parameters

#### 2.1.1. Gamma ($\gamma$) - Inventory Sensitivity

- **Default**: 10000 (1.0x)
- **Purpose**: Multiplier controlling inventory skew strength
- **Effect**: Higher gamma → steeper linear slope → stronger price incentive for rebalancing

**Formula** (linear with bounded progress):

$$\begin{aligned} \pi &= \frac{|c - c_t|}{|c_b - c_t|} \\ \psi &= \operatorname{sign}(c_t - c) \cdot \gamma \cdot 100 \cdot \pi \end{aligned}$$

where:
- $\pi$ = progress toward boundary, in $[0, 1]$
- $c$ = current coverage ratio
- $c_t$ = target coverage = 1.0
- $c_b$ = critical boundary = 0.5 or 2.0
- $\gamma$ = multiplier, basis 10000
- $\psi$ = inventory skew, in $[-100, +100]$

**Key Properties**:
- At criticalMin ($c = 0.5$): $\psi = +\gamma \cdot 100$ (max premium, pool buying)
- At target ($c = 1$): $\psi = 0$ (equilibrium)
- At criticalMax ($c = 2$): $\psi = -\gamma \cdot 100$ (max discount, pool selling)
- Linear interpolation -predictable, matches Avellaneda-Stoikov theory

| Gamma | Multiplier | Effect | Use Case |
|-------|------------|--------|----------|
| 5000 | 0.5x | Half skew | Volatile assets, softer response |
| 10000 | 1.0x | Full skew | Default/balanced |
| 15000 | 1.5x | 150% skew | More aggressive rebalancing |
| 20000 | 2.0x | Double skew | Stablecoins, strong incentives |

#### 2.1.2. Vega ($\nu$) - Volatility Sensitivity

- **Default**: 10000 (1.0x)
- **Purpose**: Controls how volatility affects both liquidity dispersion and base spread
- **Effect**: Higher vega -> wider spreads during volatile markets

**Dispersion Formula**:

$$\kappa = \operatorname{clamp}(\kappa_0 + \sigma \cdot \nu,\; \kappa_{\min},\; \kappa_{\max})$$

**Spread Formula**:

$$S_v = 100 + \sigma_p \cdot \nu$$

[^1]: **Solidity implementation**: Dispersion: `scaledVol = (volatility * vega) / (1000 * MULT_BASE)` where `MULT_BASE = 10000`. Spread: `scaledVol = (volatility * vega) / (100 * MULT_BASE)`. Both use `BPS_PRECISION = 1,000,000`. See `LibPricing._calculateDispersion()` and `_calculatePathSpreadCached()`.

**Tuning**:
- Increase vega (e.g., 20000 = 2.0x) to widen spreads more aggressively with volatility
- Decrease vega (e.g., 5000 = 0.5x) to dampen volatility impact

#### 2.1.3. Lambda ($\lambda$) - Momentum Sensitivity

- **Default**: 10000 (1.0x)
- **Purpose**: Controls how price deviation (fast vs slow TWAP divergence) affects spread surcharge
- **Effect**: Higher lambda -> larger penalty when fast/slow EMAs diverge (toxic flow detection)

**Formula**:

$$U = \overbrace{\Delta_p \cdot \lambda}^{\text{toxic flow penalty}}$$

[^2]: **Solidity implementation**: `surcharge = (deviation * lambda) / MULT_BASE` where `MULT_BASE = 10000`. Result is in 0.0001% units. See `LibPricing._calculatePathSpreadCached()`.

**Tuning**:
- Increase lambda (e.g., 15000 = 1.5x) to penalize trending/uncertain price action heavily
- Decrease lambda (e.g., 5000 = 0.5x) for smoother response to price uncertainty

#### 2.1.4. Haircut Suppressor ($\eta$) - Withdrawal Penalty Modifier

- **Default**: 10000 -> exponent $p = 2$
- **Purpose**: Controls withdrawal haircut curve when $c < 1$
- **Effect**: Higher suppressor -> gentler haircut curve

**Power-Law Formula**:

$$h(c) = \underbrace{(1 - c)}_{\text{coverage deficit}}^{p}$$

where:
- $p$ = 1 + $\eta$

[^3]: **Solidity implementation**: `exponent = 1 + eta/MULT_BASE` where `MULT_BASE = 10000`. The parameter `haircutSuppressor` (eta) uses basis 10000, so `eta = 10000` yields `p = 2`. See `LibPricing.applyWithdrawalHaircut()`.

| $\eta$ | Exponent $p$ | Curve Shape |
|-------|--------------|-------------|
| 0 | 1 | Linear (harsh) |
| 10000 | 2 | Quadratic (moderate) |
| 30000 | 4 | Gentle |
| 40000 | 5 | Very gentle |

**Example**: At 80% coverage with $\eta = 10000$ ($p = 2$):

$$h = (1 - 0.8)^2 = 0.04 = 4\%$$

### 2.2. Dispersion Bounds

**Dispersion** defines the price range over which liquidity is distributed (in 0.0001% units).

- **`minDispersion`**: Lower bound (e.g., 1000 = 0.1% range)
- **`maxDispersion`**: Upper bound (e.g., 10000 = 1.0% range)

**Effect**: Tighter bounds create denser liquidity (better execution), wider bounds spread liquidity (more slippage).

### 2.3. Liquidity Floor (minLiquidity)

**Type**: uint128 (raw token units)
**Default**: 0 (no floor)

Minimum reserves that must remain in the pool. Swaps and flash loans that would reduce reserves below this threshold revert.

**Use Cases**:
- Prevent complete liquidity drain attacks
- Ensure minimum depth for price discovery
- Protect against dust/griefing attacks
- Maintain market functionality during extreme events

**Example**: For WETH (18 decimals), set `minLiquidity = 1e18` to ensure at least 1 WETH remains.

**Implementation**:
```solidity
if (assetOut.reserves < assetOut.minLiquidity) {
    revert IErrors.InsufficientAmount(assetOut.reserves, assetOut.minLiquidity);
}
```

### 2.4. Price Floor (reservationPrice)

**Type**: uint64 (B64 format)
**Default**: 0 (disabled)

Minimum price vs anchor asset. Swaps that would push the price below this floor revert.

**Use Cases**:
- Prevent flash crash attacks
- Protect against oracle manipulation
- Circuit breaker for extreme market conditions
- Protect LST peg (e.g., wstETH vs WETH floor at 0.95)

**Configuration**:
```solidity
// Example: Set wstETH reservation at 95% of WETH
// B64 encode price 0.95 with 18 decimals
uint64 reservationPrice = LibMaths.encodeB64(0.95e18, 18);
```

**Implementation**:
```solidity
if (feed.lastPriceB64 < reservationPrice) {
    revert IErrors.PriceBelowReservation(feed.lastPriceB64, reservationPrice);
}
```

### 2.5. Fee Bounds

Fees use **0.0001% precision** (BPS_PRECISION = 1,000,000):

| Parameter | Type | Max Value | Max Fee |
|-----------|------|-----------|---------|
| `minFeeBps` | uint16 | 65,535 | ~6.5% |
| `maxFeeBps` | uint16 | 65,535 | ~6.5% |

**Examples**:
- `100` = 0.01% (1 standard bps)
- `1000` = 0.1%
- `10000` = 1%
- `50000` = 5%

## 3. Risk Configuration

**Contract**: `IPool.RiskConfig` (struct)
**Admin Function**: `Admin.updateRiskConfig()`

| Field | Type | Units | Description | Typical Values |
|-------|------|-------|-------------|----------------|
| `decayStartRatioBps` | uint16 | 0.0001% | Coverage threshold for liability decay | 980000 (98%) |
| `coverageMin` | uint16 | 0.01% | Critical coverage floor for max premium (inventory skew boundary) | 5000 (50%) |
| `coverageMax` | uint16 | 0.01% | Critical coverage ceiling for max discount (inventory skew boundary) | 20000 (200%) |
| `decaySlope` | uint32 | WAD/second | Linear decay rate for liabilities | ~3.17e10 (1%/year) |
| `depthAmplifier` | uint16 | Raw | Depth curve amplifier (scaled by 1e6 in formula) | **200000-300000 (recommended default)** |
| `flags` | uint16 | Bitfield | Feature flags | See flags table |

### 3.1. Risk Flags (Bitfield)

> **Canonical Reference**: All flags are defined in `contracts/src/libraries/LibConstants.sol`

| Bit | Hex | Flag | Description |
|-----|-----|------|-------------|
| 0 | `0x0001` | `FROZEN_BIT` | Freeze all operations (emergency, owner-only) |
| 1 | `0x0002` | `SWAP_ENABLED_BIT` | Enable/disable swaps for this asset |
| 2 | `0x0004` | `LIABILITY_SWAP_ENABLED_BIT` | Enable/disable liability swaps |
| 3 | `0x0008` | `DECAY_ENABLED_BIT` | Enable/disable liability time decay |
| 4 | `0x0010` | `FLASH_ENABLED_BIT` | Enable/disable flash loans |
| 5 | `0x0020` | `FEE_ON_TRANSFER_BIT` | Asset has transfer fees (handle differently) |
| 6 | `0x0040` | `STAKEABLE_BIT` | LP token can be staked for emissions |
| 7 | `0x0080` | `BRIDGEABLE_BIT` | Asset can be bridged cross-chain |

**Common flag combinations**:
- `0x0002` = Swap only
- `0x0016` = Swap + Flash + Liability Swap (standard)
- `0x0056` = Full features (includes staking)

### 3.2. Coverage Thresholds

| Threshold | Default | Purpose |
|-----------|---------|---------|
| `coverageMin` | 50% (5000) | Maximum premium boundary (parametric, 0.01% units) |
| `target` | 100% (hardcoded) | Equilibrium point (zero skew) |
| `coverageMax` | 200% (20000) | Maximum discount boundary (parametric, 0.01% units) |

### 3.3. Depth Amplifier

**Formula**:

$$D = R + \overbrace{k \cdot (L - R) \cdot \pi^e}^{\text{virtual depth}}$$

 where:
- $k$ = depth amplifier
 - $\pi = (c - c_f)/(1 - c_f)$ = progress from floor to target
 - $e = 1/(1 + 2k)$ = exponent

[^4]: **Solidity implementation**: The depth amplifier parameter is stored as a raw uint16 and divided by `1,000,000` in the formula: `virtualDepth = (depthAmp * deficit * concaveProgress) / (1_000_000 * WAD)`. The exponent uses the same normalization: `exponent = WAD * 1_000_000 / (1_000_000 + 2 * depthAmp)`. See `LibPricing.calculateEffectiveDepth()`.

**Key Properties**:
- At critical floor ($c = c_f$): $D = R$ (no virtual depth)
- At target ($c \ge 1$): $D = R$ (full reserves)
- Between: concave interpolation adds virtual depth

| Value | Effect |
|-------|--------|
| 0 | No amplification ($D = R$ always) - pure reserve-backed, most conservative |
| **200000-300000** | **Recommended default: 20-30% virtual depth amplification at peg.** Production deployments should start in this range. |
| 500000 | 50% max virtual addition - upper bound for aggressive curated blue-chip pools after coverage is proven |
| 1000000 | 100% max virtual addition - research/experimental ceiling, not recommended for production |

> **Recommended default: 200000-300000** (20-30% virtual depth amplification at peg). Production deployments should start in this range; `0` disables amplification entirely (pure reserve-backed); `500000` is the upper bound for aggressive curated blue-chip pools.

### 3.4. Liability Decay

When coverage falls below $c_d$ (decay start ratio):

$$\Delta L = L \cdot \rho \cdot \Delta t$$

where:
- $\Delta L$ = liability decay amount
- $L$ = current liabilities
- $\rho$ = decay slope
- $\Delta t$ = time elapsed

[^5]: **Solidity implementation**: Returns `(liabilities * decaySlope * dt) / WAD` where `WAD = 1e18`. The decay slope `decaySlope` is stored in WAD/second units. See `LibPricing.calculateDecay()`.

**Constraints**:
- Never decays below reserves
- Stops when $c \ge 1$

## 4. Oracle Configuration

**Contract**: `IPool.OracleConfig` (struct)
**Admin Function**: `Admin.requestOracleUpdate()` (timelocked)

| Field | Type | Description |
|-------|------|-------------|
| `primary` | address | Primary oracle address |
| `secondary` | address | Secondary oracle (fallback) |
| `feedId` | bytes32 | Pyth/Redstone feed ID |
| `modeFlags` | uint16 | Oracle mode flags |
| `accDecimals` | uint8 | Accumulator decimal precision |

**Oracle Modes**:
- `USE_INTERNAL`: Pool's internal TWAP oracle
- `USE_EXTERNAL`: External oracle (Chainlink, Pyth, etc.)
- `ALLOW_FALLBACK`: Try secondary if primary fails

## 5. Fee Parameters

**Contract**: `IPool.FeeParams` (struct)

| Field | Type | Units | Description | Typical Values |
|-------|------|-------|-------------|----------------|
| `protoShare` | uint8 | Percentage (base 100) | Protocol's share of total fees | 25 (25%) |
| `flashFeeBps` | uint16 | 0.0001% | Flash loan fee rate | 90 (0.009%) |

### 5.1. Fee Calculation

**Spread Model**:

$$\begin{aligned} S_v &= 100 + \sigma_p \cdot \nu \\ U &= \Delta_p \cdot \lambda \\ S &= \begin{cases} S_v & \text{if improves coverage} \\ S_v + U & \text{otherwise} \end{cases} \\ S_f &= \operatorname{clamp}(S,\; f_{\min},\; f_{\max}) \end{aligned}$$

**50/50 Split**:

$$\begin{aligned} \phi_i &= \frac{x \cdot S}{2} \\ \phi_o &= \frac{y \cdot S}{2} \end{aligned}$$

**Protocol Split**:

$$\phi_p = \phi \cdot r$$

where:
- $r$ = protocol share fraction (0-1)

[^6]: **Solidity implementation**: See Spread & Fees §2-4 for scaling details. Uses `BPS_PRECISION = 1,000,000`, `MULT_BASE = 10000`.

## 6. Liquidity Profile

**Contract**: `IPool.LiquidityProfile` (struct)

| Field | Type | Description | Constraints |
|-------|------|-------------|-------------|
| `weights` | uint8[16] | Segment weights (each unit = 0.5% of depth) | Must sum to 200 |
| `knots` | int8[17] | Price offset knots as % of dispersion range around TWAP. Dispersion defines liquidity breadth (e.g., dispersion=1% -> +-1% around TWAP), knots are % of this (knot=-50 -> -0.5% from TWAP) | Monotone increasing, max - min == 100 |

### 6.1. Profile Structure

- **X-axis**: Cumulative depth `[0, 10000]` = 0-100%
- **Y-axis**: Price offset from TWAP (scaled by dispersion)
- **Interpolation**: Fritsch-Carlson monotone cubic Hermite spline with exact analytical integration (Fritsch & Carlson 1980, *SIAM J. Numer. Anal.* 17(2))

### 6.2. Validation Rules

1. First weight must be non-zero
2. Weights must sum to exactly 200
3. Offsets must be monotonically non-decreasing
4. First zero weight marks profile end

### 6.3. Example Profiles

**Concentrated (Stablecoins)**:
```
weights:  [80, 60, 30, 20, 10, 0, ...]
offsets:  [0,  5,  20, 50, 100, ...]
```

**Uniform (Volatile Assets)**:
```
weights:  [25, 25, 25, 25, 25, 25, 25, 25, 0, ...]
offsets:  [0,  25, 50, 75, 100, 125, 150, 200, ...]
```

## 7. Admin Functions

#### 7.1.1. setAssetParams()

Updates asset-level pricing and risk parameters including `minLiquidity` and `reservationPrice`:

```solidity
function setAssetParams(
    address token,
    uint128 minLiquidity,     // Reserves floor (raw token units)
    uint16 minFeeBps,         // Min fee (0.0001% units)
    uint16 maxFeeBps,         // Max fee (0.0001% units)

    uint16 vega,              // Volatility sensitivity
    uint16 lambda,            // Deviation sensitivity
    uint16 haircutSuppressor, // Haircut curve modifier

) external onlyOwner;
```

| **Coverage** (decayStartRatioBps) | 1,000,000 | 0.0001% | 980000 = 98% |
| **Coverage Bounds** (coverageMin, coverageMax) | 10,000 | 0.01% | 5000 = 50%, 10000 = 100%, 20000 = 200% |
| **Dispersion** (minDispersion, maxDispersion) | 1,000,000 | 0.0001% | 1000 = 0.1%, 10000 = 1.0% |
| **Protocol Share** (protoShare) | 100 | 1% | 25 = 25% |
| **Multipliers** (gamma, vega, lambda, suppressor) | 10,000 | 0.01% | 10000 = 1.0x, 5000 = 0.5x |
| **Prices** (WAD) | 1e18 | 18 decimals | 1e18 = 1.0 |

| **Decay Slope** (decaySlope) | WAD/second | 18 decimals | ~3.17e10 = 1%/year |

### 8.1. Why 0.0001% Precision (BPS_PRECISION = 1,000,000)?

AIMM uses a non-standard precision basis of **1,000,000** (0.0001%) rather than the traditional **10,000** (0.01%) used by most DeFi protocols. This design choice enables:

**1. Stablecoin Fee Granularity**

Stablecoin pools require extremely tight fees (0.001%-0.01%) to remain competitive. Standard 0.01% precision cannot express fees below 1 bps:

| Fee | Standard (10k basis) | AIMM (1M basis) |
|-----|---------------------|-----------------|
| 0.001% | Cannot express | 10 |
| 0.005% | Cannot express | 50 |
| 0.01% | 1 | 100 |

**2. Volatility Measurement Precision**

Volatility EMAs track price movements in 0.0001% units, allowing detection of sub-basis-point changes essential for accurate spread calculation during low-volatility periods.

**3. Dispersion Curve Resolution**

Liquidity dispersion curves require fine-grained control:
- Stablecoins: 0.01%-0.1% dispersion (10-1000 in our basis)
- Volatile assets: 0.1%-1.0% dispersion (1000-10000)

Standard basis would limit stablecoin dispersion to minimum 0.01%, too coarse for tight peg maintenance.

**4. Coverage Ratio Thresholds**

Coverage thresholds like 98% (980000) vs 97% (970000) require precision for fine-tuned decay activation.

**Conversion Table**:

| Standard BPS (0.01%) | AIMM Units (0.0001%) | Percentage |
|----------------------|----------------------|------------|
| 1 bps | 100 | 0.01% |
| 5 bps | 500 | 0.05% |
| 10 bps | 1000 | 0.10% |
| 100 bps | 10000 | 1.00% |

## 9. Parameter Tuning Guide

### 9.1. Conservative Pool (Stablecoins)

```solidity
Asset {
    minFeeBps: 10,        // 0.001%
    maxFeeBps: 500,       // 0.05%
    minDispersion: 100,   // 0.01%
    maxDispersion: 1000,  // 0.1%
    gamma: 10000,         // 1.0x multiplier
    vega: 5000,           // 0.5x volatility sensitivity
    lambda: 10000,        // 1.0x deviation sensitivity
    haircutSuppressor: 10000  // p=2 haircut curve
}

RiskConfig {
    coverageMin: 9000,          // 90% floor (0.01% units)
    coverageMax: 11000,          // 110% ceiling (tight bounds for stablecoins, 0.01% units)
    decayStartRatioBps: 980000,  // 98% decay trigger
    depthAmplifier: 0,           // BTR Pool 1: depthAmp=0 — conservative bootstrap config (pure reserve-backed, no virtual depth). Move to 200000-300000 once coverage is proven.
    flags: 0x07                  // Swap + Flash + Liability Swap
}
```

### 9.2. Aggressive Pool (Volatile Assets)

```solidity
Asset {
    minFeeBps: 500,       // 0.05%
    maxFeeBps: 50000,     // 5%
    minDispersion: 5000,  // 0.5%
    maxDispersion: 50000, // 5%
    gamma: 15000,         // 1.5x multiplier (steeper skew)
    vega: 20000,          // 2.0x volatility sensitivity
    lambda: 15000,        // 1.5x deviation sensitivity
    haircutSuppressor: 20000  // p=3 haircut curve
}

RiskConfig {
    coverageMin: 5000,          // 50% floor (0.01% units)
    coverageMax: 30000,          // 300% ceiling (wide bounds for volatile assets, 0.01% units)
    decayStartRatioBps: 950000,  // 95% decay trigger
    depthAmplifier: 500000,      // BTR Pool 2: depthAmp=500000 — aggressive amplification (50% virtual depth) for curated blue-chip pool after coverage is proven. Recommended default for new pools is 200000-300000.
    flags: 0x07                  // Swap + Flash + Liability Swap
}
```
