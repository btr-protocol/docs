# Composability

> Standard interfaces and integration patterns for external protocols to interact with AIMM pools

---

## 1. Overview

**Composability** enables external protocols, applications, and users to integrate with AIMM pools through standardized interfaces. Rather than forking code or maintaining custom adaptations, external developers use:

- **Public pool interfaces**: Standardized function signatures for swaps, liquidity provision, staking
- **ERC standards**: ERC-20 (tokens), ERC-3156 (flash loans), ERC-1167 (pool deployment as minimal proxy clones), ERC-1153 (transient reentrancy + price caching)
- **Singleton + clone architecture**: A single `Pool` reference impl backs every clone; one `Admin` / `Staking` / `Distributor` / `Flash` singleton serves every pool -integrators wire one set of ABIs.
- **SDK integration**: TypeScript utilities for quote calculation, transaction building, reward tracking

**Key principle**: AIMM prioritizes **open interfaces** over walled-garden integrations.

## 2. Core Integration Patterns

### 2.1. Token Swaps

**Interface**: `IPool` (on each `Pool` clone)

**Function**:
```solidity
function swap(
    address tokenIn,
    address tokenOut,
    uint256 amountIn,
    uint256 minAmountOut,
    address recipient
) external payable returns (uint256 amountOut);
```

**Integration flow**:

1. **Quote calculation**:
   - Call `getSwapQuote(tokenIn, tokenOut, amountIn)` (read-only)
   - Returns spread, fees, oracle prices, skew impact
   - Calculate `minAmountOut = quote.amountOut × (1 - slippageBps / 10000)`

2. **Token approval**:
   - If swapping non-native token: `approve(pool, amountIn)`
   - If swapping native (ETH): Send via `msg.value`

3. **Execute swap**:
   - Call `pool.swap()` with recipient address
   - Pool route: Swap module → Oracle update → Fee distribution
   - Emits: `SwapExecuted(tokenIn, tokenOut, amountIn, amountOut)`

4. **Slippage protection**:
   - Revert if `amountOut < minAmountOut`
   - Typical slippage: 0.1% - 1% depending on swap size

**SDK helper** (`sdk/src/flows/swap.ts`):
```typescript
const quote = await pool.getSwapQuote(tokenIn, tokenOut, amountIn);
const minOut = quote.amountOut * BigInt(9999n) / BigInt(10000n);  // 0.01% slippage
const tx = await pool.swap(tokenIn, tokenOut, amountIn, minOut, recipient);
```

**Common use cases**:
- DEX aggregators (1inch, 0x) routing through AIMM
- Liquidation bots swapping collateral for repayment assets
- Rebalancing protocols maintaining portfolio ratios
- Cross-chain operations (see [Section 5.2: Bridging](../../5.%20Developer%20Guides/5.2.%20Bridging.md))

### 2.2. Liquidity Provision

**Interface**: `IPool` (on each `Pool` clone)

**Functions**:

**Deposit**:
```solidity
function deposit(
    address token,
    uint256 amount
) external payable returns (DepositResult memory);

struct DepositResult {
    uint256 lpAmount;      // LP tokens minted
    uint256 actualDeposit; // Amount actually deposited (post-slippage)
}
```

**Withdrawal**:
```solidity
function withdraw(
    address token,
    uint256 lpAmount,
    uint256 minAmountOut
) external returns (WithdrawResult memory);

struct WithdrawResult {
    uint256 amountOut;     // Underlying tokens received
    uint256 lpBurned;      // LP tokens burned
}
```

**Integration flow**:

1. **Approve LP token (if withdrawing)**:
   - Not needed for deposit (tokens only)
   - For withdrawal: `lpToken.approve(pool, lpAmount)`

2. **Deposit flow**:
   - Check coverage ratio before deposit (ALM constraint)
   - Approve token: `token.approve(pool, amount)`
   - Call `pool.deposit(token, amount)`
   - Receive LP tokens for deposited assets
   - Can immediately stake LP tokens for emissions

3. **Withdrawal flow**:
   - Check if LP tokens exist: `lpBalance = lpToken.balanceOf(user)`
   - Call `pool.withdraw(token, lpAmount, minAmountOut)`
   - Receive underlying asset (may include haircut if coverage < 100%)

4. **Haircut mechanism**:
   - Applied only when **coverage ratio < 100%**
   - Protects remaining LPs during undercollateralization
   - See [Inventory Management §5](../1.1.%20Pricing/1.1.1.%20Inventory%20Management.md#5-withdrawal-haircut) for formula and examples

**SDK helper** (`sdk/src/flows/deposit.ts`):
```typescript
// Deposit
const result = await pool.deposit(token, amount);
console.log(`Received ${result.lpAmount} LP tokens`);

// Withdraw
const result = await pool.withdraw(token, lpAmount, minExpected);
console.log(`Withdrew ${result.amountOut} tokens`);
```

**Common use cases**:
- Protocol-owned liquidity (treasury deploying LP positions)
- Liquidity aggregators balancing across multiple pools
- Yield farming protocols earning LP emissions
- Market makers providing liquidity for trading incentives

### 2.3. Liquidity Staking (Emissions Eligibility)

**Interface**: `IStaking` (standalone module)

**Functions**:
```solidity
function stakeLP(
    address lpToken,
    uint256 amount
) external;

function unstakeLP(
    address lpToken,
    uint256 amount
) external;

function claimLiquidityTokenUnstaked(
    address lpToken
) external;
```

**Integration flow**:

1. **Obtain LP token**:
   - Deposit into pool → receive LP token (1:1 per pool)
   - Can also acquire LP tokens from other sources (trades, transfers)

2. **Stake LP tokens**:
   - Approve staking module: `lpToken.approve(stakingModule, amount)`
   - Call `stakingModule.stakeLP(lpToken, amount)`
   - Receive receipt token (sLP): `sLP.balanceOf(user)` increases
   - **Voting power**: sLP holdings contribute to governance voting power

3. **Earn emissions**:
   - 90% of total emissions distributed pro-rata to LP stakers
   - Per-asset earnings depend on: `(tvl_asset / total_tvl) × (coverage_weight) × (utilization_weight)`
   - Claim rewards anytime via: `distributor.claim()`
   - Rewards accumulate using O(1) accumulator pattern

4. **Unstake flow**:
   - Call `stakingModule.unstakeLP(lpToken, amount)`
   - Triggers 2-week cooldown (prevents flash-loan voting attacks)
   - After cooldown: `claimLiquidityTokenUnstaked(lpToken)` to withdraw LP tokens

**Key constraints**:
- **Minimum lock**: Default 14 days before LP tokens can be fully withdrawn (configurable by governance via `stakeLockDuration`)
- **Flow guard**: Protection against Just-In-Time liquidity attacks
- **Voting snapshot**: LP stake counted at governance snapshot block only

**SDK helper** (`sdk/src/rewards/`):
```typescript
// Stake LP tokens
await stakingModule.stakeLP(lpToken, amount);

// Claim rewards
const earned = await distributor.claim();

// Unstake (2-week wait)
await stakingModule.unstakeLP(lpToken, amount);
// After 14 days...
await stakingModule.claimLiquidityTokenUnstaked(lpToken);
```

**Common use cases**:
- LP incentive farming (earn BTR for providing liquidity)
- Governance participation (sLP holders have voting power)
- Liquidity mining campaigns (highest earning pool per period)
- Strategic capital positioning (earn emissions while maintaining voting power)

### 2.4. Governance Token Staking

**Interface**: `IStaking` (standalone module)

**Functions**:
```solidity
function stakeGov(uint256 amount) external;

function unstakeGov(uint256 amount) external;

function delegateVoting(address to) external;
```

**Integration flow**:

1. **Stake BTR**:
   - Approve treasury: `btToken.approve(stakingModule, amount)`
   - Call `stakingModule.stakeGov(amount)`
   - Receive sBTR (staked BTR) 1:1
   - Voting power = `sBTR × btPrice / 1e18` (scaled by govMultiplier, subject to quadratic damping)

2. **Delegate voting power**:
   - Call `sBTR.delegate(delegatee)` or `stakingModule.delegateVoting(delegatee)`
   - Voting power transfers to delegatee at next snapshot block
   - Revoke: `delegatee.delegate(address(0))` to self-delegate

3. **Earn staking rewards**:
   - 5% of total emissions allocated to BTR stakers
   - Pro-rata distribution: `rewards = (sBTR_balance / total_sBTR) × annual_emissions`
   - Claim: `stakingModule.claim()` or governance dashboard

4. **Unstake flow**:
   - Call `stakingModule.unstakeGov(amount)`
   - Immediate withdrawal (no cooldown for governance token)
   - sBTR burned, BTR transferred to user

**Key constraints**:
- **Self-delegation required**: Must call `delegate(self)` to enable voting power (ERC-20 Votes pattern)
- **Snapshot voting**: Voting power calculated at snapshot block height
- **No unstake cooldown**: Unlike LP staking, BTR unstaking immediate

**SDK helper** (`sdk/src/rewards/voting-power.ts`):
```typescript
// Stake
await stakingModule.stakeGov(amount);

// Delegate
await sBTR.delegate(delegatee);

// Claim rewards
await stakingModule.claim();

// Voting power calculation (off-chain)
const votingPower = await computeVotingPower({
  sBTR: userBalance,
  sLP: userLPBalances,
  btPrice, lpPrices,
  applyDamping: true
});
```

**Common use cases**:
- Governance participation (stake BTR, earn voting power)
- Protocol alignment (stakers earn 5% of emissions)
- Delegation to governance organizations (stake, delegate to council)
- Multi-sig treasury management (treasury stakes BTR, delegates to Counsel)

### 2.5. Flash Loans

**Standard**: ERC-3156 (`IERC3156FlashLender`)

**Interface**: `IFlash` (on the Flash singleton; first arg is `address pool`)

**Functions**:
```solidity
function flashLoan(
    address pool,
    IERC3156FlashBorrower receiver,
    address token,
    uint256 amount,
    bytes calldata data
) external returns (bool);

function maxFlashLoan(address pool, address token)
    external view returns (uint256);

function flashFee(address pool, address token, uint256 amount)
    external view returns (uint256);

// Receiver contract must implement:
interface IERC3156FlashBorrower {
    function onFlashLoan(
        address initiator,
        address token,
        uint256 amount,
        uint256 fee,
        bytes calldata data
    ) external returns (bytes32);
}
```

**Integration flow**:

1. **Create flash loan receiver**:
   ```solidity
   contract MyFlashReceiver is IERC3156FlashBorrower {
       IERC3156FlashLender pool = IERC3156FlashLender(...);

       function executeFlashLoan(address token, uint256 amount) external {
           bytes memory data = abi.encode(msg.sender, token, amount);
           pool.flashLoan(this, token, amount, data);
       }

       function onFlashLoan(
           address initiator,
           address token,
           uint256 amount,
           uint256 fee,
           bytes calldata data
       ) external override returns (bytes32) {
           // Now have 'amount' of 'token' in contract
           // Execute any logic (swap, arbitrage, liquidation, etc.)

           // Must repay: amount + fee
           IERC20(token).transfer(msg.sender, amount + fee);

           return keccak256("ERC3156FlashBorrower.onFlashLoan");
       }
   }
   ```

2. **Query flash loan parameters**:
   - `maxFlashLoan(token)` → Maximum amount available for borrowing
   - `flashFee(token, amount)` → Borrowing fee in basis points (90 bps = 0.09%)

3. **Execute flash loan**:
   - Initiate via `pool.flashLoan(receiver, token, amount, data)`
   - Pool transfers tokens to receiver contract
   - Receiver executes arbitrary logic in `onFlashLoan()`
   - **Must repay** in same transaction: `token.transfer(pool, amount + fee)`
   - Transaction reverts if repayment fails

**Fee calculation**:
- **Flat fee**: `fee = amount × flashFeeBps / 10000`
- Current: 0.009% (90 basis points per year annualized)
- Treasury receives 25%, LP emission pool receives 75%

**Common use cases**:
- **Arbitrage**: Borrow assets, execute profitable trades, repay from profits
- **Liquidations**: Liquidator borrows repayment token, pays penalty fee, retains spread
- **Collateral swaps**: Swap collateral types without external capital
- **Yield farming**: Borrow to temporarily increase LP position, repay from earned fees
- **Oracle manipulation tests**: Security teams testing oracle resilience

**SDK helper** (`sdk/src/flows/`):
```typescript
// Create receiver contract (Solidity)
const receiverAddress = await deployFlashLoanReceiver();

// Execute flash loan
const fee = await pool.flashFee(token, amount);
const tx = await pool.flashLoan(
    receiverAddress,
    token,
    amount,
    abi.encode({ arbitrageData })
);
```

## 3. Advanced Integration Patterns

### 3.1. Liability Swaps (Hedging)

**Function**: `swapLiability(address tokenIn, address tokenOut, uint256 lpAmountIn, uint256 minLpAmountOut)`

**Use case**: Liquidity providers hedging directional exposure while maintaining LP position.

**Example**:
- LP has 100 LP-tokens for ETH/USDC pool (50 ETH equivalent, 50 USDC equivalent)
- Expects ETH to fall, wants to reduce ETH exposure
- Executes liability swap: Sell ETH liabilities, buy USDC liabilities
- Result: LP position becomes more USDC-heavy (reduced ETH risk)

---

### 3.2. Protocol-Owned Liquidity (POL)

**Pattern**: Treasury deposits assets, receives LP tokens, stakes for emissions.

```solidity
// Treasury flow
1. treasury.approve(pool, 1M_USDC);
2. (lpAmount,) = pool.deposit(USDC, 1M_USDC);
3. stakingModule.stakeLP(lpToken, lpAmount);
4. // Now earning 90% of pool emissions
5. // LP fees auto-compounded in treasury
```

**Benefits**:
- Treasury earns trading fees
- Provides baseline liquidity
- Eligible for full emission rewards
- Can be unwound if needed for operations

---

### 3.3. Cross-Pool Routing

**Pattern**: Swap routes through multiple pools if single pool lacks depth.

**Example**: ETH → DAI may route ETH → USDC (Pool A) → DAI (Pool B)

**Implementation**:
- Quote system returns `routeHops` and `hopAmounts`
- SDK aggregates best route
- Pool interface handles multi-hop execution (via modules or caller orchestration)

---

### 3.4. Reward Distribution Integration

**Contract**: `IDistributor`

**Campaign types**:
1. **Points campaigns**: Issue non-transferable SBT tokens (achievement tracking)
2. **Token campaigns**: Distribute actual ERC-20 rewards

**Integration pattern**:
```solidity
// Create campaign (governance-approved)
uint256 campaignId = distributor.createTokenCampaign(
    rewardToken, // e.g., BTR
    manager      // Who updates merkle roots
);

// Manager periodically updates reward snapshots
distributor.updateCampaignRoot(
    campaignId,
    merkleRoot,
    updatedAt,
    totalClaimable
);

// Users claim via merkle proof
distributor.claimCampaign(
    campaignId,
    index,
    address,
    totalEarned,
    merkleProof
);
```

**Common use cases**:
- Incentive campaigns (e.g., "earn 1M BTR for $10M TVL")
- Airdrop distribution (merkle tree of allocations)
- Governance rewards (voting participation bonuses)

```typescript
    minOut,

```

## 5. Safety & Best Practices

### 5.1. Slippage Protection

**Always specify minimum output**:
```solidity
// ✅ Safe: Revert if output too low
swap(tokenIn, tokenOut, amountIn, minAmountOut, recipient);

// ❌ Unsafe: Accept any output
swap(tokenIn, tokenOut, amountIn, 0, recipient);
```

**Recommended slippage bounds for volatile swaps**:
- Small swaps (< $10k): 0.01% - 0.05%
- Medium swaps ($10k - $100k): 0.05% - 0.25%
- Large swaps (> $100k): 0.25% - 1%+

### 5.2. Flash Loan Safety

**Always verify flash loan callback**:
```solidity
function onFlashLoan(
    address initiator,
    address token,
    uint256 amount,
    uint256 fee,
    bytes calldata data
) external override {
    require(msg.sender == address(pool), "Untrusted pool");
    // Execute logic
    require(
        IERC20(token).balanceOf(address(this)) >= amount + fee,
        "Insufficient repayment"
    );
    IERC20(token).transfer(msg.sender, amount + fee);
}
```

### 5.3. Staking Cooldowns

**Respect 2-week LP unstaking delay**:
```solidity
// Day 1: Initiate unstake
stakingModule.unstakeLP(lpToken, amount);

// Day 15: Can now claim
stakingModule.claimLiquidityTokenUnstaked(lpToken);
```

**Don't assume immediate liquidity.**

### 5.4. Voting Power Snapshots

**Voting power is frozen at snapshot block**:
```typescript
// Snapshot block set during proposal creation
// e.g., block 18500000

// These counts:
await sBTR.transfer(user, 100); // Before snapshot

// These don't:
await sBTR.transfer(user, 100); // After snapshot
```

**Flash loans won't work for voting attacks** due to snapshot finality.
