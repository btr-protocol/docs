# Curation & Pool Deployment

> Deploying and configuring custom AMM pools

---

## 1. Overview

**Pool curation** enables any protocol, exchange, or operator to:

- **Deploy custom pools** with their own token lists, pricing curves, and risk parameters
- **Manage pool ownership** via timelock governance (protect against rash decisions)
- **Add/remove tokens dynamically** using an asset request + execute pattern
- **Extend functionality** by deploying custom hook contracts for specialized logic
- **Implement modules** as diamond-pattern facets for unlimited extensibility

> For protocol-level white-labelling (branding, frontend integration), see [Section 5.3: White Labelling](../../../5.\ Developer\ Guides/5.3.\ White\ Labelling.md).

**Key principle**: AIMM is **fully permissionless** — anyone can deploy, no central approval needed.

## 2. Pool Ownership & Control

### 2.1. Pool Deployment

Any address can deploy a new `PoolProxyV1` instance:

```solidity
// Step 1: Deploy proxy (no initialization)
PoolProxyV1 pool = new PoolProxyV1();
// - Immutably sets msg.sender as DEPLOYER
// - Prevents initialization front-running
// - Each deployment is independent (no factory registry)

// Step 2: Initialize with base config
pool.initialize(
    owner,           // Who controls the pool
    baseToken,       // Anchor token (e.g., USDC, USDT)
    wnative,         // Wrapped native asset (WETH, WAVAX, etc.)
    FeeParams(...)   // Protocol fee configuration
);
// - Only DEPLOYER can call (prevents hijacking)
// - owner becomes the pool administrator
```

**Access control**:
- **Deployer**: Can only initialize once
- **Owner**: Controls all administrative functions (add assets, set hooks, freeze assets)
- **Timelock**: Protects major changes (1-7 day delays)
- **Users**: Can deposit, withdraw, swap freely (subject to pool settings)

### 2.2. Ownership Model

| Action | Authority | Timelock | Notes |
|--------|-----------|----------|-------|
| **Freeze/unfreeze asset** | Owner | None | Emergency powers, immediate effect |
| **Set pricing anchor** | Owner | None | Adjust pricing tree topology |
| **Set flow cooldown** | Owner | None | Configure JIT protection (0-256 sec) |
| **Add asset** | Owner | 1 day | Request → wait → execute pattern |
| **Update risk config** | Owner | 1 day | Change per-asset risk parameters |
| **Register module** | Owner | 3 days | Add new functionality (swap, flash, etc.) |
| **Transfer ownership** | Owner | 3 days | Handoff to new operator |
| **Migrate base token** | Owner | 7 days | Change anchor (critical, rarely done) |

## 3. Token Curation

### 3.1. Adding Tokens (Asset Manager Pattern)

**Two-phase process** (prevents configuration mistakes):

**Phase 1: Request asset addition**
```solidity
pool.requestAddAsset(
    address token,                      // ERC-20 token address
    IPoolV1.OracleConfig calldata cfg,  // Price feed configuration
    IPoolV1.RiskConfig calldata risk,   // Risk parameters
    IPoolV1.LiquidityProfile calldata profile,  // Liquidity distribution
    uint16 minFeeBps,                   // Minimum swap fee (0.01% = 1 basis point)
    uint8 decimals,                     // Token decimals (6, 8, 18, etc.)
    uint64 initialPrice,                // Starting price in B64 format
    uint32 initialFastVolEMA,           // Fast volatility estimate
    uint32 initialSlowVolEMA            // Slow volatility estimate
);
```

**Phase 2: Execute after 1-day timelock**
```solidity
vm.warp(block.timestamp + 1 days);  // Wait 1 day
pool.executeAddAsset(token);
```

### 3.2. Oracle Configuration

**Three oracle modes**:

| Mode | Primary | Secondary | Use Case |
|------|---------|-----------|----------|
| **Internal** | Pool's TWAP | None | Stablecoins (USDC, USDT), most tokens |
| **External** | Chainlink/Pyth | Pool TWAP fallback | Exotic assets, price feeds unavailable internally |
| **Hybrid** | Fallback chain | Pool TWAP | Maximum safety for critical assets |

**Configuration struct**:
```solidity
struct OracleConfig {
    address primary;            // Primary feed (address(pool) = internal TWAP)
    address secondary;          // Fallback (address(0) = none)
    bytes32 feedId;            // Oracle identifier (Pyth feed ID, etc.)
    uint16 modeFlags;          // Oracle mode bits
    uint8 accDecimals;         // B64 accumulator precision (6-18)
}
```

**Precision selection** (`accDecimals`):
- **6 decimals**: Stablecoins (USDC, USDT, USDS) - price change: $0.000001
- **8 decimals**: Natural assets (LINK, UNI) - price change: $0.00000001
- **12 decimals**: ETH/major assets - price change: $0.000000000001
- **18 decimals**: Exotics with extreme volatility - maximum precision

### 3.3. Risk Configuration

```solidity
struct RiskConfig {
    uint16 decayStartRatioBps;     // Coverage ratio to activate decay (e.g., 9800 = 98%)
    uint16 coverageMin;          // Critical safety floor (e.g., 5000 = 50%)
    uint16 coverageMax;          // Critical safety ceiling (e.g., 20000 = 200%)
    uint32 decaySlope;             // Liability decay rate per year (%)
    uint16 depthAmplifier;         // Depth curve sensitivity (higher = steeper)
    uint16 flags;                  // Feature enablement
}
```

**Coverage-based logic**:
- **100%+ coverage**: No penalties, optimal pricing
- **98-100%**: Decay activates, withdrawal haircuts begin (encourages rebalancing)
- **50% floor**: Critical; swaps may be disabled if breached

**Flags**: See [Parametrization §3.1](../1.1.\ Pricing/1.1.7.\ Parametrization.md#31-risk-flags-bitfield) for complete flag reference.

Common flags: `SWAP_ENABLED` (0x0002), `FLASH_ENABLED` (0x0010), `FROZEN` (0x0001).

### 3.4. Liquidity Profile (Distribution Curve)

**Segment-based distribution** for customizable price impact:

```solidity
struct LiquidityProfile {
    uint8[16] weights;       // Relative depths per segment (sum = 200 = 100%)
    int8[17] knots;          // Price offset knots (weights.length + 1 points)
}

// Example: Concentrated (like Uniswap v4 concentrated liquidity)
weights = [100, 50, 20, 10, 0, ...];     // Front-loaded (4 segments)
knots = [-50, -25, 0, 25, 50, ...];      // 5 knots for 4 segments, narrow range

// Example: Uniform (like Uniswap v3 full range)
weights = [50, 50, 50, 50, 0, ...];      // Balanced (4 segments)
knots = [-50, -25, 0, 25, 50, ...];      // 5 knots for 4 segments, wide range
```

**Impact on pricing**:
- Concentrated profiles: Lower slippage near mid-price, higher on extremes
- Uniform profiles: Consistent slippage across range
- Custom profiles: Tailor to expected trading patterns (stablecoin vs. exotic)

### 3.5. Token Removal (Freeze vs. Deletion)

**Freeze token** (temporary):
```solidity
pool.freezeAsset(token);   // Disable swaps, deposits allowed for cleanup
pool.unfreezeAsset(token); // Re-enable
```

**Permanent removal**: Not supported (prevents liquidity loss). Instead:
- Owner should freeze the asset
- Allow users to withdraw LP positions over time
- Migrate to new pool if needed

## 4. Custom Hooks

> For complete hook interface and developer documentation, see [Hooks System](./1.3.2.\ Hooks.md).

### 4.1. Curator Use Cases

Hooks enable **per-asset customization** without modifying core pool logic:

| Use Case | Hook | Example |
|----------|------|---------|
| Deposit bonuses | `postDeposit` | Return `hookIncentiveOut` to add LP bonus |
| Withdrawal taxes | `postWithdraw` | Return `exitFee` to charge exit fee |
| Dynamic fees | `preSwap` | Return `newFeeBps` based on volatility |
| KYC/whitelist | `preDeposit` | Revert for non-approved addresses |

### 4.2. Installing Hooks

```solidity
// Owner sets custom hook for specific token
pool.setHook(token, address(customHookContract));
```

Hooks are called automatically on relevant events—no additional integration needed from users.

## 5. Custom Modules (Diamond-Lite Extensibility)

### 5.1. Module System Architecture

AIMM uses **EIP-2535 Diamond pattern lite** — a lightweight function selector registry:
- Each `PoolProxyV1` has a `mapping(bytes4 selector => address)` module registry
- Fallback delegates calls to registered implementation based on selector

**Benefits**:
- Modular code (each function family in separate contract)
- Upgradeable (swap implementations without redeployment)
- Composable (combine features freely)
- Efficient (minimal proxy overhead)

### 5.2. Pre-Built Modules

| Module | Functions | Status |
|--------|-----------|--------|
| **CoreV1** | `swap()`, `deposit()`, `withdraw()`, `swapLiability()`, `donate()` | ✅ Core |
| **AdminV1** | Asset management, risk config, timelock | ✅ Core |
| **InternalOracleV1** | TWAP tracking, price feeds | ✅ Core |
| **FlashV1** | ERC-3156 flash loans | ✅ Core |
| **StakingV1** | LP/Gov token staking | ✅ Core |
| **DistributorV1** | Reward distribution | ✅ Core |
| **RescueV1** | Emergency asset recovery | ✅ Core |

### 5.3. Implementing Custom Modules

**Custom module requirements**:
1. Define interface with function selectors
2. Implement using `LibStorage.load()` for shared state access
3. Register via `requestModuleUpdate()` with 3-day timelock

**Example interface** (MarketMaker incentives):
```solidity
interface IMarketMakerV1 {
    function claimMakerRewards(address token) external returns (uint256 amount);
    function getMakerAPY(address token) external view returns (uint256);
}
```

**Registration flow**:
```solidity
pool.requestModuleUpdate(address(module), selectors, initData);
// Wait 3-day timelock
pool.executeModuleUpdate();
// Call via: IMarketMakerV1(address(pool)).claimMakerRewards(token);
```

## 6. Complete Deployment Walkthrough

### Example: Stablecoin Exchange Deployment

```solidity
// 1. Deploy pool
PoolProxyV1 pool = new PoolProxyV1();

// 2. Initialize
pool.initialize(
    owner,  // My exchange address
    USDC,   // Anchor token
    WETH,   // Native wrapper
    FeeParams(25, 5, ...)  // 25% protocol fee, 0.005% flash fee
);

// 3. Register core modules (swap, deposit, oracle, etc.)
bytes4[] memory selectors = [
    0x1234abcd,  // swap selector
    0x5678efgh,  // deposit selector
    // ... etc
];
pool.requestModuleUpdate(address(coreModule), selectors, "");
vm.warp(block.timestamp + 3 days);
pool.executeModuleUpdate();

// 4. Add stablecoins
pool.requestAddAsset(
    USDC,
    OracleConfig(address(pool), address(0), 0, 0, 6),
    RiskConfig(9800, 5000, 31709791, 10000, SWAP_ENABLED),
    LiquidityProfile(...),  // Uniform distribution
    10,  // 0.1% min fee
    6,   // 6 decimals
    encodeB64(1e6),  // $1.00
    10000,  // volatility
    10000
);
vm.warp(block.timestamp + 1 days);
pool.executeAddAsset(USDC);

// Similar for USDT, USDS, etc.

// 5. Deploy custom hook: 0.5% deposit bonus
DepositBonusHook hook = new DepositBonusHook(500);  // 5 bps bonus
pool.setHook(USDC, address(hook));
```

## 7. Security Considerations

### 7.1. Timelock Protection

All major changes are **timelocked** to allow users to exit if governance is compromised:

- **1-day delay** (low risk): Asset additions, risk config updates
- **3-day delay** (medium risk): Module registration, ownership transfers
- **7-day delay** (critical): Base token migration

**Example**: If owner proposes adding a malicious token, users have 1 day to withdraw before it takes effect.

### 7.2. Access Control

- **Owner only**: Freeze assets, add tokens, register modules
- **No owner transfer without notice**: 3-day timelock prevents hijacking
- **No user impact**: Users can always deposit/withdraw/swap (no owner powers here)

### 7.3. Hook Safety

- Hooks can return arbitrary values, but **cannot access other tokens** (isolated execution)
- Hook failures revert transactions (prevent orphaned rewards)
- Owner can disable hooks anytime

## 10. Getting Started

**Quick checklist** for deploying your custom pool:

1. **Define token list**: Which tokens to support?
2. **Configure risk**: What coverage ratios, decay rates?
3. **Design hooks**: Any custom deposit bonuses, fees, incentives?
4. **Deploy**: Use provided factory or deploy directly
5. **Launch**: Pool is operational!

**No approval required** — permissionless by design.
