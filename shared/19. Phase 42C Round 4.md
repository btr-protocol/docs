### A1 -Invariants & State

**Pool._exec token conservation (Pool.sol:422-448):**
- tkIn side: external deposit credits the pool's tkIn balance by `amtIn` (via
  `_pull` at Pool.sol:256). State updates: `aIn.reserves += amtIn - inFee`,
  `$.protocolFees[tkIn] += inFee`. Sum of state deltas on tkIn = `amtIn`. ✓
- tkOut side: `aOut.reserves -= q.amountOut`, `$.protocolFees[tkOut] += q.protoFee`.
  User receives `out` via `_push` (Pool.sol:269). The LP fee `feeOut` is implicit:
  `q.amountOut = acc.currentAmount - feeOut` (LibPricing.sol:310) so the LP fee
  remains in `aOut.reserves` (LP earnings). The protoFee book-keeping at line 441
  is a ledger entry against the same physical reserves bucket; `collectProtocolFees`
  later burns it from reserves into the Treasury (Treasury.sol:191-194).
  Conservation: pool's physical tkOut balance decreases by exactly `out` (user
  withdrawal). ✓
- `_reconcile` (Pool.sol:529-535) closes the gap between the predicted `q.amountOut`
  and the actual `out` after pre/post hook fees, by adjusting `a.reserves`.
- See **R4-A1-1 LOW** below for a subtle accounting asymmetry between pre- and
  post-swap hook-fee paths.

**Treasury balance accounting:**
- Vesting: `vested(s)` (Treasury.sol:170-186) clamps cliff/post-cliff arithmetic;
  `vested > s.claimed ? vested - s.claimed : 0` prevents underflow. Mint paths
  (`claimVested`, etc.) all gated by `maxSupply` check via `_checkAndUpdateEmissions`
  (validated in R1, unchanged). ✓
- Emissions cap change (Treasury.sol:251-261): `maxSupply = (maxSupply - oldCap) + newCap`;
  underflow-safe iff `oldCap ≤ maxSupply`, which is invariant since maxSupply is
  initially seeded from `oldCap` plus other ceilings. ✓ (no test directly proves
  invariant -see §3 disposition.)

**Bridge LZ accounting (Bridge.sol:393-435):**
- `_checkAndUpdateLimit` resets daily counters on `cfg.day != today`. Ratio cap
  for inbound: `limitDecoded * cfg.inRatio / RATIO_DENOM`. `M.add64` overflow-checks
  via b64 packed arithmetic. Excess revert path (line 427-431) consistent. ✓
- `failedMessages` keyed by guid, peer trust verified pre-queue (validated R1

#### R4-A1-1 (LOW, **NEW**) -Post-swap hook fee silently donated to LP reserves
- **File:line:** `src/modules/Pool.sol:294-298` (vs `:291-292`).
- **Repro:** `_processSwap` calls `_applyHookFee(extraFee)` BEFORE `_exec` (line
  291), so updates to `q.protoFee`/`q.lpFee` are picked up by the ledger write at
  Pool.sol:441 (`$.protocolFees[tkOut] += q.protoFee`). It then calls
  `_applyHookFee(uint256(delta))` AFTER `_exec` (line 295). The post-`_exec`
  invocation mutates `q.protoFee`/`q.lpFee` IN MEMORY only (Pool.sol:524), but the
  protocolFees ledger has already been committed. `_reconcile(actual, expected)`
  (Pool.sol:298, 529-535) then adds the delta back into `aOut.reserves`. Net
  effect: post-swap hook protoFee is accounted as LP yield in reserves rather
  than tracked in the protoFees ledger; the `Swapped` event still emits the
  inflated `q.protoFee` value, leading to event-vs-storage divergence.
- **Severity rationale:** LOW. Not exploitable (no value escapes the pool). Two
  observable effects: (a) `Swapped.protoFee` event field over-reports vs ledger
  by the post-hook portion; (b) post-hook proto allocation is silently donated
  to LPs. Pre-existing pattern, not introduced by R1/R2/R3 fixes; hooks rarely
  use `postSwap` delta>0 to push fees, so practical impact is minimal.
- **Fix sketch (deferred):** either persist post-hook `q.protoFee` delta to
  `$.protocolFees[tkOut]` between line 295 and 298, or document explicitly that
  post-swap hook fees are LP-donations and only emit pre-hook protoFee in the
  event.

### A4 -Numerical & Precision

**TWAP overflow at extremes (`InternalOracle.getFastTWAP`):**
- `spot * uint256(mult)` (InternalOracle.sol:75). `mult ≤ ORACLE_PBPS + int32.max ≈
  2.16e9`. `spot` decoded via `M.b64To1e18`: in worst-case b64 mantissa+shift
  encoding (mantissa ≤ 2^52, max positive shift = 61) → `spot ≤ ~4.5e76`.
  Theoretical product `~1e86 > uint256.max ≈ 1.16e77` → overflow possible.
- In practice `encodeOffset1e18` policy bounds `spot` to realistic price ranges
  (e.g. ≤ 1e30 for 30-decimal tokens), where product is ≤ 2.16e39 ≪ 2^256. Safe.
- This is the R3-A4-1 INFO; re-verified, severity unchanged.

**LibPricing.powWad domain (LibPricing.sol:570-578):**
- Solady `FixedPointMathLib.powWad(int256(x), int256(y))` with `x ∈ [0, WAD]`,
  `y ∈ (0, WAD]`. Solady requires `x > 0` for `y != 0` (uses `lnWad`); the
  short-circuit at line 571 (`x == 0 → 0`) covers the only domain hole.
- Casts `int256(x)` and `int256(y)` safe: `WAD = 1e18 ≪ 2^255`. ✓
- Negative-result clamp `r < 0 ? 0 : uint256(r)` (line 577) defensive against
  fractional underflow. ✓

**netCoverageImpact PBPS scaling:**
- `(amountOut * feeBps) / 1_000_000` where `feeBps = acc.maxFeePath` in PBPS.
  Worst-case: `feeBps ≤ uint16.max = 65535` → `(amountOut * 65535) / 1e6` overflow
  at `amountOut > 2^256 / 65535 ≈ 1.77e72`. Impossible in practice (amountOut is
  token units bounded by reserves ≤ uint128.max ≈ 3.4e38). ✓

**Rounding biases:**
- `Pool.sol:437` `(amtIn * spreadBps / 2) / 1_000_000` floors odd PBPS (R1 A4-3
  LOW carry-over).
- `LibPricing.sol:390` `if (priceOut == 0) priceOut = 1;` clamp (R1 A4-2 LOW
  carry-over).
- `splitFee` (LibPricing.sol:482) `protoFee = (totalFee * protoShare) / 100;
  lpFee = totalFee - protoFee;` -rounding-down protoFee, residual in lpFee.
  LP-favorable, intentional. ✓

**Adversarial probe -encoding round-trip stress:**
- `LibOracle.encodeOffset1e18(spot, twap)`: `ratio = (twap * ORACLE_PBPS) / spot`.
  Overflow at `twap * ORACLE_PBPS`: `twap ≤ 1e30, ORACLE_PBPS = 1e7` → product
  ≤ 1e37 ≪ 2^256. ✓ Saturation clamps at int32 boundaries (line 36-37) protect
  storage. ✓
- `getFastTWAP` decode round-trips: `spot=1e18, twap=0.95e18` → `off = -500_000` →
  decode `1e18 * 9_500_000 / 10_000_000 = 0.95e18`. ✓ (regression test
  `test_R2_A1_1_fastTWAP_roundTrip_*` covers this.)

No new A4 findings.
