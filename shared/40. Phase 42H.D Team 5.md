## 4. New / outstanding gaps

### G18 (NEW · CRITICAL · Governance + Test rigor) -UUPS upgrade path is non-functional and tests certify the bug
**Where:** `Treasury.sol:336`, `Bridge.sol:393`, `Router.sol:345` all call `this.upgradeToAndCall(newImpl, "")` from `executeUpgrade()`.
**Why broken:** `this.X(...)` is an external call → enters proxy from outside → `msg.sender == address(proxy)` inside `upgradeToAndCall` → `_authorizeUpgrade(...) onlyOwner` (Solady `Ownable`) reverts with `Unauthorized` because the proxy ≠ owner. Solady's `UUPSUpgradeable.upgradeToAndCall` (`shared/.deps/solady/src/utils/UUPSUpgradeable.sol:69-75`) is `payable onlyProxy { _authorizeUpgrade(newImplementation); ... }` -it does not relax the auth context.
**Test confirms bug:** `UUPSTimelock.t.sol:64-73` literally asserts `vm.expectRevert(Ownable.Unauthorized.selector); _executeUpgrade(OWNER);` AFTER the timelock delay -i.e., the test treats post-delay revert as the *expected* outcome and asserts pending state stays intact.
**Impact:** **No upgrade can ever succeed in production for Treasury, Bridge, or Router.** Once deployed, the contracts are effectively immutable for the wrong reason. Coupled with the timelock contract, governance has no working upgrade path.
**Fix:** call internal `_upgradeToAndCallUUPS` (or pull Solady-internal lib equivalent), OR change pattern to `LibBytes.delegatecall(impl, callData)` style, OR remove the wrapper and have governance call `upgradeToAndCall` *directly* from the owner address (post-delay, after a `validate(pendingUpgrade)` precondition is enforced via a separate `validateThenUpgrade` modifier or off-chain governance UI). Either way, replace the `this.upgradeToAndCall` self-call.
**Test fix:** the post-delay test must assert `_authorizeUpgrade` succeeds AND `ERC1967Utils.implementation()` slot updates to `newImpl`, not that the call reverts.

### G19 (NEW · Governance + Test rigor) -Bridge LZ peer-config timelock has no test
`Bridge.sol:334-348`: `requestSetPeer` → `executeSetPeer` is timelocked via `TL.pack/validate`, but `grep -n "executeSetPeer\|requestSetPeer" test/` returns 0 hits. A misuse (e.g., wrong `eid`, double-request) is silently accepted. Same shape as G14 in spirit.
**Fix:** extend the abstract `UUPSTimelockHarness` pattern to a `BridgePeerTimelockTest` that exercises the request/execute/cancel + double-request reverts.

### G20 (NEW · Governance) -`cancelOperation` has no `OpType` whitelist
`Bridge.sol:351-355`: `cancelOperation(bytes32 id) external onlyOwner` deletes any `pendingOps[id]` + `pendingData[id]` regardless of which `OpType` produced it. If a hash collision were ever achieved (extremely low risk given 2^256 space, but still), or more pragmatically if the caller passes a stale id from a different op family, the cancel succeeds silently. Not an attack vector but a correctness/UX concern: there's also no `cancelUpgrade` re-entry path covered when `cancelOperation` is used on the upgrade id specifically.

### G21 (NEW · Cross-repo) -`BridgeableERC20.supportsInterface` made `virtual` but `GovToken` still does not override
