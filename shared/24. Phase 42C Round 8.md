### A1 -Invariants & state (Pool token conservation, all branches)

#### Where does the discrepancy come from?

LibPricing.sol:308–313:
```
uint256 halfSpread = uint256(quote.spreadBps) / 2;
uint256 feeIn  = (amountIn * halfSpread) / 1_000_000;
uint256 feeOut = (acc.currentAmount * halfSpread) / 1_000_000;
(quote.protoFee, quote.lpFee) = splitFee(feeOut + (feeIn * acc.currentAmount) / amountIn, $.feeParams.protoShare);
quote.amountOut = acc.currentAmount - feeOut;
```

`q.protoFee` (= `pf₀`) is denominated in `tOut`. It includes both:
- `feeOut`-derived proto portion (real: pool retains `feeOut` of tOut as `acc.currentAmount > Q`).
- `(feeIn * acc.currentAmount / amountIn)`-derived proto portion (phantom: this is `feeIn` of `tIn` re-quoted in tOut; the actual `feeIn` was already retained on the tIn side at Pool.sol:463 `protocolFees[tkIn] += inFee`).

Pool.sol:464 debits `Q = acc.currentAmount − feeOut` from `R(tOut)`. So reserves over-state LP holdings by `feeOut + 0`. The credit at Pool.sol:465 of `P(tOut) += pf₀` then claims those LP-over-reserves PLUS the phantom `feeIn`-converted portion.

For the invariant `B = ΣR + ΣP` to hold, line 464 should debit `Q + pf₀` (so that the proto portion explicitly leaves LP-owned reserves and enters protocol-owned escrow). It currently debits only `Q`.

#### A1-R8-1 (HIGH -re-derived from source, NOT a regression of R6, but pre-existing & undiscovered)

**Finding:** Token-conservation invariant `B(tOut) == R(tOut) + P(tOut)` is violated by `q.protoFee` per multi-token swap. Pool.sol:464–465 debits reserves by `Q = q.amountOut` only and credits `P(tOut) += q.protoFee`, but `q.protoFee` represents tokens that were never carved out of `R(tOut)` separately from `Q`. Net effect: ledger over-states pool tOut holdings by `q.protoFee` per swap. Treasury collection via `Admin.collectProtocolFees` (Admin.sol:74–85) silently drains LP reserves' over-collateralization until exhausted; subsequent collects revert.

**Magnitude estimate:** `pf₀ = (feeOut + feeIn·Q'/amtIn) · protoShare/100`. With `protoShare = 25`, `spreadBps = 30 bps = 30_000 PBPS`, swap = 1000 USDC: `feeOut ≈ 1000 · 15_000/1_000_000 = 15`; `pf₀ ≈ 15 · 0.25 ≈ 3.75 USDC` per swap. At 1000 swaps/day, drift ≈ 3,750 USDC/day until Treasury collection drains it.

**Severity rationale:**
- Not user-facing exfil (no tokens leak to swap recipient) → not CRIT.
- Persistent ledger inflation that drains LP reserves on collection → strictly worse than a one-shot drift.
- Pre-existing; un-introduced by R6. R6's reserves-debit at Pool.sol:319 actually hides the larger structural issue by stitching only the hook-side delta.

**Decision required:** treat as a NEW HIGH (R8 dirty) or as carry-over LOW (R8 clean with caveat).

**Determination:** This is functionally identical in structure to the R4-A1-1 finding originally raised in Phase 42C R4 (silent insolvency drift on hook proto). The R4 patch handled the hook side only. Same class of bug remains on the quote side. Severity per R4 precedent was HIGH. Therefore **HIGH**.

**A1-R8-2 (LOW -pre-hook proto silently donated to LP):** Pool.sol:291 `_applyHookFee(extraFee, …)` increments `q.protoFee += pf_pre` but `protoDelta` (Pool.sol:295,301) is computed AFTER this call (`protoBefore = q.protoFee` at line 297 already includes `pf_pre`). ### A4 -Numerical / precision

- **LibOracle.\_applyOffset / InternalOracle.getFastTWAP fullMulDiv equivalence**: 
  - `_applyOffset` (LibOracle.sol:21): `fullMulDiv(price1e18, uint256(multiplier), ORACLE_PBPS)`.
  - `getFastTWAP` (InternalOracle.sol:78): `fullMulDiv(spot, uint256(mult), LibOracle.ORACLE_PBPS)`.
  - Both use identical formula `price · (ORACLE_PBPS + offset) / ORACLE_PBPS`. Symmetric with `encodeOffset1e18` (LibOracle.sol:36): `ratio = emaPrice · ORACLE_PBPS / currentPrice; off = ratio − ORACLE_PBPS`. Round-trip: `decode(encode(spot, ema)) ≈ ema` modulo single-step floor rounding in encode + multiply rounding in decode. Equivalent. ✓
  - Edge: `multiplier ≤ 0` clamps both -`_applyOffset` returns 1 (line 19), `getFastTWAP` returns `M.encodeB64(1, 18)` (line 75). Symmetric. ✓
- **Pool.sol:319 `uint128(protoDelta)` cast**: `protoDelta = q.protoFee − protoBefore` where `q.protoFee` increments only via `_applyHookFee.splitFee` (proto = total · protoShare/100, total = `Δ`). So `protoDelta ≤ Δ`. `Δ` originates from `IPoolHooks.postSwap` int256 return. If hook returns `Δ > uint128.max`, `protoDelta` could exceed `uint128.max` → silent truncation. BUT: `_applyHookFee` clamps `out` to ≥0 (line 550), so practically `Δ ≤ Q ≤ uint128.max` (since `Q` originates from `aOut.reserves` which is `uint128`). Bound holds. ✓ Pre-existing LOW (carried from R7 A4-1; not introduced by any Phase 42C round).
- **LibPricing.\_executeLeg priceOut==0 revert**: LibPricing.sol:392–394: `priceOut = price18 / (10^(18-decimalsTo))`; if `amountOut * 10^scale = 0` (round-down to zero), `priceOut == 0` → reverts (Phase 42D A4-2). Replaces prior silent clamp-to-1 that poisoned oracle. Verified clean. ✓
- **LibPricing.netCoverageImpact PBPS units**: Line 513: `totalOut = amountOut + (amountOut * feeBps) / 1_000_000`. `feeBps` is in PBPS (per docstring at line 511). Line 510 signature parameter is `uint16 feeBps`; max value 65535 = 6.5535%. Sufficient for `acc.maxFeePath` which is `uint16` (LibPricing.sol:288, fed from `asset.maxFeeBps` which is also uint16). ✓ No truncation.
- **Pool.sol:319 underflow**: `aOut.reserves` was just credited `extraFee + Δ` by `_reconcile`. Before that it was `R₀ - Q`. Net before R6 debit: `R₀ - Q + extraFee + Δ = R₀ - out`. Subtracting `pf_post`: `R₀ - out - pf_post`. Since `pf_post ≤ Δ ≤ Q ≤ R₀`, `R₀ - out - pf_post ≥ 0` iff `out + pf_post ≤ R₀` iff `Q − extraFee − Δ + pf_post ≤ R₀`. Worst case `out = 0, pf_post = Δ` ⇒ need `Δ ≤ R₀`. Since `_exec` already pre-checked `R₀ ≥ Q + minLiquidity` (line 455–456), and `Δ` from hook is bounded by `Q` (else `out` clamp at line 550 makes `out = 0` and `Δ_effective` is capped), no underflow. ✓
- **Pre-existing LOW carry-over (A1-R8-1 "fix" suggestion)**: implementing the proper fix would require Pool.sol:464 to read `aOut.reserves -= uint128(q.amountOut + q.protoFee)` and a corresponding underflow guard. Out of scope for R8 verdict but logged.
