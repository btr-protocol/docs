## 2. R5 fix correctness analysis -END-TO-END VALUE-FLOW TRACE

The critical question (A3 hypothesis): **Is the augmented `out` purely a bookkeeping-internal value used only for `_reconcile`, or does it propagate to `_push` and become what the user receives?**

### 2.1 Trace `out` from `_processSwap` to user

| Step | Site | Statement |
|---|---|---|
| 1 | `Pool.sol:278` | `function _processSwap(...) private returns (uint256 out)` -named return. |
| 2 | `Pool.sol:280` | `out = q.amountOut;` -initial value = quoted output. |
| 3 | `Pool.sol:291` | `out = _applyHookFee(...)` -pre-swap hook fee deducted (`out` shrinks). |
| 4 | `Pool.sol:292` | `_exec(...)` -debits `aOut.reserves -= q.amountOut`, credits `protocolFees[tkOut] += q.protoFee` from quote (out NOT mutated by `_exec`). |
| 5 | `Pool.sol:297` | `out = _applyHookFee($, uint256(delta), q, out);` -post-swap hook fee deducted; `q.protoFee += pf`. |

### 2.2 Verdict -A3 hypothesis CONFIRMED. R5 fix is INCORRECT.

The augmented `out` is **NOT** a bookkeeping-internal value. It is the named return that propagates verbatim to `_push` (line 269). Therefore, after R5:

- User receives `q.amountOut - hookFee + protoDelta` instead of `q.amountOut - hookFee`.
- The protocol-fee portion of the post-swap hook fee is **transferred to the user**.
- `protocolFees[tkOut]` ledger is incremented by `protoDelta` (line 302), but the corresponding tokens have left the contract (sent to user at line 269).
- When Treasury later calls `Admin.collectProtocolFees` (`src/modules/Admin.sol:74-83`), it attempts to transfer `protocolFees[tkOut]` from contract balance -but those tokens are gone. Either:
  - (a) the transfer reverts on insufficient balance (if no excess reserves); OR
  - (b) the transfer succeeds by **draining LP reserves** (the contract holds aggregate `Σ reserves[t] + Σ protocolFees[t]`; draining proto-fees that aren't backed by real tokens silently extracts from LP-owned token balance).

This is a **HIGHER-severity regression than R4-A1-1**: R4-A1-1 (pre-R5 state) was a silent accounting drift internal to the pool; R6's confirmed bug **directly leaks tokens to a swap user every time a post-swap hook returns `delta > 0` with `protoShare > 0`**.

### 2.3 Why the R5 unit test missed this

`test/Phase42CR5Test.t.sol:100, 181` defines:
```
uint256 userOut = out - protoDelta;
```
The test author **manually subtracted `protoDelta`** when computing the user-facing amount -implicitly assuming the production code would also do so. Production code (`Pool.sol:269`) does **not** subtract; it pushes the augmented `out`. The fuzz invariant
```
reserves + protoLedger + userOut == reservesBefore
```
holds *only because the test models a fictional `userOut = out - protoDelta`* that does not match the on-chain code path. The test is a pure-math replica of `_processSwap` internals, not an integration test of `swap()`'s actual external behaviour, so the leak is invisible.

### 2.4 Correct fix sketch

Two viable options that preserve token conservation **and** push only `userOut` to the recipient:

- **Option a' (corrected):** keep `out += protoDelta` for `_reconcile`'s benefit, then **subtract `protoDelta` after `_reconcile`** so the named return reverts to user-amount semantics:
  ```solidity
  if (protoDelta != 0) {
      $.protocolFees[tk[1]] += protoDelta;
      out += protoDelta;          // bookkeeping value for _reconcile
  }
  ...
  _reconcile($.assets[tk[1]], out, q.amountOut);
  if (delta > 0) out -= protoDelta;   // restore user-facing value
  ```
  (or use a separate local `bookkeepingOut` for `_reconcile`).
- **Option b (preferred -narrower mutation):** leave `out` untouched, perform an explicit reserve debit after `_reconcile`:
  ```solidity
  _reconcile($.assets[tk[1]], out, q.amountOut);
  if (delta > 0 && protoDelta != 0) {
      $.assets[tk[1]].reserves -= uint128(protoDelta);
  }
  ```
  This keeps the user-facing semantics of `out` intact and makes the proto-fee debit explicit.

### A3 -economic/MEV

**Finding A3-R6-1 (CRIT): R5 patch leaks `protoDelta` of `tokenOut` to every swap user whose route triggers a post-swap hook with `delta > 0` and `protoShare > 0`.**

- File: `src/modules/Pool.sol:259, 269, 273-313`.
- Trace: see §2.1 above (steps 8 → 13).
- Magnitude per swap: `protoDelta = floor(hookFeeDelta * protoShare / 100)`. With default-magnitude `protoShare` (e.g. 25%) and a hook fee on the order of basis points of trade size, this is non-trivial.
- Severity escalation rationale:
  - Before R5: silent insolvency drift (HIGH) -proto fee donated to LP reserves but ledgered as proto.
  - After R5: **direct token outflow to swap users** (CRIT) -proto fee tokens are sent out of the contract while the ledger still claims them. First-mover Treasury-collect attempts will succeed by draining LP reserves; later attempts revert.
- MEV angle: a hook author colluding with a searcher could craft `postSwap` to return a calibrated `delta > 0` so that `protoDelta` becomes a free siphon on every routed swap. The user receiving the bonus need not be aware -the leak is automatic.

### A4 -numerical/precision

No new findings.
- `out + protoDelta` overflow: both are `uint256`; in this context `out ≤ q.amountOut ≤ uint128.max` (enforced via reserve cast at `Pool.sol:456`) and `protoDelta ≤ delta ≤ uint128.max` (the `_postSwap` delta is consumed by `_applyHookFee` which clamps `out` to ≥0). Sum is bounded by `2·uint128.max << uint256.max`. ✓ no overflow.
- `_reconcile`'s `d ≤ uint128.max` guard (`Pool.sol:548`) is unaffected -`actual = out + protoDelta`, `expected = q.amountOut`, and `actual` could even be `> expected` if `protoDelta > delta` (impossible since `protoDelta ≤ delta`). ✓.
