## 3. Four-auditor independent pass (do not echo R11)

### A1 -invariants / state / conservation

Re-derived token conservation from balance ↔ ledger first principles for every flow path. Found **two latent bugs in `Flash.sol`** never exercised by any test and never raised in R1-R11.

**F-A1-R12-1 (HIGH): `Flash.flashLoan` post-callback balance check is off-by-`amount`** -`src/modules/Flash.sol:53`.

Sequence:
- Line 46: `balanceBefore = _balanceOf(token)` -captured BEFORE the push.
- Line 47: `_push(token, address(receiver), amount)` -`safeTransfer` decreases pool balance by `amount`.
- Line 49: borrower callback returns; ERC-3156 borrowers repay `amount + fee` (Aave/MakerDAO convention).
- Line 52: `balanceAfter = _balanceOf(token)` -should equal `balanceBefore - amount + (amount + fee) = balanceBefore + fee` for a compliant borrower.
- Line 53: `if (balanceAfter < balanceBefore + amount + fee) revert` ⇒ requires `repaid >= 2 * amount + fee`. Correct check is `balanceAfter >= balanceBefore + fee`.

Impact: every standards-compliant flash loan call **reverts** at line 53. Feature is non-functional. No funds at risk because the path never completes; this is a feature DoS. Severity HIGH (not CRIT) ∵ no fund loss path is reached, but a top-level public ERC-3156 entry is broken across all conditions. No flash-loan test exists in the suite (`grep -rn flashLoan test/` → zero hits) which is why the bug has survived twelve audit rounds.

Recommended fix: `if (balanceAfter < balanceBefore + fee) revert ...;`

**F-A1-R12-2 (HIGH, latent behind F-A1-R12-1): reserves-overwrite absorbs prior `protocolFees + leftover`** -`src/modules/Flash.sol:55`.

Even after F-A1-R12-1 is fixed, line 55 sets `asset.reserves = uint128(balanceAfter - protoFee)`. Pre-flash invariant: `pool.balance(token) = asset.reserves + protocolFees[token] + leftover` (where `leftover` = stuck or salvageable surplus). Post-flash with line 53 corrected:
- `balanceAfter = balanceBefore + fee = reserves_old + protocolFees_old + leftover + fee`.
- Code overwrites: `new reserves = balanceAfter - protoFee = reserves_old + protocolFees_old + leftover + lpFee`.
- Code increments: `new protocolFees = protocolFees_old + protoFee`.
- Sum: `new reserves + new protocolFees = reserves_old + 2 · protocolFees_old + leftover + fee`.
- Actual pool balance: `reserves_old + protocolFees_old + leftover + fee`.
- ⇒ **Ledger over-counts `protocolFees_old` by exactly that amount** every flash loan. Reserves silently absorb prior fee escrow + any leftover.

Operational consequence (post F-A1-R12-1 fix): each flash loan steals the entire current `protocolFees[token]` accumulator into LP reserves. When Treasury subsequently calls `collectProtocolFees`, the transferred ledger value (`protocolFees_old + protoFee`) exceeds the actual recoverable balance segregated for protocol -the Admin transfer call would either revert (insufficient balance) or, if reserves are debited to satisfy, drain reserves. Either way the conservation invariant `pool.balance == Σ reserves + Σ protocolFees` is violated.

Recommended fix: replace `asset.reserves = uint128(balanceAfter - protoFee)` with `asset.reserves += uint128(lpFee)` (where `lpFee = fee - protoFee`), preserving the original ledger split.

**Severity classification:** F-A1-R12-2 is HIGH (under post-fix conditions of F-A1-R12-1, this becomes a fund-misappropriation path: protocolFees ledger over-allocates, Treasury collection reverts or drains LP reserves). Currently latent only because F-A1-R12-1 prevents the flash loan from completing -but both must be fixed together.
