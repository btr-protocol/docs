## 3. New 4-auditor pass -R2 findings

### A1 -Invariants & State

#### R2-A1-1 (HIGH) -`InternalOracle.getFastTWAP` formula uses wrong constant + wrong direction
- **File:line:** `src/modules/InternalOracle.sol:68-74`.
- **Repro:**
  - `fastOffset` is encoded by `LibOracle.encodeOffset1e18` (LibOracle.sol:32-39) against
    `ORACLE_PBPS = 10_000_000` (LibOracle.sol:12), explicitly documented as "10× finer than PBPS".
  - Encoding: `ratio = (emaPrice * ORACLE_PBPS) / currentPrice; off = ratio - ORACLE_PBPS`.
  - Correct decode (matches `LibOracle._applyOffset` LibOracle.sol:15-20):
    `twap = currentPrice * (ORACLE_PBPS + off) / ORACLE_PBPS`.
  - Remediation instead computes (line 73):
    `twap = (spot * C.PBPS) / (C.PBPS + off)` -uses `C.PBPS = 1_000_000` AND inverts the formula
    (numerator / (denom+off) instead of (denom+off) / denom).
  - Worked example: `spot = 1.0e18, true_twap = 0.95e18`:
    - `encodeOffset1e18(1e18, 0.95e18)` → `ratio = 9_500_000`, `off = -500_000` (i.e. -5 % in ORACLE_PBPS).
    - Correct decode: `1e18 * (10_000_000 - 500_000) / 10_000_000 = 0.95e18` ✓.
    - Remediation result: `1e18 * 1_000_000 / (1_000_000 - 500_000) = 1e18 * 2 = 2.0e18` -off by **>2×**.
  - The on-chain check `mult <= 0` (line 72) only guards the trivially-degenerate denominator;
    for any realistic `fastOffset` ∈ [-10⁵, +10⁵] the result is wildly wrong (off by ~10×).
  - The R1 test `test_A1_2_fastTWAP_formula_notSpot` (test/Phase42CR1Test.t.sol:264-271) is a **pure-math
    regression that re-asserts the same wrong formula**; it does not cross-validate against
    `LibOracle.encodeOffset1e18 / _applyOffset`, so it passes despite the bug.
- **Severity rationale:** R1 A1-2 was LOW because `getFastTWAP` was an unused external view returning
  spot. R2 raises it because (a) the docstring + name now claim *true* TWAP, integrators *will* trust
  it, and (b) the attack surface is wider than R1: a downstream consumer relying on this view in a
  flash-loan context now reads a number that can deviate from the true TWAP by 10× -a worse
  oracle-attack primitive than the original "returns spot".
- **Fix sketch (one line):**
  ```solidity
  uint256 twap = (spot * uint256(int256(LibOracle.ORACLE_PBPS) + int256(off))) / LibOracle.ORACLE_PBPS;
  ```
  (matches `LibOracle._applyOffset` exactly), guarded by `int256(ORACLE_PBPS) + off > 0` (extreme
  negative offset → return 1, mirroring `_applyOffset`). Also rewrite the regression test to
  `LibOracle.encodeOffset1e18(spot, expected_twap)` → `getFastTWAP` → assert `≈ expected_twap`.

No other invariant findings. Carry-over from R1: A1-3 LOW (Bridge stale `pendingOps`), A4-3 LOW

- **File:line:** `src/Router.sol:260` -gates by `isPool` not `isOfficialPool`.

- **Fix sketch:** either tighten exec to `isOfficialPool`, or relax discovery to `isPool` + document.

#### R2-A3-1 (LOW) -Router `getBestRoute` seeds per-hop `minOut` with zero slippage tolerance
- **File:line:** `src/Router.sol:54, 149-150, 194-196`.
- **Repro:** `minOut = quoted_amount` exactly. Any block-to-block price drift between quote and exec
  (e.g. user signs tx at block N, executes at N+1 after another swap moves the pool) reverts the
  whole route. Functionally equivalent to "always 0 slippage" unless integrator constructs `Route`
  themselves with a discount.
- **Severity:** LOW (UX, not exploit). Documented behavior may save users from accidental
  high-slippage execution; for sandwich-resistance the exact-quote floor is conservatively safe.
- **Fix sketch:** apply a small default tolerance (e.g. `quote * 9_995 / 10_000` = 5 bps) inside
  `getBestRoute`, OR document that integrators MUST adjust per-hop `minOut` before submitting.

A3-2 `feeBps` source verified: `acc.maxFeePath` is the per-path upper bound on the path's spread cap
(LibPricing.sol:284-285), consistent with conservative impact estimation. Unit is PBPS (matches the
`/ 1_000_000` divisor). Naming `feeBps` is misleading but math is correct.

### A4 -Numerical & Precision

A4-1 powWad replacement verified:
- Short-circuits preserved (LibPricing.sol:571-574): `x=0→0`, `x=WAD→WAD`, `y=0→WAD`, `y=WAD→x`.
- Solady `FixedPointMathLib.powWad` domain: signed-wad input; requires `x > 0` for `y != 0` (uses
  `lnWad`). Our `x = progress ∈ [0, WAD]`, `y = exponentWad ∈ (0, WAD]` (always positive). The
  `x = 0` short-circuit covers the only domain hole.
- Cast `int256(x)` safe: `x ≤ WAD = 1e18 ≪ 2^255`. `y ≤ WAD = 1e18` likewise.
- Negative result clamp `r < 0 ? 0 : uint256(r)` (line 577): defensive for fractional powers near
  zero where Solady may underflow; correct.
- A3-2 `feeBps` (PBPS scaling): `(amountOut * feeBps) / 1_000_000` matches PBPS unit; pre-remediation
  hard-coded `* 100 / 100_000` was ALSO PBPS-scaled (== 0.1%); semantic continuity preserved when
  `maxFeePath ≈ 1000 PBPS = 0.1%`. For larger maxFeePath the new code yields larger impact estimates
  → wider spreads → safer. ✓

(R1 A4-2 LOW `priceOut→1` clamp, A4-3 LOW `spreadBps/2` floor, A4-4 INFO `_skewToDepth` unchecked,
A4-5 INFO `MAX_DEV_THRESHOLD` event-only -unchanged.)
