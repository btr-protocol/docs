## 4. New 4-auditor findings

### A1 -invariants/state

**Finding A1-R5-1 (HIGH): R4-A1-1 Pool._processSwap protoFee ledger sync double-counts post-swap hook fee.**

- File: `src/modules/Pool.sol:294-306`
- Trace:
  1. `_exec` (line 451) credits `$.protocolFees[tkOut] += q.protoFee` and `aOut.reserves -= q.amountOut`. So reserves are debited the gross amount sent out.
  2. Post-swap hook with `delta > 0` invokes `_applyHookFee` (line 297) which mutates `q.protoFee += pf` and reduces `out` by the hook fee → `out = q.amountOut - delta`.
  3. New code (line 301) credits `$.protocolFees[tk[1]] += protoDelta`.
  4. `_reconcile($.assets[tk[1]], out, q.amountOut)` (line 306) detects `actual=out < expected=q.amountOut` and credits reserves back: `aOut.reserves += (q.amountOut - out) = delta` (`Pool.sol:543-544`).
- Net result for protoDelta:
  - Real tokens retained in pool: `+delta` (user got `out`, not `q.amountOut`).
  - Storage: `aOut.reserves +delta` AND `protocolFees[tkOut] +protoDelta`.
  - Sum of claims (reserves + protocolFees) = `delta + protoDelta` > `delta` (real balance retained).
- **Double-count by `protoDelta`.** When Treasury later calls `Admin.collectProtocolFees`, it pulls `protocolFees[t]` from contract balance and zeros the ledger (`Admin.sol:79-83`). Reserves are unchanged. Subsequent swap on this asset will draw against inflated reserves → eventual under-collateralization or revert deep into liability path.
- Severity: HIGH (silent accounting drift, not caught by existing tests because no test asserts `reserves + protocolFees == realBalance` invariant after a post-swap hook delta>0 path).
- Pre-fix behaviour was "silently donate to LP reserves" -bookkeeping was consistent, just allocated to LPs not protocol. Phase 42D introduced the drift by adding the ledger entry without offsetting the reserve credit in `_reconcile`.
- Fix sketch: subtract `protoDelta` from the reserve adjustment, e.g. set `out = out + protoDelta` before `_reconcile` (so reserves are credited only `delta - protoDelta` = LP share), OR after `_reconcile` add `aOut.reserves -= uint128(protoDelta)`.
- Repro: any swap on a pool with a `HOOK_POST_SWAP` hook returning `delta > 0` AND `feeParams.protoShare > 0`. Existing `Phase42DTest.t.sol` does NOT cover this path (the test docstring at line 14 explicitly states "covered by inspection of existing integration tests" -no behavioural test was added for R4-A1-1, despite it being the highest-touch invariant change).

No new findings.

Pre-fix path overflowed silently in Solidity ^0.8.35 (revert via checked arith) at extreme inputs. Net: post-fix path is a strict superset of pre-fix in-domain behaviour. ✓
- Boundary tests:
  - `a=0` → 0 (both paths) ✓
  - `b=0` → 0 (both paths) ✓ (but `mult <= 0` short-circuit at LibOracle.sol:19 / InternalOracle.sol:75 prevents this)
  - `c=1` would N/A (ORACLE_PBPS const = 1e7)
  - `a=type(uint256).max, b=2`: pre-fix reverts; post-fix returns `floor(2*max/1e7)` exact. ✓ (defence-in-depth as claimed)
- Round-trip via `encodeOffset1e18`: unchanged -`_applyOffset` decode is the inverse of `encodeOffset1e18` modulo int32 saturation (LibOracle.sol:34-41); rounding identical pre/post.
