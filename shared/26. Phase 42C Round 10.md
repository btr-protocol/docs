#### Bridge inbound rate-limit -NEW finding

**F-A4-R10-2 (MEDIUM, pre-existing -Bridge, not R8/R9 scope but in R10 widened-scope mandate)**

Location: `Bridge.sol:170-178` `_tryCheckAndUpdateLimit` (inbound, lzReceive path).

```solidity
function _tryCheckAndUpdateLimit(address token, uint256 amount, TokenConfig memory cfg) internal returns (bool) {
    if ((cfg.flags & FLAG_UNLIMITED) != 0) return true;
    uint256 outLimit = M.decodeB64(cfg.limitOutB64, 18);
    uint256 inLimit = (outLimit * uint256(cfg.inRatio)) / RATIO_DENOM;
    uint256 currentIn = M.decodeB64(cfg.bridgedInB64, 18);
    if (currentIn + amount > inLimit) return false;
    tokenConfigs[token].bridgedInB64 = M.encodeB64(currentIn + amount, 18);
    return true;
}
```

Compare with the outbound twin `_checkAndUpdateLimit` at Bridge.sol:402-441, which correctly performs day-rollover reset (lines 409-414):
```solidity
uint16 today = uint16(block.timestamp / 1 days);
if (cfg.day != today) {
    cfg.day = today;
    cfg.bridgedOutB64 = 0;
    cfg.bridgedInB64 = 0;
}
```

**Inbound path skips the day-rollover reset.** Consequence:
1. `bridgedInB64` accumulates across calendar days, never resets unless an *outbound* transfer of the same token triggers `_checkAndUpdateLimit` and resets both counters.
2. Once cumulative inbound ever exceeds the daily inbound limit, every subsequent `lzReceive` for that token is queued as `failedMessage(FC_RATE_LIMIT)`.
3. Recovery is possible (owner `recoverFailedMessage`), so funds are not lost -but inbound bridging is silently DoS'd until either (a) any outbound for that token resets the day counter, or (b) the owner intervenes.
4. The view `getRemainingLimits` (line 381-397) DOES virtualize the day-rollover (`outUsedB64 = (cfg.day == today) ? cfg.bridgedOutB64 : 0`), so it reports correct remaining limits while the actual storage check uses stale `bridgedInB64`. View↔logic divergence can mislead operators.

**Severity: MEDIUM** -not a fund-loss vector (queue+recover protects funds), but a functional regression that breaks documented inbound rate-limit semantics and creates operator surprise.

**Recommendation**: hoist the day-rollover block into a shared helper, or duplicate the reset block at the head of `_tryCheckAndUpdateLimit`. Also update the `cfg` parameter -note: inbound takes `cfg` by **memory** (line 170), so any reset must write directly to `tokenConfigs[token]` storage (already done for the increment at line 176).

This finding was missed by R9 ∵ R9 scope was R8 remediation correctness, not Bridge.
