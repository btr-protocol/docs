# ADR-001, Per-Contract Upgrade Strategy

**Status:** Accepted (Phase 42H.D)
**Scope:** btr/dex/evm + btr/alm/evm + btr/shared
**Supersedes:** -
**Authors:** Phase 42H.D remediation, Team 1 audit gap G4

## Context

Audit Team 1 (Phase 42H.D) flagged a "mixed upgrade story" across the BTR
protocol stack: Treasury and Router extend `UUPSUpgradeable`, while Pool is an
EIP-1167 minimal-proxy clone (non-upgradeable per-instance), and the remaining
singletons (Admin, Staking, Distributor, Flash, PoolFactory, Bridge) are
non-upgradeable. The audit asked whether this divergence is intentional and, if
so, that we record the rationale rather than conform to a single pattern.

This ADR formalises the per-contract decision and the trade-offs.

## Decision

| Contract | Upgrade pattern | Trigger | Timelock |
|---|---|---|---|
| `dex/Treasury` | UUPS (`UUPSUpgradeable`) | gov vote + impl swap | 7d |
| `dex/Router` | UUPS (`UUPSUpgradeable`) | gov vote + impl swap | 7d |
| `dex/Pool` | EIP-1167 clone (non-upgradeable per instance); `referencePool` swap on `PoolFactory` produces NEW clones | gov vote + new ref impl | 7d |
| `dex/PoolFactory` | Non-upgradeable (redeploy + migrate registry) | full gov migration | n/a |
| `dex/Admin` `dex/Staking` `dex/Distributor` `dex/Flash` `dex/Bridge` | Non-upgradeable (redeploy + AccessControl rewire) | gov vote + redeploy | n/a |
| `shared/AccessControl` | Non-upgradeable (singleton; rotate via 2-step `pendingOwner` + ROTATION_TIMELOCK) | owner rotation | 7d |
| `alm/Vault` | EIP-1167 clone (non-upgradeable per instance); impl swap via `alm/Factory` | gov vote + new ref impl | 14d (FACTORY_TIMELOCK) |
| `alm/Factory` | Non-upgradeable | redeploy | n/a |
| `alm/Swapper` | Non-upgradeable | redeploy | n/a |

## Rationale (per pattern)

### UUPS, Treasury, Router

- **Why upgrade?** Treasury holds gov-token mint authority + per-pool fee
  accounting; Router holds aggregator routing logic + cross-pool swap state.
  Both are change-prone: tokenomics emissions schedules evolve; new aggregator
  primitives (1inch v6, Bebop, etc.) are added quarterly.
- **Why UUPS over Transparent/Beacon?** UUPS keeps upgrade authority in the
  impl (cheaper proxy), and we already gate it through a 7d timelock + the
  shared singleton AccessControl owner. Beacons are over-engineered for two
  contracts; Transparent proxies waste gas on every call (admin slot SLOAD).
- **Single-key vector mitigation.** The 7d timelock (`requestUpgrade` →
  `executeUpgrade`) means a compromised owner key leaves a 7-day public window
  for community veto / governance migration. Treasury's mint authority is
  protected by `chargeKeeperKill` (cluster-wide kill budget on `AccessControl`)
  which can freeze emissions even mid-timelock.
- **Acknowledged trade-off.** A single key can in principle queue a malicious
  impl. We accept this because (a) full immutability of Treasury would force
  redeploy + migration of all per-pool fee state on every emission tweak, a
  prohibitive operational cost, and (b) the 7d window + kill switch + sole
  owner = `AccessControl` (governable) bound the blast radius.

### EIP-1167 clones, Pool, Vault

- **Why clone?** Both contracts are deployed *per pair / per ALM strategy* and
  carry per-instance state (reserves, oracle accumulators, position metadata).
  Clones share immutable bytecode → ~95% deploy gas savings vs full deploys at
  hundreds of pools / vaults.
- **Why non-upgradeable per-instance?** Pools and Vaults are user-facing
  liability holders; in-place storage upgrades on a live pool risk silent
  corruption (e.g., struct field reordering). We forbid in-place mutation; new
  impls go live only via factory `referencePool` / impl swap, producing new
  clones for new deployments while existing clones continue running their
  pinned impl.
- **Migration story.** Existing-pool migrations (when needed) are explicit gov
  ops: pause old pool → drain via `withdraw` → re-deposit into new clone.
  Painful by design, it forces governance to budget the migration cost
  upfront rather than ship a silent storage upgrade.
- **Storage layout pin (Pool).** `IPool.PoolStorage $` lives at slot 0 (no
  ERC-7201 namespacing), see Pool.sol NatSpec for the slot-0 decision and
  upgrade-safety constraints (append-only fields).

### Non-upgradeable singletons, Admin, Staking, Distributor, Flash, Bridge, AccessControl, PoolFactory, alm/Factory, alm/Swapper

- **Why immutable?** These contracts encode protocol invariants (access
  control, fee distribution, flash loan accounting, cross-chain bridge math).
  Any change demands a full audit cycle anyway; a "soft upgrade" via UUPS
  would lull governance into shipping changes without re-audit. Forced
  redeploy + AccessControl rewire makes the upgrade event visible and
  auditable.
- **Migration story.** Redeploy → grant new contract roles via AccessControl
  → revoke old contract roles → update front-end addresses. ROTATION_TIMELOCK
  (7d) governs role rotations.

## Considered alternatives

1. **All UUPS.** Rejected, opens too wide a single-key blast radius, esp. on
   Pool / Vault clones where per-instance impl swaps would be uncoordinated.
2. **All immutable.** Rejected, would force a full Treasury redeploy + fee
   state migration on every emissions tweak (prohibitive op cost).
3. **Beacon proxies for Pool / Vault.** Rejected, beacons re-introduce a
   single-key vector across ALL clones simultaneously, the very thing the
   per-clone pinned-impl model avoids.

## Consequences

- Auditors / integrators MUST consult this ADR before assuming a contract is
  upgradeable. Front-end / SDK code paths that rely on impl pinning (e.g.,
  cached ABI hashes) remain valid for clones; UUPS contracts require version
  detection.
- Future contracts default to **non-upgradeable** unless they meet the bar set
  by Treasury/Router (frequent, mechanical, gov-supervised state change).
- Treasury hardening (move to non-upgradeable + redeploy migration strategy)
  remains a future-work item; it is NOT executed in Phase 42H.D pending a
  formal cost/benefit analysis of fee-state migration tooling.
