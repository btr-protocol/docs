If `q.amountOut + q.protoFee > uint128.max`, then `minReq > uint128.max` ≥ aOut.reserves (uint128), so revert path fires before the truncating cast. Truncation unreachable. ✓
- **F-A4-2 (INFO)** -Same applies to Pool.sol:147 `uint128(amt + q.protoFee)` in `_withdrawTo`: line 141 checks `assetTo.reserves < amt` (not amt+q.protoFee). **Potential under-bound**: if `q.protoFee > 0`, the check at line 141 misses the protoFee component. Reserves are uint128, so subtraction `assetTo.reserves -= uint128(amt + q.protoFee)` will underflow-revert if exceeded. So safe but redundant. However, the `amt+q.protoFee` could still overflow uint128 cast silently if `amt+q.protoFee > uint128.max` AND ≤ reserves (uint128.max bound). Bound: amt ≤ assetTo.reserves ≤ uint128.max, q.protoFee ≤ uint128.max. Sum could be 2·uint128.max. Cast truncates → reserves debited by truncated value → ledger credited by full q.protoFee → conservation BROKEN. **BUT**: assetTo.reserves is uint128, and `assetTo.reserves -= uint128(amt+q.protoFee)` enforces `assetTo.reserves >= uint128(amt+q.protoFee)` (truncated). If the truncated value is small but the real value is large, debit is too small → reserves over-credit. This is theoretical since amt itself comes from haircut applied to LibPricing q.amountOut, both bounded by available reserves. **Practical risk: NEGLIGIBLE; theoretical risk: present.** Severity: INFO (pre-existing pattern; same shape as Pool.sol:475 but without the explicit pre-check). Recommend: add `if (amt + q.protoFee > type(uint128).max) revert` or strengthen line 141 check to include q.protoFee.
