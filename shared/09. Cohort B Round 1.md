## 2. 3-expert debate

### Round 1 -proposals

**B1 Tests -proposal.** Three-tier fixture topology in `peripheral/evm/test/`:
(a) `mocks/` canonical: `MockERC20`, `MockChainlink`, `MockSequencer` -consumed by both via remapping `@btr/peripheral/test/=lib/peripheral/test/`. Dex `src/mocks/{MockERC20,MockFaucet}` deleted from `src/` (off ABI surface) and re-homed to peripheral.
(b) `fixtures/Common.sol` -minimal abstract: deploys AC + PP + Treasury, makes 5 named addrs (`owner, keeper, treasury, alice, bob`), inherits `forge-std/Test`. ~80 LOC. Modeled on alm `Common.t.sol` lines 22–55 (the role + core block).
(c) Consumer extension: `alm/evm/test/AlmCommon.t.sol extends Common` adds Swapper+Factory+Vault+Dex+pool funding; `dex/evm/test/DexCommon.t.sol extends Common` adds PoolProxy+Router+modules. Domain-specific stays per-repo. Each repo's `*.t.sol` extends its own AlmCommon/DexCommon, never Common directly.

**B2 Deploy -proposal.** Single `peripheral/evm/script/Deploy.base.sol` abstract w/ helpers:
`_deployAC(owner)`, `_deployPP(sequencer, gracePeriod)`, `_bindTreasury(ac, treasury)`, `_writeRegistry(path, json)`. Each consumer extends:
- `alm/evm/script/Deploy.s.sol` (the 5-vault loop, post-Bootstrap).
- `dex/evm/script/Deploy.s.sol` (replaces both DeployBSCFork + TestDeploy; chain-specific via JSON).
Chain config externalized to `<repo>/evm/deployments/<chain>.json` (e.g. `alm/evm/deployments/base.json`, `dex/evm/deployments/bsc.json`). JSON schema: `{ chainId, rpcEnv, lifiRouter, settler, sequencer, createX, salts: {...}, tokens: [{symbol, addr, cap}] }`. Drop chain-hardcoded constants from `.sol`. `--fork-url` + `FOUNDRY_PROFILE=fork` replaces `DeployBSCFork.s.sol` variant. `Configure.s.sol` becomes peripheral helper invoked post-deploy.

**B3 Operations -proposal.** Keeper greenfield placement: **`back/services/keeper/`** matching Phase 30 pattern (back already houses `agents/`, `collector/`, `swap/`). Rust + nxr ws consumer + on-chain tx submitter via alloy. Surface API: `keeper rebalance --vault 0x...`, `keeper tend --pool 0x...`, `keeper harvest --vault 0x...`. Reads vault list from `<repo>/evm/deployments/<chain>.json` registries. NOT colocated with `dex/` or `alm/` -keeper is a consumer of both, lives one layer up.

CI uniformity: **single template** `peripheral/.github/workflow-template/foundry-evm.yml`; alm + dex copy into their `<repo>/.github/workflows/test.yml` with same forge pin via `.tool-versions` (foundry version line). `forge --version` gate fails CI on drift. `forge --match-contract` convention: `<Subject>Test` (alm: `VaultUserTest`, `VaultAdminTest`; dex: `LibPricingTest`, `LibOracleTest`). Standardize naming: `<File>Test` always, no `*.t.sol` orphans without matching contract name.

### Critiques (round 1)

- **B1 critiques B2**: JSON-driven chain config introduces parse/decode complexity in Solidity (`vm.parseJson` + decoding nested arrays). For 2 surfaces × 3 chains it's overkill; flat env vars per chain may be simpler. *Counter*: alm already uses env (`HTTPS_RPC_8453`, `CAP_*`); registry JSON is the *output*, not the input. Hybrid: env for secrets/addresses, JSON for tokens-list + salts-list (already what dex does for salts).
- **B1 critiques B3**: `back/services/keeper/` is correct dir but Rust + alloy isn't validated for tx-submission reliability vs ethers-rs / cast. Demand: pick alloy explicitly and pin version; ethers-rs is unmaintained.
- **B2 critiques B1**: 3-tier fixture (Common → AlmCommon/DexCommon → tests) deepens inheritance graph. Solidity fixture inheritance is fragile when storage layouts diverge. *Counter*: peripheral `Common` has no storage-layout concerns (it's a test contract, never deployed); inheritance is fine.
- **B2 critiques B3**: keeper at `back/services/keeper/` couples keeper release cadence to back-services release cadence. If keeper needs a hotfix and back/agents is mid-refactor, deploys block. *Counter*: each service in `back/services/` has its own Cargo.toml + dockerfile (per Phase 30 pattern), so cadence is independent at the build level; only the workspace root coordinates.
- **B3 critiques B1**: peripheral mocks must be exempt from peripheral's prod ABI export. Place under `peripheral/evm/test/mocks/` (not `src/mocks/`) and ensure peripheral `foundry.toml` does NOT export test-dir artifacts to consumer ABI registry.
- **B3 critiques B2**: `Configure.s.sol` was alm's clean separation of "deploy" vs "wire". If we collapse it into Deploy.base.sol helpers, the deploy script becomes monolithic. Keep `Configure.base.sol` as a separate peripheral abstract for post-deploy AC role grants + PP feed registration; consumers still get 2-step flow.

### Round 2 -convergence

- **B1 + B2 agree**: Common.t.sol → `peripheral/evm/test/Common.sol` minimal (AC + PP + Treasury + roles). Per-repo extends. Mocks at `peripheral/evm/test/mocks/`. Dex `src/mocks/*` purged from `src/`.
- **B2 + B3 agree**: Two peripheral scripts: `Deploy.base.sol` (deploy bytecode) AND `Configure.base.sol` (wire roles + register feeds). Consumers extend both, invoke in sequence. Chain config = hybrid (env for secrets, JSON for salts/tokens lists).
- **B1 + B3 agree**: CI template lives in peripheral as a `.yml` snippet; consumers `cp` it (Foundry has no `extends` for workflows either). `.tool-versions` pins foundry binary; CI greps `forge --version` against expected. `<File>Test` naming convention enforced via `forge --match-contract` regex `.*Test$`.
- **All 3 agree**: Keeper is greenfield → `back/services/keeper/` (Rust + alloy, pinned). Skip "migration" framing entirely; it's a new service.

## 6. Consensus -CI

**Decision (3/3): per-repo `.github/workflows/test.yml` from one template; foundry pinned via `.tool-versions`; same matrix.**

Template (`peripheral/.github/workflow-templates/foundry-evm.yml`):

```yaml
name: CI
on: [push, pull_request, workflow_dispatch]
env: { FOUNDRY_PROFILE: ci }
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { submodules: recursive }
      - uses: actions-rust-lang/setup-rust-toolchain@v1   # for tool-versions parser
      - name: Read foundry version
        id: fv
        run: echo "version=$(grep ^foundry .tool-versions | cut -d' ' -f2)" >> $GITHUB_OUTPUT
      - uses: foundry-rs/foundry-toolchain@v1
        with: { version: ${{ steps.fv.outputs.version }} }
      - run: forge --version
      - run: forge fmt --check
      - run: forge build --sizes
      - run: forge test -vvv
      - name: foundry.toml drift check
        run: diff <(grep -E '^(solc|evm_version|via_ir)' foundry.toml | sort) \
                 <(grep -E '^(solc|evm_version|via_ir)' lib/peripheral/foundry.base.toml | sort)
```

`.tool-versions` (root of each consumer repo):

```
foundry stable-2026.04
```

(Exact tag pinned at peripheral bootstrap; consumers inherit.)

Naming: `forge --match-contract '.*Test$'` -every test contract MUST end `Test`. Doc 05 update to enforce. Existing alm/dex contracts already mostly comply; lint via grep gate.

`FOUNDRY_PROFILE=ci` requires `[profile.ci]` block in each `foundry.toml` -peripheral template includes it.

## 7. Consensus -keeper migration

**Decision (3/3): NOT a migration. Greenfield at `back/services/keeper/`.**

Justification:

- On-disk: NO `dex/keeper` exists. User mandate referenced it, but file system is empty. Reframe.
- Phase 30 pattern: `back/services/{agents, collector, swap}/` -keeper joins as a peer service. Rust crate, own `Cargo.toml`, own Dockerfile (`Dockerfile.keeper`), shared workspace.
- Consumer of BOTH alm and dex on-chain surfaces; lives one layer up. Reading deployment registries from `<alm|dex>/evm/deployments/<chain>.json`.
- Rebalance/tend/harvest entry points already in alm `Vault.sol`, `Dex.sol` adapters (per `grep` above). Dex pool `Admin*` modules expose timelock-gated maintenance; keeper interfaces with those too.

Stack: Rust + alloy 0.x (pinned, ! ethers-rs which is unmaintained) + nxr `prime-nxr` ws consumer for px-driven rebalance triggers. CMA-ES + Parkinson σ-aware tend cadence (per BTR project memory).

⚠ Conflict flagged: **`dex/keeper` placement** rejected (2/3 -B1 + B3 vs B2-mild). B2 noted release-cadence coupling concern at `back/`; addressed by per-service Dockerfile. If user intent was `dex/keeper` based on prior plan, override here (file system + Phase 30 pattern dominate).
