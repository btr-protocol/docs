## 3. External network tiered roadmap

The pragmatist position: **we don't list BTR DEX on external aggregators in v1**. We publish the package and let inbound interest pull the listings. Tiered effort estimate for *our* outbound work when each aggregator says yes:

| Tier | Aggregator | Required artifacts | Our effort | When |
|---|---|---|---|---|
| **T1 (publish-only)** | 1inch Fusion+, 0x, KyberSwap | Verified contracts + ABI + 1-page math spec + `tokens.json` whitelist | **2â€“3 dev-days total** (one-time package) | v1 |
| T2 (subgraph) | Matcha, CoW, OpenOcean | Subgraph schema + indexer + uptime | 1â€“2 weeks | v1.1 |
| T3 (SDK + business deal) | LiFi, Squid (cross-chain) | SDK contribution, audit handoff, partnership BD, liquidity guarantees | 4â€“8 weeks + legal | v1.2+ |

**Tier 1 pick for v1:** publish a single `listing-package/` folder in `dex/` containing:
1. Deployed addresses CSV.
2. ABI files (already produced by foundry).
3. `INTEGRATION.md` (math, fee model, hooks, gas profile).
4. Reference TS client snippet (reuse `swap/packages/core/src/Btr/`).

That is the **complete** v1 outbound surface. No per-aggregator PRs. We sit and wait for inbound.

**Friction signals from public aggregator docs (qualitative, based on prior integrations in `swap/packages/core/`):**
- 1inch: lowest friction; published "Add a DEX" form; mostly automated once contracts are verified + TVL threshold met.
- 0x: requires a PR to their orderbook router; medium friction; reviewer-gated.
- KyberSwap: requires a meta-aggregator integration PR or DMM-style adapter; higher friction but they actively solicit new venues.

Recommendation: open the conversation with 1inch first (cheapest win), use that listing as social proof for the other two.

## 4. Frontend: **rename** `earn/` â†’ `vaults/`, **add** sibling `pools/`

### 4.A Routing
- `earn/+page.svelte` â†’ `vaults/+page.svelte`. Move, no rewrite.
- `earn/[address]/+page.svelte` â†’ `vaults/[address]/+page.svelte`.
- Add 301-equivalent: a thin `earn/+page.svelte` that `goto('/vaults' + rest)` for bookmark stability. Delete in v1.1.
- New `pools/+page.svelte` from scratch (~250 LOC).

### 4.B `/pools` UX -one unified list, **not** two tabs

Pragmatist+UX joint position: **don't split into "BTR DEX pools" vs "3rd-party pools" tabs**.

Reasons:
1. A casual user does not know or care who runs the AMM behind a pair; they want APR + TVL + risk.
2. Two tabs imply two product categories. They're not -they're one category ("pools you can route through / track") with a venue attribute.
3. A unified list with a `Venue` column + a chip filter ("All / BTR DEX / External") gives power users the split-view in 1 click, costs new users zero cognitive load.
4. Empty-state for an external network is trivially handled: row absent. No empty tab.

### 4.C Columns (v1 minimum)
- Pair (token icons + symbol)
- Venue (badge)
- TVL (USD)
- Volume 24h
- Fee tier
- APR (fee-based, no incentives in v1)

Skip: pool depth curves, IL warnings, advanced sort. Add in v1.1 if users ask.

### 4.D Empty states
- "/pools -no pools match your filter." Single sentence, link to docs.
- For a network with no indexed pools: do nothing (silently absent). Don't display "coming soon" placeholders; they age badly.
