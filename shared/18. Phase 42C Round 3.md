### A4 -Numerical & Precision

**TWAP overflow analysis (`spot * (ORACLE_PBPS + off)`):**
- `spot = M.b64To1e18(acc.lastPriceB64)`. `b64To1e18` decodes via `decodeB64(_, 18)` (LibMaths.sol:71-73). Bounds: `mantissa ≤ 2^52 ≈ 4.5e15`, `MAX_SAFE_POSITIVE_SHIFT = 61` → `spot ≤ 4.5e15 * 10^61 ≈ 4.5e76`. Realistic prices encoded for token decimals ≤ 30 → `spot ≤ ~1e30`.
- `ORACLE_PBPS + off`: `off ∈ [int32.min, int32.max] = [-2.15e9, +2.15e9]`. `mult ≤ 2.15e9 + 1e7 ≈ 2.16e9`. Negative-clamp guard (`mult ≤ 0 → 1 wei`) prevents underflow.
- Product upper bound: theoretical `4.5e76 * 2.16e9 ≈ 9.7e85`, but `uint256.max ≈ 1.16e77`. **Theoretical overflow possible at extreme b64 inputs.** Realistic `1e30 * 2.16e9 = 2.16e39 << 2^256` → safe in practice.
- Bounded in real usage: `encodeOffset1e18` (LibOracle.sol:32-39) clamps off to int32 range; `_rollWindow` only encodes against TWAPs derived from same-class prices → off magnitude bounded by `(twap/spot - 1) * ORACLE_PBPS`, typically ∈ [-1e7, +1e7].
- **Verdict:** INFO (theoretical overflow at b64 magnitudes never reached by encodeB64 for sensible token decimals).

**Negative offset clamp at off = -ORACLE_PBPS exactly:** `mult = 0`, branch returns `M.encodeB64(1, 18) = 1 wei`. `LibOracle._applyOffset` returns `1` (raw 1e18) -same numeric meaning (1 wei). Test `test_R2_A1_1_fastTWAP_degenerateOffset_clampsToOneWei` (test/Phase42CR1Test.t.sol:342-352) asserts behavior. ✓

#### R3-A4-1 (INFO) -`getFastTWAP` theoretical multiplication overflow at b64 extremes
- **File:line:** `src/modules/InternalOracle.sol:75`.
- **Repro:** at extreme `acc.lastPriceB64` (mantissa near 2^52, decimal-shift near +61) and `|fastOffset|` near int32 max, `spot * uint256(mult)` could overflow uint256. Not reachable from any in-tree price-feed path (encoded prices are O(1e18-1e30) at most), but no explicit assertion.
- **Severity:** INFO. Encoding policy bounds spot well below the theoretical b64 ceiling.
- **Fix sketch (optional):** add bound `if (spot > type(uint128).max) revert Overflow()` or use `Math.mulDiv(spot, uint256(mult), ORACLE_PBPS)` for full-range safety. Mirrors `_applyOffset` shape; consider applying same hardening there.

(R1 carry-overs A4-2 LOW `priceOut→1` clamp LibPricing.sol:390, A4-3 LOW `spreadBps/2` floor Pool.sol:437, A4-4 INFO `_skewToDepth` unchecked, A4-5 INFO `MAX_DEV_THRESHOLD` event-only -all unchanged, non-blocking.)
