### A3 -Economic / MEV / ordering

- A3-1 (CONFIRMED CLEAN): R6 reserves debit at Pool.sol:319 occurs inside `_processSwap`, before control returns to `swap()`; `_push` to user (Pool.sol:269) happens AFTER that debit. Atomic within the same external call → no MEV-observable intermediate state.
- A3-2 (CONFIRMED CLEAN): No reserves underflow. `_exec` pre-checks `aOut.reserves ≥ q.amountOut + minLiquidity` (Pool.sol:455-456). After `_exec` (-out0), `_reconcile` (+Δ when delta>0), R6 debit (-pf): net `-out0 + lf`. Since `lf ≤ Δ` and pre-check left `R ≥ out0 + minLiquidity`, post-state ≥ `minLiquidity` (still safe; the post-exec floor check at Pool.sol:262-264 enforces invariantly).
- A3-3 (CONFIRMED CLEAN): `_push` debits the pool's actual ERC20 balance, not reserves. Reserves drift cannot trigger an unexpected `_push` revert.

### A4 -Numerical / precision

- **A4-1 (LOW, not a regression -pre-existing bound)** -Pool.sol:319: `uint128(protoDelta)` cast. `protoDelta = (Δ * protoShare)/100` where `Δ` is a hook-supplied `int256` cast to `uint256`. Although `Δ` could in theory exceed `uint128.max`, in practice `Δ ≤ q.amountOut` is the only operationally meaningful range (and the test fuzz pins this). When `Δ > q.amountOut`, `_applyHookFee` clamps `out → 0` but `q.protoFee` still increases by full `pf` -and `protoDelta = pf` may exceed `uint128.max`, silently truncating on the cast at :319. Recommendation: replace with `_toUint128(protoDelta)` (Base.sol:130) for explicit revert. Severity LOW because hooks are admin-set trusted code, and the same magnitude concern exists in `_reconcile`'s diff cast (which DOES revert on overflow). This is not a R5→R6 regression -it is inherited behavior.
- A4-2 (CONFIRMED CLEAN): `protoDelta = q.protoFee - protoBefore` cannot underflow -`_applyHookFee` only adds to `q.protoFee` (LibPricing.splitFee is non-negative).
