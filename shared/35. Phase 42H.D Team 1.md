## 4. Top 5 gaps preventing 9.5+

### G1 -Pool.sol fat singleton (1320 LOC) [criteria 1,4,5,6]
File `dex/evm/src/Pool.sol` is a single 1320-LOC contract mixing oracle TWAP rolling (368-558), liquidity ops (564-738), exchange (759-927), admin gateways (1069-1186), staking adapter, flash adapter, and helpers. Deploy bytecode pressure + cognitive surface ↑ vs Morpho-Blue minimalism (~600 LOC).
**Fix:** extract `_pushFeedInternal/_updateFeeds/_rollWindow/_updateVolEMA` (lines 449-558, ~110 LOC) into a `PoolOracle` library taking `IPool.PoolStorage storage $` -same delegatecall-free model already used by `Pricing` lib. Target: Pool.sol < 1000 LOC.

### G2 -Reentrancy guard slot duplicated 4× verbatim [criteria 4,5,6]
`Pool.sol:70-71`, `Staking.sol:68-69`, `Distributor.sol:34-35`, `Flash.sol:24-25` all inline the SAME constant `0xe22c27e8d25bc3725093027126bd674994df6625365bae10cf4b95c8b45f98b6` plus 8-line yul modifier copy. Solady `ReentrancyGuardTransient` is already used in `alm/Vault.sol:21` -the dex side does NOT consume it.
**Fix:** delete the inline yul modifiers in all 4 dex contracts; `import {ReentrancyGuardTransient} from "solady/utils/ReentrancyGuardTransient.sol";`. Saves ~32 LOC × 4 files = ~130 LOC of duplication; aligns dex with alm.

### G3 -Cross-repo timelock bifurcation [criterion 6]
- `shared/Timelock.sol:19` packs `(eta48, grace48)` into uint96 (used by `dex/Admin/Staking/Distributor/Treasury/Bridge/Router`).
- `shared/AccessControl.sol:22` uses `struct Queued { address addr; uint64 eta; }` (no grace).
- `alm/Swapper.sol:31` uses `struct Pending { address addr; uint64 eta; }` (no grace).
- `dex/PoolFactory.sol:21-22` uses raw `uint256 upgradeTimelock` storage var (no shared lib).
Four parallel timelock encodings across the same protocol.
**Fix:** route AC + Swapper + PoolFactory rotation through `shared/Timelock.{pack,validate}` OR formally document why `Queued{addr,eta}` ≠ `Timelock` packed (multisig ETA-only model). Currently inconsistent.

### G4 -Treasury still UUPS, others non-upgradeable [criteria 1,2,6]
`dex/Treasury.sol:21` extends `UUPSUpgradeable` (with full `requestUpgrade`/`executeUpgrade`/`cancelUpgrade` plumbing 319-354) and `Router.sol:18` likewise (330-356). But `Pool` is now EIP-1167 clone (non-upgradeable), `Admin/Staking/Distributor/Flash` are non-upgradeable singletons, `alm/Vault` clones from `Factory`. Mixed upgrade story confuses integrators + auditors. Treasury holds gov token mint authority -UUPS = single-key upgrade vector despite 7d timelock.
**Fix:** Either (a) make Treasury also non-upgradeable + replaceable via gov-token redeploy migration (Morpho-style immutable) OR (b) extend the same UUPS pattern to PoolFactory's `referencePool` (already has 7d timelock) for consistency. Document the rationale.

### G5 -Bytecode + storage density on Pool clones [criterion 4]
- `Pool.sol:37` `IPool.PoolStorage internal $` -single struct, but no explicit storage-layout slot pinning. Per-clone storage starts at slot 0 (fine for clones, but makes future `referencePool` upgrade via timelock risky if struct order changes silently -no `assembly { $.slot := ... }` ERC-7201 anchor).
- `Pool.sol:55-63` constructor takes 4 immutables (`AC, admin, staking, flash`) -these become baked into impl bytecode, NOT clones, ✓ correct. But `Pool.sol:44-53` re-declares all 4 as `address public immutable` -getters cost ~200 bytes bytecode × 4 in impl.
- `Spline.sol:11-39` `eval()` allocates `Point[] memory` on every interpolation. Hot path during `Pricing._traverseSplineByVolume` calls `Spline.eval` + `Spline.area` separately, double-building the same `points` array (cf. `Pricing.sol:163,176`).
**Fix:** Pin `PoolStorage` to ERC-7201 namespaced slot OR add a structural-hash assertion test guarding upgrade safety. Cache `points` array in `Pricing._traverseSplineByVolume` to avoid double-build (saves ~5–8k gas per swap).
