# ADR-002 — Timelock Encodings

**Status:** Accepted (Phase 42H · G6 — decision record only, no source change required)
**Date:** 2026-05-10
**Supersedes:** none
**Related:** ADR-001 (Upgrade Strategy), Phase 42H.D Team 5 (G3 audit note)

---

## 1. Context

Four distinct timelock encodings exist across the BTR Solidity codebase as of Phase 42H. The Phase 42H · G6 audit flagged the divergence as a "soft residual": it is not a correctness defect, but a stylistic / DRY concern. This ADR records the rationale for keeping each variant rather than forcing unification.

The four encodings:

| # | Site | Type / Shape | Grace? | Cancel semantics |
|---|---|---|---|---|
| 1 | `shared/src/Timelock.sol` | packed `uint96 = (eta << 48) | grace` | Yes (auto-expiry after `eta + grace`) | Optional explicit cancel + auto-expiry |
| 2 | `shared/src/access/AccessControl.sol` | `struct Queued { address addr; uint64 eta; }` | No | Explicit cancel only (no auto-expiry) |
| 3 | `alm/evm/src/adapters/Swapper.sol` | `struct Pending { address addr; uint64 eta; }` | No | Explicit cancel only (no auto-expiry) |
| 4 | `dex/evm/src/PoolFactory.sol` | raw `uint256 upgradeTimelock` (eta) | No | Explicit cancel only (no auto-expiry); cancel paths `cancelReferenceUpgrade` |

Consumers of (1) include `Bridge`, `Admin`, `Staking`, `Treasury`, `Router`, and historical `PoolProxy` / `LibUpgradeQueue` (the latter two were removed in Phase 42H).

## 2. Decision

**Keep all four encodings.** No unification effort scheduled. Document the boundaries in this ADR and in the `Timelock.sol` library header.

## 3. Rationale per encoding

### 3.1. `shared/Timelock.sol` packed `uint96` — generic grace-bearing locks

- Many short-lived per-op timelocks per pool/per-token (`Admin.requestAddAsset`, `Admin.requestUpdateRisk`, etc.). Each lives keyed in a `mapping(bytes32 key => uint96 packed)`.
- Pack (eta | grace) into a **single SSTORE / SLOAD** per op — material gas saving across thousands of governance ops over a pool's lifetime.
- Auto-expiry after grace is the desired UX: forgotten op queues should clear themselves so the queue map doesn't grow without bound.
- Used by every place where (a) the op is small/repetitive, (b) auto-expiry is wanted, and (c) the target/payload lives elsewhere (in `pendingData[key]`).

### 3.2. `shared/AccessControl.sol::Queued{address addr; uint64 eta}` — single-target gov rotation

- Three slots only: `pendingFactory`, `pendingTreasury`, `pendingSwapper`. One in-flight rotation per role.
- Pairing **target address + eta** in one struct/SLOAD removes the second storage access needed under encoding (1) (where the target would live in a separate `pendingData` slot).
- No grace because role rotations are gov-controlled, high-stakes, and explicitly cancelled if no longer desired. **Auto-expiry would be a footgun** — silently voiding a rotation that gov assumed was still queued would create an unexpected window of stale role.
- Worth the encoding divergence because the call-site is tiny (a handful of getter / setter pairs) and the read-path is hot during owner takeover.

### 3.3. `alm/Swapper.sol::Pending{address addr; uint64 eta}` — restricted-router update

- Identical shape to (3.2) for identical reasons. Two slots only: `pendingLifi`, `pendingSettler`.
- Restricted-router swaps must NOT silently expire — a stale "queue still pending" assumption would let a settled swap go through an unintended router.
- Storage layout is also load-bearing for alm `Vault` clones (downstream consumers index this field directly).

### 3.4. `dex/PoolFactory.sol::upgradeTimelock (uint256)` — single in-flight impl swap

- One in-flight pending reference-impl swap (`pendingReferencePool`) — no per-op keying.
- Packing (eta | grace) buys nothing: `pendingReferencePool` already lives in its own slot.
- Grace = ∞ semantics: governance MUST explicitly cancel via `cancelReferenceUpgrade`; the swap must never silently void. The packed lib's `uint48 grace` cannot encode this — encoding (4) sidesteps the issue.
- `uint256` is ABI-stable for external integrations (front-ends, keepers, indexers) that already read this slot.

## 4. Consequences

- **+** Each call-site uses the encoding that fits its hot-path / safety semantics.
- **+** No silent-expiry footguns on critical-tier rotations (3.2 / 3.3 / 3.4).
- **+** Auto-expiry available for noisy per-pool / per-asset ops (3.1).
- **−** Four encodings means new contributors must consult this ADR before adding a fifth.
- **−** Tooling that introspects timelock state (block explorers, keepers) must handle multiple shapes.

## 5. Alternatives considered

| Option | Reason rejected |
|---|---|
| Unify on packed `uint96` everywhere | Would force grace semantics onto rotations that explicitly disallow auto-expiry; would also add an SLOAD to (3.2)/(3.3) hot paths. |
| Unify on `struct {addr, eta, grace?}` everywhere | Doubles storage cost for the high-frequency (3.1) call-sites; defeats the gas optimisation that motivated the packed layout. |
| Add a `grace=∞` sentinel to `Timelock.sol` and migrate (3.4) | Possible, but the sentinel adds a branch in every `validate()` call and breaks the ABI of `upgradeTimelock` that integrators already read. |
| Forbid the use of (3.4)-style raw uint256 in new code | Acceptable as future guidance; this ADR adds a rule: new in-flight single-slot timelocks SHOULD use encoding (3.2)/(3.3) unless ABI stability or explicit grace=∞ semantics require encoding (3.4). |

## 6. Future direction

If a fifth call-site emerges, evaluate it against the boundary conditions:
- Many small per-op locks needing auto-expiry → encoding (3.1).
- Single critical rotation with explicit cancel → encoding (3.2) / (3.3).
- Single critical rotation with ABI stability for external readers → encoding (3.4).

If none of those fit, return to this ADR and add a new encoding row with explicit rationale.
