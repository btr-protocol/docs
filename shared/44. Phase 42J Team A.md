## 2. Section B -BTR Swap aggregator integration spec

### 2.1 Current swap SDK

`swap/packages/core/src/` (15 aggregator dirs incl. LiFi, OneInch, ZeroX, KyberSwap, Odos, ParaSwap, Socket, Squid, etc).
Base class: `BaseAggregator` (`swap/packages/core/src/abstract.ts:18`) with chain-keyed `routerByChainId`, `aliasByChainId`, `approvalAddressByChainId`.
`AggId` enum at `swap/packages/core/src/types.ts:6` -add `BTR_DEX = "BTR_DEX"`.

### 2.2 BTR DEX router surface

`dex/evm/src/Router.sol`:
- `executeSwap(Route route, uint256 amountIn, uint256 minOut, address recipient)` → `uint256 amountOut` (Router.sol:261).
- `getSwapQuote(address pool, address tokenIn, address tokenOut, uint256 amountIn)` → `IExchange.SwapQuote` (Router.sol:313). **NOTE: non-view** (mutates oracle? confirm with team) -for off-chain quotes, use `eth_call`.
- Route discovery: `getBestBatchPool` + `_getBestDirectQuote` + `PoolFactory.getCommonPools(tokenA, tokenB)` (PoolFactory.sol:138).
- Single-hop and multi-hop both expressed as `Route { RouteStep[] steps }` where `RouteStep { pool, tokenIn, tokenOut, minOut }`.

### 2.3 Adapter spec -`BtrDex` aggregator

```ts
// swap/packages/core/src/BtrDex/index.ts (NEW)
import { BaseAggregator } from "../abstract";
import { AggId, IBtrSwapParams, ITransactionRequestWithEstimate } from "../types";

export class BtrDex extends BaseAggregator {
  constructor() { super(AggId.BTR_DEX); }

  // No external API. Pure on-chain. baseApiUrl unused.

  async getQuote(p: IBtrSwapParams): Promise<ITransactionRequestWithEstimate> {
    // 1. RPC eth_call PoolFactory.getCommonPools(tokenIn, tokenOut)
    // 2. For each candidate pool: eth_call Pool.getSwapQuote(tokenIn, tokenOut, amountIn)
    //    (or Router.getSwapQuote via eth_call w/ from = zero addr)
    // 3. Pick max(amountOut), build Route { steps: [{ pool, tokenIn, tokenOut, minOut }] }
    // 4. For multi-hop: greedy bridge-token (base token) routing -phase 2.
    // 5. Encode calldata: Router.executeSwap.selector + abi.encode(route, amountIn, minOut, recipient)
    return { to: ROUTER_ADDR_BY_CHAIN[p.chainId], data, value, estimate: { amountOut } };
  }
}
```

Wiring:
- Register in `swap/packages/core/src/index.ts` next to LiFi.
- Populate `routerByChainId` from deployment registry.
- `approvalAddressByChainId === routerByChainId` (Router pulls ERC20 in `_transferIn`).

### 2.4 Frontend swap aggregation flow

`SwapPage` → `SwapStore` → fan-out to N aggregators in parallel (already the pattern) → BTR-DEX one of them → user picks best quote → execute via the picked aggregator's `to`/`data`/`value`.

### 2.5 Placeholder phasing

Phase 1: ship the `BtrDex` class with single-hop direct-quote only. No fee modelling (Pool computes its own swap fee internally).
Phase 2: multi-hop routing via shared base token (`PoolFactory.poolBaseTokens`).
Phase 3: cross-pool arb-aware routing (deferred -not needed for v1).

## 3. Section C -External liquidity network integration matrix

### 3.1 Public-docs reality

Public docs for 1inch / 0x / KyberSwap **do not document a self-listing path**. All three integrate DEXs via internal connector modules maintained by their own engineering teams. Listing = collaboration with their team, typically opened by reaching out + delivering a reference integration.

### 3.2 Per-network requirements (best-known)

| Requirement | 1inch (Pathfinder/Fusion) | 0x (Settler/AllowanceHolder) | KyberSwap (MetaAggregator) |
|---|---|---|---|
| On-chain registry (pool list) | ✓ Pathfinder reads via factory event-scan | ✓ Settler routes through specific pool interface | ✓ Reads on-chain via factory |
| Subgraph (TheGraph) | Optional but expected for analytics | Strongly preferred | **Required** for pool discovery |
| Pool interface stability | swap-like ABI w/ deterministic quote | swap function + getReserves-equivalent | swap + quote view |
| Audit | Required (reputable firm) | Required | Required |
| TVL minimum | Informal ~$1-5M | Informal | Informal |
| Gas profile | Tracked per-pool; high gas → deranked | Tracked | Tracked |
| Connector code | 1inch maintains a Solidity connector in their `AggregationRouterV6` library | 0x writes a Settler component in `0xProject/protocol` | Kyber writes an off-chain encoder in their meta-aggregator |
| Smart-order-routing manifest | Internal -they author it | Internal | Internal |

### 3.3 Gaps in current BTR DEX surface

1. **`getSwapQuote` is non-view** in Pool.sol:319 region (Router.sol:313 not annotated view). Aggregators call quote via `eth_call`; non-view still works on `eth_call` (state is reverted), but it's a red flag -confirm idempotent.
2. **No subgraph yet** for BTR DEX. Required for KyberSwap; recommended for others.
3. **`Pool.swap` per-hop interface** (`IExchange.swap(tokenIn, tokenOut, amountIn, minOut, recipient)`) matches the V2-style ABI most aggregators expect -✓.
4. **Token registry**: `PoolFactory.tokenToPools(token)` and `getCommonPools(a,b)` are perfect for aggregator pool-discovery -✓.
5. **Fee discovery**: aggregator routing engines need a deterministic fee curve; BTR DEX uses `PoolDecay` + dynamic fees per `feeParams`. Must be exposed via a `feeParams()` view (already present: `getFeeParams()` -Pool.sol:702). Aggregator connector must replicate the fee math off-chain to quote, OR `eth_call` `getSwapQuote` per route candidate (more RPC, less smart-routing).
6. **No `flash` / `flashSwap` interface** standardized with aggregator expectations (1inch v6 sometimes uses callback-pattern routes). Out of scope for v1.

### 3.4 Phasing per network

| Network | Effort | Path |
|---|---|---|
| **0x** | Heaviest (Settler is Solidity-level integration); estimate 6-10 weeks engineering on their side. | Open issue in `0xProject/protocol`, supply reference Settler component. |
| **1inch** | Medium-heavy. Pathfinder ingests pools via factory events; needs their Solidity connector. Estimate 4-8 weeks. | Reach via `business.1inch.com` portal. |
| **KyberSwap** | Lightest because their aggregator is more open. Needs subgraph + connector. Estimate 3-6 weeks. | Reach via KyberSwap DEX-integration team; deliver subgraph. |

**Quick win:** **None** -none are self-listing. Recommend starting with KyberSwap (most open) + delivering a subgraph as the first deliverable (also unlocks BTR-internal analytics).

**Pre-requirement (universal):** Public audit + 90-day mainnet TVL track record. Until then, focus on the BTR-Swap-internal `BtrDex` aggregator (§ B) -that gives users access to BTR DEX liquidity even before external listings.

## 4. Section D -ALM ↔ BTR DEX integration

### 4.1 Interface compatibility check

| Concept | `alm/Dex.sol` (CL adapter) | BTR DEX `Pool.sol` | Compatible? |
|---|---|---|---|
| Position model | Tick range `(tickLower, tickUpper)` (Dex.sol:64-66) -concentrated-liquidity NFT-like | Single-sided per-token LP balance `lpBalances[user][token]` (Pool.sol:338) | ✗ Fundamentally different |
| Deposit shape | `pull(uint256 amount)` taking `token0` only, mints internal "shares" of `principalOf` (Dex.sol:259) | `deposit(address token, uint256 amount)` returns `DepositResult{lpAmount, actualDeposit}` (Pool.sol:319) | ✗ |
| Withdraw shape | `withdraw(address to, uint256 amount)` of `token0` only (Dex.sol:283) -burns internal shares | `withdraw(token, lpAmount, minOut)` returns `WithdrawResult` (Pool.sol:367); also `withdrawTo(from, to, lp, minOut)` cross-asset (Pool.sol:375) | ✗ |
| Valuation | `assetValue(token)` worst-of(pool, oracle) for token0/1 (Dex.sol:357) | Per-asset `liquidityIndex` (Pool.sol:334) | ✗ |
| Rebalance | `rebalance(tl, tu, routes)` (Dex.sol:305) -burn old, swap, re-mint at new range | Not needed -Pool auto-rebalances via PoolDecay | n/a |
| Frozen check | `frozen()` view (Dex.sol:396) | `riskConfigs[tk].flags & FROZEN_BIT` (Pool.sol:329) | Different semantics |
| Holder accounting | `principalOf[holder]` internal shares (Dex.sol:93) | Native ERC-LP via `lpBalances[u][tk]` (Pool.sol:338) | ✗ |

### 4.2 Verdict -`alm/Dex.sol` does **NOT** support BTR DEX pools as-is

The CL adapter is hard-wired to V3/V4/Algebra/Ramses **tick-range mint/burn callback** semantics (Dex.sol:481-521: `_burnPosition` / `_collectFees` / `_mintPosition` / `fallback` mint-callback). BTR DEX pools have no such surface. Forcing them through Dex.sol via a `kind` discriminator would:
- Break the `(token0, token1)` invariant -BTR pools are multi-asset.
- Break `assetValue` (no `sqrtPrice` analogue).
- Break the JIT-defence model (`frozen()` reads pool slot0).
- Pollute hot-path branches with dexType-switch.

### 4.3 Recommended path -**new `BtrPoolAdapter.sol`**

```solidity
// alm/evm/src/adapters/BtrPoolAdapter.sol (NEW)
contract BtrPoolAdapter is Initializable, ReentrancyGuardTransient {
    // Implements IDex (so Vault treats it uniformly).
    AccessControl public ac;
    IPool         public btrPool;
    address       public asset_; // single-token base asset for this adapter
    mapping(address => uint128) internal principalOf;
    uint128 internal totalPrincipal;
    bool    internal killed;

    // pull(amount) → btrPool.deposit(asset_, amount) → store DepositResult.lpAmount; share-mint pro-rata
    // withdraw(to, amount) → compute lp needed; btrPool.withdraw(asset_, lp, minOut); transfer
    // assetValue(token) → query Pool.getAsset(token).liabilities * lpBalance / totalSupply
    // vaultAssets(holder) → assetValue(asset_) * principalOf[holder] / totalPrincipal
    // healthSnapshot → ( (Pool.getRiskFlags(asset_) & FROZEN_BIT) != 0 , killed )
}
```

Key adaptations:
- LP balance via `Pool.getLPBalance(address(this), asset_)`.
- NAV via `Pool.getAsset(token).liquidityIndex` * lp / WAD.
- Single-token deposit is **native** to BTR Pool → no swap routes needed in deposit path → can drop the entire Swapper-route plumbing.
- Multi-asset variant: deposit one token, withdraw another via `withdrawTo` (cross-asset) -gives the Vault real cross-asset rebalance without external swaps. Defer to v2.

### 4.4 IDex interface stability

The interface (`alm/evm/src/IDex.sol:9`) is symmetric -`asset()`, `assetValue(token)`, `vaultAssets(holder)`, `pull`, `withdraw`, `healthSnapshot`. **No change needed.** Both `Dex` (CL) and `BtrPoolAdapter` can satisfy it.

### 4.5 Factory adjustment

`alm/Factory.sol:47` `deployDex()` clones a single `dexImpl`. To support both kinds, either:
- (a) Add `deployBtrAdapter()` with a separate `btrAdapterImpl` (NEW immutable in Factory). **Recommended.**
- (b) Make Factory parameterized over an array of impls keyed by kind. Heavier -defer.

Trade-off (a vs b): (a) preserves Factory immutability & 14d-timelock rotation model (`CLAUDE.md` Factory pinning); (b) requires touching the `ac.queueFactory` model.

## 6. Section F -Risk register

| # | Risk | Severity | Mitigation |
|---|---|---|---|
| R1 | Two adapter impls (`Dex`, `BtrPoolAdapter`) share `IDex` but diverge in valuation semantics -vault treats them uniformly | **High** | Strict test matrix; fuzz vault accounting across both adapters; pessimistic NAV in both. |
| R2 | BTR DEX has no audit / TVL track record yet → external aggregator listings blocked | **High** | Sequence external listings AFTER audit; ship `BtrDex` in-house aggregator first. |
| R3 | `Pool.getSwapQuote` may be non-view (state-touch via PoolDecay.applyDecay) → aggregators must `eth_call` to side-effect-revert | Medium | Document; verify gas; consider exposing a pure quote helper. |
| R4 | Factory-pinning model means adding `btrAdapterImpl` mutates immutable surface → requires new Factory deployment + 14d AC.queueFactory timelock | Medium | Plan deployment cycle; coordinate with adapter rotation. |
| R5 | Frontend mock data still wired in `EarnPage` / `VaultDetailPage` | Low | Replace before ship; tracked. |
| R6 | No subgraph yet -Pools page falls back to event-scan; slow first paint | Low-Med | Cache via `back/services/collector`; subgraph is v2. |
| R7 | Aggregator `BtrDex` multi-hop routing missing → users get worse-than-optimal quotes for indirect pairs | Medium | Phase 1 single-hop is acceptable when LiFi etc. provide indirect routes via external pools; revisit after main pairs land. |
| R8 | Cross-asset withdraw (`Pool.withdrawTo`) opens fee-arbitrage; ALM adapter using it may bleed | Medium | Limit BtrPoolAdapter v1 to same-asset withdraw. |
| R9 | `CLAUDE.md` claims SvelteKit; codebase is Preact | Low | Update CLAUDE.md after this phase. |
| R10 | KyberSwap subgraph schema spec unknown; estimate may slip | Low | Spike subgraph against a public Uniswap-style schema first. |
| R11 | `Router.executeSwap` requires `f.isOfficialPool(step.pool)` (Router.sol:282) -only protocol-deployer pools route via Router → aggregator integration constrained to official pools | Medium | Acceptable for v1; documents the trust model. |

### Must-have v1 vs defer

**v1 must-have:** Sections A, B (single-hop), D (BtrPoolAdapter), E (Tab 1 wired live; Tab 2 via event scan).
**Defer to v1.1+:** Section C (external listings post-audit), Tab 2 polish via subgraph, multi-hop BtrDex routing, cross-asset BtrPoolAdapter, back/services/collector pool aggregation API.
