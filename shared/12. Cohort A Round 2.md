## 2. 3-expert debate -unresolved items

### Item A -dex AC deferral

**A1 Architect.** Diamond storage (`IPool.PoolStorage.owner`, `IPool.sol:129`) is per-pool by ERC-7201 design. Forcing peripheral AC consumption (option a) means swapping every `_s().owner` read in `Base.onlyOwner` (`Base.sol:36–38`) plus the 26+ `onlyOwner` sites across `Admin.sol:26`, `Distributor.sol:5`, `Treasury.sol:17`, `Bridge.sol:14`, `Router.sol:3`, `PoolProxyFactory.sol:4`, `Staking.sol:5`, `Rescue.sol:1` for `IAccessControl(ac).owner()`. Storage slot `owner` becomes orphan unless preserved as cache (storage-layout-safe). My vote: **(b)** -keep dex local, document the asymmetry. Diamond owner ≠ singleton owner: factory rotates per-pool deployer immutable, AC rotates protocol-wide multisig. Distinct semantics.

**A2 Security.** Round-1 mandate said "consumers MUST read `ac.owner()` live, no caching" -explicitly to enable atomic key rotation. Option (b) breaks that invariant: rotating the multisig requires a per-pool tx storm across N pools. Audit-blast-radius argument: dex pools currently trust their factory deployer to set `_s().owner` once, then it's frozen barring `setProtocolDeployer` (`PoolProxyFactory.sol:185`). That is *less* trust-coupled than a live AC read but *more* operationally brittle. Option (c) hybrid (`AccessControlDiamond` view) is over-engineering -same auth surface, more code. My vote: **(a)** with storage-layout discipline (append-only struct migration, deprecate `owner` field as `_legacyOwner` reserved slot).

**A3 Conventions.** Round-1 was aspirational on this point -(a)/(b)/(c) was deferred via "may migrate to consume this AC in a later sub-stage" comment in `peripheral/src/access/AccessControl.sol:14–16`. Code now documents the deferral inline; that *is* the singleton-mandate exception. Option (b) with a concrete migration trigger (e.g. "when storage migration tooling lands in Phase 43") is honest. Option (a) requires a redeploy or proxy-upgrade per pool -currently no dex pool ships UUPS at the proxy layer (`PoolProxy.sol` is fixed-storage Diamond), so migration cost is N-redeploys. Vote: **(b)** for now, **(a)** when the next breaking dex deploy ships.

**Convergence.** 2/3 → **(b) keep dex AC local**, with an explicit deferral marker (already in `AccessControl.sol:14–16`) and a concrete trigger: next breaking dex storage-layout change → consume peripheral AC. Governance score impact: -0.4 from a hypothetical 10.0 (single-source-of-truth violated for the dex surface).

### Item B -Treasury extraction

**A1.** alm uses treasury as an *address sink* (`Vault.sol:243,264,383,434,463` -`ac.treasury()` returns `address`). dex `Treasury.sol` (334 LOC) is a *contract* with vesting + distribution + UUPS upgrade flow. Different abstraction tier. Extracting dex Treasury to peripheral would force alm to either ignore most of its surface or adopt vesting (which alm has no need for). **Vote: stay dex-local.**

**A2.** Singleton-mandate scope was AC + PP, not Treasury. Treasury content is dex-specific (vesting schedules, distribution rules). alm consults `ac.treasury()` for *where to send fees*; the recipient identity is governance, the recipient *behavior* is dex. **Vote: dex-local.** Add NatSpec to AC: "treasury address may be a passive EOA/multisig OR an active contract like dex/Treasury.sol".

**A3.** No alm need anticipated. Premature extraction violates "extract only after second consumer materializes" (style guide §6 paraphrase). **Vote: dex-local.**

**Convergence.** 3/3 → **stay dex-local.** alm keeps `ac.treasury()` address read; dex Treasury.sol implements behavior at that address.
