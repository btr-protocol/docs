## 1. Diamond removal -file-by-file plan (Architect-led)

### 1.1 Architect verdict
User mandate is explicit: "no diamond." Two viable shapes:

| Shape | LoC | Storage | Cross-call | Verdict |
|---|---|---|---|---|
| (a) Single monolith Pool.sol | ~3.5k | one contract, plain layout | inlined | rejected -too big to audit, blast radius for any change |
| (b) Split contracts, refs by addr | ~3.5k spread across 5 files | per-contract | normal external call OR `is`-inheritance for hot path | **CHOSEN** |

**(b) chosen** because:
- Parity with ALM (Vault holds adapter refs, calls them by address).
- Each contract is independently auditable + upgradeable via UUPS (already in shared/AccessControl semantics).
- No ERC-7201 storage contortions; default storage layout.
- Trivial to test in isolation.

### 1.2 Mapping (current diamond → target contracts)

Current shape (`dex/evm/src/PoolProxy.sol:1` + `dex/evm/src/modules/*`):

```
PoolProxy (delegatecall router) ─┬─ Pool.sol            (core deposit/swap/withdraw)
                                 ├─ Admin.sol           (governance)
                                 ├─ Staking.sol         (gov + LP staking)
                                 ├─ Distributor.sol     (campaigns)
                                 ├─ Flash.sol           (flash loans)
                                 ├─ InternalOracle.sol  (TWAP/EMA)
                                 └─ Lend.sol            (empty stub -DELETE)
```

Target shape:

```
Pool.sol                         -deposit/withdraw/swap, holds PoolStorage
  ├─ refs: admin, oracle, staking, distributor, flash, treasury, ac
  └─ inherits: Base (reentrancy, AC modifier)
Admin.sol                        -owner/keeper config + timelock queue, holds AdminStorage
Staking.sol                      -gov + LP stake, holds StakingStorage
Distributor.sol                  -campaigns (TOKEN-only after SBT removal)
Flash.sol                        -ERC-3156, no own storage (reads pool)
InternalOracle.sol               -TWAP/EMA, holds OracleStorage
```

`PoolProxy.sol` and `PoolProxyFactory.sol` collapse into a single `PoolFactory.sol` (no proxy, just deployer + registry).

### 1.3 Storage migration story (Security-led)

**Current** (dex/evm/src/libraries/LibConstants.sol:7-15): five ERC-7201 namespaces collide-free in PoolProxy slot space:
- `CORE_STORAGE_LOC` -Pool
- `STAKING_STORAGE_LOC` -Staking
- `DISTRIBUTOR_STORAGE_LOC` -Distributor
- `ORACLE_STORAGE_LOC` -InternalOracle
- `FLOW_GUARD_STORAGE_LOC` -flow guard

**Target:** each contract owns its own default storage (slot 0+). ERC-7201 namespaces become **unnecessary**. Drop `_s()` indirection (`PoolProxy.sol:30-33`, all `modules/*` `_s()` accessors).

**Risk:** mainnet redeployment required. **Not an in-place upgrade.** Migration ⇒ pause + drain + redeploy + re-init. Document as breaking. (S verdict: acceptable since current code is unaudited / pre-launch per repo state.)

### 1.4 Cross-contract call shape

Current: `Pool.deposit` → `_s().` → `InternalOracle._updateFeeds` (delegatecall, same address).

Target: `Pool.deposit` → `oracle.updateFeed(...)` (external call to InternalOracle deployed separately). Three options for the hot path:

| Option | Gas | Auditability |
|---|---|---|
| External call | +2.5k gas/hit | clear boundary |
| `is InternalOracle` inheritance | 0 overhead | code dup if Admin also needs it |
| `library` for pure logic + state via interface | +0.5k | best of both |

**Architect rec:** `Pool is InternalOracle` (Pool inherits InternalOracle as an internal mixin since both need OracleStorage in same address). Admin reaches oracle via external call (`InternalOracle(oracleAddr).pushFeed`). This is exactly how `dex/evm/src/modules/Admin.sol:482` already calls `InternalOracle(address(this))` today -same shape, just resolves to a real address instead of self-delegate.

**Critical alignment:** `dex/evm/src/modules/Pool.sol:5` and `dex/evm/src/modules/Admin.sol:11` already import `InternalOracle` directly -code already assumes this shape. Refactor is mostly removing the `delegatecall` middleman.

### 3.2 Events -consolidation

**No `shared/Events.sol` exists today.** Events are inherently per-contract for ABI/indexer reasons. **DRY verdict: do NOT create shared/Events.sol.** Events look identical by name but Solidity ABI-resolves them per contract anyway; centralizing creates indexer brittleness with no gas/code benefit.

**Exceptions worth deduping** (can go on a shared interface OR shared library `EventsLib`):
- `OwnershipTransferred` -already on Solady's `Ownable` (alm + dex inherit). NO duplication; Solidity already shares.
- `ConfigSet(bytes32 key, uint256 val)` -`Vault.sol:83`, `Dex.sol:98` (alm). Same shape. **Move to shared interface** `IConfigurable { event ConfigSet(...); }` and have both inherit. Saves 1 line each.
- `Salvaged(token, to, amount)` -`Vault.sol:90` (alm), `IBridge.sol:35`, `ITreasury.sol:29` (dex). Same shape. **Pull onto shared interface `ISalvageable`**.
- `UpgradeAuthorized(id, newImpl, executableAt)` + `UpgradeCancelled(id)` -`IBridge.sol:33-34`, `ITreasury.sol:20-21`, `IRouter.sol:46`. **Pull onto shared `IUpgradeable`**.

**Pragmatist callout:** event consolidation is low-leverage (no security gain, mild maintainability gain). Skip if time-constrained.
