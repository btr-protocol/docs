# 03. Repo Structure — final layout + 3-option debate

## Recommendation: **Option A — `~/Work/btr/peripheral/`** (sibling Foundry repo)

Justification: aligns w/ Phase 30 split doctrine (per-domain repos: `alm`, `dex`, `sdk`, `swap`, `back`, `front`). Shared Solidity peripheral mirrors the `sdk` (TS) sibling. Versioned independently. Consumed via `lib/peripheral` git submodule + `remappings.txt`. Audit scope = bounded to `peripheral/src/`. ABI artifacts re-exported via `@btr-protocol/sdk/abis/peripheral`.

## Options considered

### Option A — `~/Work/btr/peripheral/` (sibling) — **PICKED**

Pros: matches Phase 30 split; independent CI; bounded audit scope; symmetric w/ `sdk/`. Cons: requires submodule discipline; versioning across alm + dex must be coordinated.

### Option B — `~/Work/btr/sdk/contracts/`

Reuses sdk repo. Cons: sdk is TS-first (`@btr-protocol/sdk` ABIs/types); mixing Solidity sources confuses consumers and CI. ABI emission flow (Foundry → `sdk/src/abis/`) becomes circular. Reject.

### Option C — git submodule `btr-protocol/peripheral` (separate GitHub org repo)

Pros: clearest separation; usable by external consumers (e.g. integration partners). Cons: premature externalization; ops overhead (release tags, CI cross-org). Defer until ≥3 internal consumers (dex + alm + future) are stable. Revisit Phase 50.

## Pragma + solc

Pin `^0.8.29` (alm anchor). dex currently `^0.8.33` — downgrade is a no-op (no 0.8.30+ features in use grep'd from dex). Foundry: `solc = "0.8.29"`, `via_ir = true`, `optimizer_runs = 200` (matches alm).

## Solady pinning

Single Solady submodule in `peripheral/lib/solady` pinned to alm's current commit. alm + dex remappings target peripheral's `lib/solady` to enforce one Solady version cluster-wide. Eliminates current alm vs dex Solady drift.

## ABI emission

Post-build, run `forge inspect <Contract> abi > out/<Contract>.json`. Sync to `~/Work/btr/sdk/src/abis/peripheral/` via existing sdk regen script. front consumes `@btr-protocol/sdk/abis/peripheral/AccessControl` etc.

## Deploy topology (per chain)

1. Deploy `AccessControl(owner, treasury)` — owner = governance multisig.
2. Deploy `PriceProvider(ac)`; bind sequencer feed if L2.
3. alm Factory: `Factory(ac, vaultImpl, dexImpl)` → `ac.bootstrapFactory(factory)`.
4. dex PoolProxyFactoryV1: deploy w/ `ac` immutable; modules registered via `setModuleTrust`.
5. Both consume same `(ac, priceProvider)` instances.

## Cross-cluster constraints (NXR k0s anchor)

Per CLAUDE.md routing rule: alm + dex deploy to NXR k0s-fronted chains only (btr.supply / btr.markets domains). Peripheral inherits same target. No DU AKS deploys — keep registry separation strict.
