## 1. Expert sections

### 1.1 Security (lead)

**Approval surface per page**
- **Swap**: ERC-20 approvals to BTR Swap router (Li.Fi today, BTR DEX `Router.sol` future) and/or Permit2. Per-tx Permit2 sig preferred over infinite approval. Two distinct allowance namespaces ⇒ confusion-risk if a single "approve" button rotates targets. UI **must** label per-target.
- **Vaults**: approval to `Vault.sol` (ERC-4626/7540 entry). Distinct allowance -must NOT collapse with Swap router approval.
- **Pools (BTR tab)**: approval to `Pool.sol` directly for `deposit()`. Pool addresses are per-asset; one approval per asset, not one per pool -verify in front via `lpBalances[user][token]` not per-pool tracking.
- **Pools (3rd-party tab)**: approval to external pool / position manager / zap adapter. **Threat**: if 3rd-party adapter is implemented as a single front contract that proxies to many externals, an exploit at adapter = aggregated allowance loss. **Recommend**: per-protocol allowance scoping; never approve adapter for blanket spend.

**Vault.salvage path** (`Vault.sol:540`) crosses into adapter routerData → cross-contract reentrancy lane. With BTR DEX as adapter target, salvage → DEX swap → DEX `_oracle` push → returns. `nonReentrant` on Vault holds, but DEX has its own re-entry transient guards (`PoolDecay.applyDecay` mutates state). Reentrancy still safe at compiler level **only** if BTR DEX adapter does **not** expose user-callable hooks to the salvage path.

**Compromised-owner posture**
- ALM `Dex.sol` has FREEZE_TIMELOCK on unkill + risk-param raise. ✓
- BTR `Pool.sol` admin domain (`Admin.sol`, `PoolAdmin.sol`) must mirror this discipline for adapter-relevant params (haircutSuppressor, riskConfigs[token].flags, liquidityIndex). Outstanding -not verified here.

### 1.2 Solidity Auditor

**Architectural fault (F1)**. Confirmed by reading both contracts:

- `alm/evm/src/adapters/Dex.sol`: enum `DexType { UniV3, UniV4, PancakeV3, PancakeInfinity, Algebra, Ramses }` -six external CL types, **BTR DEX is absent**. Manages `tickLower/tickUpper`, `_burnPosition/_mintPosition`, `_slot0Branched` over external pools. State model = paired token0/token1 CL.
- `dex/evm/src/Pool.sol`: **single-token** liquidity model. `deposit(address token, uint256 amount)` per-asset; `lpBalances[depositor][token]`; pricing via `Pricing.getAnchorPathQuote` (internal spline) + `PoolOracle` TWAP. Not Uniswap-shaped.

⇒ A new adapter `adapters/BtrDex.sol` is required. Cannot bolt onto `Dex.sol`. Verdict for the user mandate ("ALM vaults must deposit single-token into BTR DEX pools") = **NEEDS-NEW-ADAPTER** (detail in §2).

**Auth boundary (F2)**. `Pool.deposit` line 338: `$.lpBalances[msg.sender][tkn] += lpAmt`. msg.sender = the new BtrDex adapter. So:
- LP shares accrue to **adapter**, not Vault.
- Vault must read NAV from `adapter.assetValue(asset)` exclusively. Direct `Pool.lpBalances[Vault]` lookup will return 0 and silently mis-NAV.
- Withdraw path: only the adapter can call `Pool.withdraw` (because only it holds LP). Vault → adapter.withdraw → Pool.withdraw → push to adapter → adapter forwards to Vault. **Two extra hops vs current external CL adapter**; each hop must enforce its own slip and `nonReentrant`.

**Decay surface**. `PoolDecay.applyDecay($, tkn, asset)` runs on every deposit/withdraw -mutates `asset.reserves`/`liabilities`/`liquidityIndex`. Vault.totalAssets() will tick down between blocks without any flow -ERC-4626 invariant (`previewDeposit` ≅ stable in same block) holds only same-block. Off-block share-price preview ≠ executed price → small but persistent UI/UX drift. Acceptable; document.

**Haircut composition (F3)**. `_applyHaircut(withdrawValue, reserves, liabilities, haircutSuppressor)` (Pool.sol:495) applies a reserves-vs-liabilities haircut on withdraw. Vault then applies `exitPip` gross-up on its outer redeem. Worst case on stressed redeem:
```
user_net = (pool_amt × (1 − dexHaircut)) × (1 − exitPip) × (1 − adapterMaxSlip)
```
Three multiplicative bites. Need an invariant test: `dexHaircut + exitPip + adapterMaxSlip ≤ HARD_CAP` (suggest 10%). Otherwise stressed redeem realises >10% haircut without breaching any single guard.

**Initialize front-run**. `Router.initialize` line 43 -unguarded, deploy-script atomicity assumed. Mirrors Bridge disposition (acknowledged in comment). Acceptable if deploy script is single-tx; CI must enforce. Same posture should be confirmed for the new BtrDex adapter's `init`.

### 1.3 MEV / Sandwich

**External-network listing threat model (F4)**

When 1inch/0x/KyberSwap call our pool, the call shape is:
```
their_router → IExchange(pool).swap(tokenIn, tokenOut, amountIn, minOut, recipient)
```
- Caller is `their_router`, not user EOA.
- `recipient` may be Settler (0x) or PMM contract (1inch Fusion+).
- **Atomic batching**: their router can pre-pump another pool, hit ours, post-dump → classic sandwich, but cross-pool. Our pool's `_oracle` push (line 539) updates TWAP on every swap → an attacker can poison the BTR Pool TWAP within one tx, then exploit via a second pool that uses our TWAP. Verify which downstream uses Pool oracle (`PoolOracle.computeFastTWAP`).
- **JIT-LP / sandwich on our pool**: BTR Pool has no MEV-share / no commit-reveal; the TWAP-aware pricing in `Pricing.getAnchorPathQuote` is the inline defense. Confirm anchor-path is robust to single-block tick movement before listing.
- **Slip enforcement**: per-block atomicity ⇒ aggregator's `minOut` is fresh-quoted in the same block ⇒ slip = signaling only, not protection. Our pool must enforce its own *spot-vs-TWAP deviation* check on swap, not trust caller's minOut.

**Listing-cost summary (F4 continued)**: 1inch / 0x / KyberSwap each require:
- Permissionless `swap(tokenIn, tokenOut, amountIn, minOut, recipient)` signature.
- Deterministic gas (no unbounded loops on settler-callable path).
- No callbacks to caller after token-out (avoids settler reentrancy class).
- Permit2-compatible spending OR conventional approvals -our pool currently is approval-based ⇒ compatible with both Settler models, but 0x v2 Settler prefers a TransferFrom *from settler*, not from user → cost = either accept settler as `msg.sender` (lose user→pool reentrancy isolation) **or** integrate Permit2.

**ALM-as-LP MEV exposure** (atomic-router-on-our-pool while our adapter LP is in range):
- Vault LP earns fees on pool swaps. ✓
- But: hostile router can sandwich our adapter's `rebalance` tx (mempool) -push pool tick to range edge → adapter mints at bad mid → adapter `assetValue` self-prices → Vault `_healthy` may not catch it because adapter's `deviationBps` defense is **inside** adapter, not at Vault. Verify `adapter.frozen()` triggers before Vault credits NAV.
- Existing `_checkLive` (`Dex.sol:421`) handles in-flight intents. Add equivalent for in-mempool rebalance -keeper should use private mempool (Flashbots Protect equiv).

**Pools-page info disclosure (F5)**: ranges, TVL, last-rebalance ts → publicly reconstructable from chain anyway. UI surfacing only accelerates retail discovery, not adversary. **Acceptable to publish**. Mitigate strategic leakage by randomising rebalance ticks within tolerance (keeper-side).

## 2. Verdict: ALM `Dex.sol` → BTR DEX

**NEEDS-NEW-ADAPTER.**

Rationale:
1. `Dex.sol` enum + state assume external CL pool (tick range, slot0, V4 StateView). BTR Pool exposes none of these -it is a single-token spline-priced pool.
2. Single-token semantics map cleanly to a thinner adapter: `pull(amount)` → `Pool.deposit(asset, amount)` → store returned `lpAmt`. `withdraw(to, amount)` → `Pool.withdraw(asset, lpAmt, minOut)` → push.
3. No paired token0/token1 ⇒ no `liquidityToAmounts`, no rebalance routes, no `Swapper` dependency.
4. `assetValue(token)` becomes `(lpBalance × liquidityIndex / WAD) − conservative_haircut_estimate` + idle. **Pessimistic** estimate must include `haircutSuppressor`-aware floor.
5. Risk knobs different: no `deviationBps` (no external oracle pair); replace with `maxAcceptableHaircutBps` from `Pool` view. Keep `kill`, `frozen`, `_checkLive` shape.

Reusable from `Dex.sol`: kill machinery, holder share-mint pattern (`principalOf` / `totalPrincipal`), timelocked risk params, ACL wiring.

NOT reusable: `rebalance`, `_burnPosition`, `_mintPosition`, `_slot0Branched`, V4 selectors, Swapper integration (no token-pair swaps needed).

## 3. External listing security cost (1inch / 0x / KyberSwap)

| Requirement | Current state | Cost |
|---|---|---|
| Permissionless `swap()` entrypoint | `Pool.swap` exists; uncertain if it's open or only via `Router` w/ `isOfficialPool` gate | MED -must expose Pool-direct swap w/ same hardness as Router path |
| Per-call slippage enforcement (caller-supplied `minOut`) | Pool already supports `minAmountOut` in withdraw; need same in swap | LOW |
| TWAP poisoning resistance | `_oracle` pushed every swap → poisonable in atomic batch | HIGH -gate TWAP push on EMA-bounded delta, OR require ≥1 block between TWAP-eligible swaps |
| No outbound callbacks to caller | Pool pushes tokenOut via `_push` (Safe.safeTransfer + WETH wrap). Plain ERC-20 = no callback ✓ | NONE if all listed tokens are plain ERC-20; non-zero if any 777/hook token. Maintain a token allowlist. |
| Permit2 integration | None today | MED -required by 0x v2 Settler. Add Permit2-aware `depositWithPermit2`/`swapWithPermit2`. |
| Deterministic gas | `PoolDecay.applyDecay` is bounded; `Pricing.getAnchorPathQuote` spline is bounded. ✓ | LOW |
| MEV-share / private orderflow incentive | None | OPTIONAL -could capture ~10–30 bps back via 1inch MEV-share program (off-critical-path). |
| Audit & formal listing fee | n/a | EXTERNAL -each network has its own diligence. Budget. |

## 5. Risk-mitigated phased execution

1. **Phase A -frontend restructure (LOW risk)**: rename Earn → Vaults; add Pools page w/ two tabs. No contract changes. Approval-namespace audit pass.
2. **Phase B -`BtrDex.sol` adapter (greenfield)**: new contract; mirrors `Dex.sol` kill/freeze/holder primitives; thin deposit/withdraw to `Pool`. Full test coverage matching ALM's 263-test bar. Audit before Vault wiring.
3. **Phase C -Vault wiring**: ALM adds BtrDex adapter via `addAdapter`. Run on testnet w/ canary capital cap (`depositCap` ratcheted low). Triple-slip invariant test in CI: `dexHaircut + exitPip + adapterMaxSlip ≤ 10%`.
4. **Phase D -BTR Swap integration on Swap page**: BTR DEX as one liquidity source in BTR Swap (Li.Fi + BTR). Permit2 path here first (front-only) before listing externally.
5. **Phase E -TWAP hardening + Permit2 entrypoints on Pool**: required for external listing. Independent audit pass.
6. **Phase F -External listing**: 1inch / 0x / KyberSwap onboarding. Start with one (recommend 1inch -most mature MEV-share program). Monitor for 4 weeks before adding the next.

Hard gate between B and C: independent audit of new adapter.
Hard gate between E and F: re-audit of Pool with the new external entrypoints.
