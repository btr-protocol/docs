# 8. Cross-Chain (Forward-Looking)

> **Status: design exploration.** Nothing in this page is implemented in the current `alm/evm/` codebase. All claims are subject to change pending design, audit, and governance approval.

---

## 1. Today's Footprint

The contracts in `alm/evm/` are **single-chain**. Each chain receives an independent deployment of:

```text
AccessControl  → owner / keepers / treasury / swapper / factory pinned per chain
PriceProvider  → Chainlink feeds for that chain's token set
Swapper        → LiFi router + ERC-7683 settler (chain-local)
Factory        → vaultImpl + dexImpl (chain-local clones)
Vault[i]       → per-asset
Dex[j]         → per-pool
```

Cross-chain coordination is currently external -multisigs sign on each chain, keeper bots run per-chain, and asset bridging happens off-system before deposit.

## 2. Extension Scenarios

### 2.1. Intent-Native Cross-Chain Rebalance

**Premise**: rebalances on chain A occasionally need to source liquidity from chain B (e.g., when a stablecoin pair is concentrated on Arbitrum but a vault on Base is short).

**Hook**: `Swapper.openIntent` already speaks ERC-7683. ERC-7683 is **chain-agnostic by design** -`tokenIn` on chain A, `tokenOut` on chain B is a valid intent shape.

**Open questions**:

- Per-chain Swapper currently enforces `tokenIn != tokenOut` (`Swapper.sol:145, :180`). A cross-chain intent has same-symbol but different-chain -needs a `(token, chainId)` shape or a chain-routing wrapper.
- `outBaseline` watermark assumes `tokenOut` arrives at the local Swapper. Cross-chain settle requires an inbound destination contract to credit the original caller -likely a sibling Swapper instance on the destination chain.
- `activeOut[token]` serialization is per-chain; cross-chain intents need a global de-dup mechanism or accept per-leg races.

### 2.2. Shared Vault Share Across Chains

**Premise**: a user deposits `USDC` on chain A, the protocol mints share tokens on chain A, but the underlying allocation runs on whichever chain has best opportunity.

**Hooks**:

- `Vault` is single-asset -vault asset is bound at `init` and immutable. A cross-chain variant would be a wrapper that holds the canonical share and mints a transit / messenger share for the user.
- `AccessControl` is per-chain; cross-chain governance requires a message-passed mirror or a Hyperlane / LayerZero / Chainlink CCIP relay.

**Open questions**:

- Whose `lastReported` is canonical? -likely the deploying chain's Vault, with destination chains reporting via signed updates.
- Async-redeem ticket portability -does a `requestRedeem` on chain A settle from chain B's idle? Almost certainly no; the simpler model is per-chain liability.

### 2.3. Cross-Chain Adapter

**Premise**: a single Dex adapter could manage LP on a remote chain via a proxy.

This is the **most invasive** extension. The current `Dex.sol` directly `staticcall`s pool / oracle. A cross-chain adapter would need:

- An async `assetValue` that returns last-known-mark and a freshness window.
- Async `pull` / `withdraw` semantics that mint shares against committed-but-not-yet-bridged value (and a corresponding gating treatment akin to the in-flight intent freeze, see [Risk Model §5](/docs/7.4-Risk-Model)).
- Transport layer with replay protection.

**Likely model**: defer this. Use 2.1 (cross-chain intent rebalances on each chain's local adapter) instead.

## 3. Forward-Looking Constraints

If and when cross-chain is implemented, these invariants from the current codebase carry over:

- **Worst-of marks** (`Dex.assetValue`, `Dex.sol:356`) -extend to `min(localPool, localOracle, lastBridgedMark, peerVaultReportedMark)` or any superset.
- **Timelock authority** (`FREEZE_TIMELOCK`, `ROTATION_TIMELOCK`, `FACTORY_TIMELOCK`) -must be honoured on every chain in the cluster. Cross-chain governance MUST NOT bypass them.
- **Sequencer guard** (`PriceProvider._checkSequencer`) -applies per-chain and must remain blocking; cross-chain freezes when **any** participant chain's sequencer is down or in grace.
- **Keeper kill cap** (`AccessControl.KEEPER_KILL_DAILY_CAP`) -semantics need adjustment: per-cluster (cross-chain) vs per-chain. Likely per-chain to limit blast radius of a compromised keeper key on a single chain.

## 4. What Is NOT Planned

To avoid surprise:

- **Cross-chain mint of vault shares without underlying delivery confirmation.** Always asset-backed.
- **Bridging risk absorbed by the vault.** Bridging cost / risk surfaces to the user via the chosen route's `minOut`.
- **Native messenger** (BTR-bespoke transport). The protocol will use established transports (LayerZero, Hyperlane, CCIP, Across, native Hyperlane / OP-stack messaging) rather than build one.

## 5. Open Items Requiring Governance Approval

- Selection of cross-chain transport(s) per deployment chain.
- Per-chain `Swapper` upgrade to support `(token, chainId)` intent shape.
- Whether cross-chain rebalances bypass `minIntervalSecs` (likely no).
- Treasury policy for cross-chain rescue funds (per-chain treasury vs canonical mainnet treasury).
