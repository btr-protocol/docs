# 5. Risk Model

> Identifying, bounding, and mitigating the loss vectors faced by BTR Vault depositors and operators.

---

## 1. Impermanent Loss (IL)

For a CLMM position with range `[a, b]` and current sqrt-price `√P`:

- **In-range**: position value tracks a CFMM-shape between `(a, b)`. Compared to a buy-and-hold of the deposit token, IL accrues on directional moves; it can be partially or fully offset by collected fees.
- **Out-of-range**: position is fully converted to the side toward which price exited. IL crystallizes -the position becomes a single-sided holding at the boundary tick.

BTR Vaults do not eliminate IL. They quantify it via the worst-of NAV (`Dex.assetValue`), bound it via range selection, and surface it to the user transparently through share price.

## 2. LVR (Loss Versus Rebalancing)

LVR is the structural shortfall of a passive AMM LP relative to a perfectly-rebalanced book that quotes the same prices. The canonical form (Milionis-Moallemi-Roughgarden 2022, [arXiv:2208.06046](https://arxiv.org/abs/2208.06046); see also the CowSwap [LVR exposition](https://docs.cow.fi/cow-amm/concepts/the-problem-of-lvr)):

**LVR rate** (per unit time):

$$ \ell(P) = \tfrac{1}{2} \cdot \sigma^2 \cdot P^2 \cdot x'(P) $$

where $x'(P)$ is the marginal liquidity (∂reserve/∂price). For a constant-product CFMM at value $V$, this simplifies to $\ell \approx \sigma^2 \cdot V / 8$. BTR vault LVR depends on the underlying venue's curve and the active range — not a single closed form across all adapters.

LVR is the dominant cost driver for an unmanaged LP position. BTR Vaults attack it on three axes:

1. **Tighter ranges** -concentrate fees relative to LVR per range-width unit.
2. **Adaptive rebalances** -re-centre on volatility regime change, gated by `minIntervalSecs` and post-mint slip envelope.
3. **Worst-of marks** -the price at which a withdrawer can extract value is bounded below by `min(pool, oracle)`. An informed exit through a stale-pool window cannot capture more than `oracle` deviation.

LVR is **not** zeroed. Operator skill in range selection and rebalance cadence determines the realized LVR-to-fee ratio.

## 3. Oracle Deviation Freeze

`Dex.frozen()` (`Dex.sol:frozen`, ≈ L493) returns `true` if **any** of:

- Pool `slot0()` reverts or returns `sqrtP == 0`.
- Either token's Chainlink feed is unset, stale, sequencer-down, or returns 0.
- `|pPool − pOracle| / max > deviationBps`.

Freeze gates: `rebalance` (twice -pre and post swap), `pull`, `withdraw`. `Vault._healthy()` ORs each adapter's freeze flag, so any frozen adapter pushes the vault into the async-redeem path.

`deviationBps` and `heartbeatDriftBps` are owner-only, **timelocked** (`FREEZE_TIMELOCK = 1 day`, Pass-29 P29-1 hoist to `Constants.sol:FREEZE_TIMELOCK`; `Dex.sol:queueRisk` ≈ L288, `:executeRisk` ≈ L301). Loosening these knobs cannot ship instantly.

## 4. Sequencer Outage (L2)

`PriceProvider._checkSequencer()` (`PriceProvider.sol:_checkSequencer`, ≈ L120) consults the Chainlink L2 sequencer-up feed:

- If `ans != 0` (sequencer down) OR `block.timestamp - startedAt < SEQ_GRACE (1 hour)`, all `getPrice` calls revert with `SeqDown`.
- This cascades: every adapter `frozen()` returns true, every vault `_healthy()` returns false, sync deposit/redeem disabled, async tickets queue.

When the sequencer returns and grace expires, `getPrice` resumes and `Vault.syncAdapter` (permissionless) re-bumps `lastSync` for each adapter, restoring health automatically.

## 5. In-Flight Intent Gating

`Dex._hasOpenIntent()` (`Dex.sol:_hasOpenIntent`, ≈ L525) staticcalls `Swapper.activeOut(token)` for both adapter tokens. Any open intent on either side ⇒ `healthSnapshot` reports frozen.

**Why**: when an ERC-7683 intent is open, value is escrowed at the settler -invisible to `assetValue`. Allowing share-mint or share-burn during this window would mis-price NAV. The sync deposit/redeem path closes; the async ticket path remains open.

Fail-closed semantics: a missing swapper, ABI drift, or staticcall failure all return `true` (see `_hasOpenIntent` body, ≈ L525–L533).

## 6. Kill Switch

Per-adapter `kill()` (`Dex.sol:kill`, ≈ L244):

- **Owner** kill -instant; revertible only via `queueUnkill` + `executeUnkill` (FREEZE_TIMELOCK = 1 day delay) so a compromised owner cannot replay-toggle.
- **Keeper** kill -rate-limited per adapter (`KILL_RATE_LIMIT = 1 hour`) and charged against `AccessControl.chargeKeeperKill` daily cluster cap (`KEEPER_KILL_DAILY_CAP = 2`).

Once killed, `assetValue` reverts (`Dex.sol:assetValue` ≈ L454), so vault `_healthy()` returns false and share-mint freezes. Existing async tickets remain claimable from idle balances.

## 7. Adapter Deregistration

Owner-only `AccessControl.deregisterAdapter(a)` (`AccessControl.sol:deregisterAdapter`, ≈ L100) flips `isAdapter[a]` off. Vaults using that adapter become unhealthy on the next `_healthy()` check (`Vault.sol:_healthy`, ≈ L633). The owner subsequently calls `Vault.removeAdapter(a)` to drain accounting state -permitted because `isAdapter[a] == false` overrides the `lastReported != 0` busy check (`Vault.sol:removeAdapter`, ≈ L491).

This is the **operator escape hatch** for a compromised or buggy adapter clone -without requiring a Vault redeploy.

## 8. Risk Knobs Summary

| Knob | Default | Bounds | Authority | Timelock |
|------|---------|--------|-----------|----------|
| `deviationBps` | 100 (1%) | [10, 5000] | Owner | 1 day |
| `heartbeatDriftBps` | 50 (0.5%) | [10, 5000] | Owner | 1 day |
| `maxSlipBps` | 100 (1%) | [10, 5000] | Owner ↑/↓ · Keeper ↓ only | none |
| `minIntervalSecs` | 1 hour | ≤ 30 days | Owner ↑/↓ · Keeper +1d/call only | none |
| `exitPip` | -| [50, 3_000], floor 1.2× max(dev+hb)·10 (Pass-21 P21-1) — i.e. factor of 12 = 10 (bps→pip conversion) × 1.2 safety margin | Owner | none (floor enforced) |
| `mgmtPip` | -| ≤ 50_000 (50%/yr) | Owner | none |
| `perfPip` | -| ≤ 20_000 (20%) | Owner | none |
| `depositCap` | 0 (∞) | uint128 | Owner ↑/↓ · Keeper ↓ ≥ totalAssets | none |
| `maxStaleSecs` | 1 day | (0, 30 days] | Owner | none |

## 9. Worth-Documenting Invariants

- An owner-deregistered adapter cannot block `removeAdapter` -drained or not (`Vault.sol:removeAdapter`, ≈ L491).
- Keeper kill propagates to the cluster daily cap; 1 keeper × 8 adapters × 1/h cannot freeze a vault (`AccessControl.sol:KEEPER_KILL_DAILY_CAP` L20, `:chargeKeeperKill` ≈ L85).
- Oracle and pool both required healthy for `assetValue` to return; either failure → revert (see `Dex.sol:assetValue` body, ≈ L454–L477).
- Post-rebalance slip envelope only applies to all-atomic route sets; mixed sets defer to per-route `minOut` + Swapper enforcement (`Dex.sol:rebalance` body, ≈ L440–L444).
- `frozen()` is checked twice per rebalance; an atomic route that moves the pool tick into a deviating zone aborts before LP mint (`Dex.sol:rebalance`, ≈ L406 pre-swap + ≈ L435 post-swap).
- `ADAPTER_SLIP_HEADROOM_BPS ≥ Vault.MAX_EXIT_PIP / 10` -currently `300 ≥ 3_000/10 = 300` (exact-fit). Raising `MAX_EXIT_PIP` without bumping `ADAPTER_SLIP_HEADROOM_BPS` in lock-step would soft-DoS adapter `executeSlip` via `Vault._assertJointSlip` (Pass-21 P21-1, Pass-23 P23-6).
- Owner-permitted `(deviationBps + heartbeatDriftBps) × 12 ≤ MAX_EXIT_PIP = 3_000` i.e. dev+hb sum ≤ 250 bps; exceeding it wedges `setFees` since `_exitFloor` rises above `MAX_EXIT_PIP` (Pass-23 P23-5).
