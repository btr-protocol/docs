# Voting

> Off-chain snapshot voting with on-chain voting power calculation and signature aggregation

---

## 0. TL;DR (user-facing)

BTR is governed through a layered model:

1. **Admin & operational roles** today: a multisig + timelock controller administers upgrades, parameter changes, and emergency pauses. Role assignments are bounded by `ROTATION_TIMELOCK`, `FACTORY_TIMELOCK`, and friends (see [Roles and Timelock](../security/roles-and-timelock.md)).
2. **On-chain voting** later (Phase 2): once veBTR launches, BTR holders cast votes on proposals affecting fees, emissions, treasury spend, and protocol-level parameters.

> **Status**: Phase-1 governance is **advisory + multisig-admin**. Full on-chain voting is Phase-2.

**Why it matters to users**
- **Predictable upgrades**: every protocol change passes through timelock with a public delay window.
- **Future voice**: lock BTR → veBTR → vote (Phase 2).
- **Audit trail**: all role rotations and parameter changes recorded on-chain.

---

## 1. Overview

The protocol implements a **snapshot-based governance voting system** that combines off-chain convenience with on-chain security. Voting power is calculated deterministically from on-chain state (BTR holdings, LP stakes, Vault shares once sVault ships) at a specific block height, then votes are cast off-chain and aggregated via signed messages.

### 1.1. Governance Scope

Governance authority spans **all protocol surfaces**:

| Domain | Examples |
|--------|----------|
| **Vault params** | `mgmtPip`, `perfPip`, `exitPip`, adapter registry (`isAdapter[]`), freeze flags, HWM policy |
| **DEX params** | Pool fees, profiles, matching ratio, claim-power floor |
| **Treasury policy** | Spending caps, POL allocation, fee routing (Ops vs POL split) |
| **Emission routing** | Per-recipient % (sLP / sBTR / sVault / treasury) within bounds |
| **Bridge config** | Peer registry (`setPeer`), bridgeable token whitelist, LZ DVN/executor set |
| **AccessControl rotations** | Treasury / Swapper / Factory / Owner - all timelocked, cancellable |

In short: **one governance, one authority, all protocol surfaces** (no per-surface fork of voting).

**Key characteristics:**
- **Block-height snapshots**: Voting power frozen at a specific past block (prevents flash-loan voting)
- **Off-chain voting**: Uses Snapshot or similar platforms for user-friendly proposal/voting interfaces
- **Signature-based aggregation**: Votes gathered as signed messages, no transaction fees during voting
- **Quadratic damping**: Large holders' voting power is suppressed to prevent plutocracy
- **Multi-chain aggregation**: Voting power aggregated across all chains where user holds sBTR or staked LP, or sVault (forward-looking)

## 2. Voting Power Formula

### 2.1. Components

A user's voting power consists of three components:

**1. BTR Governance Token (G)**
- Amount of sBTR (staked BTR) held at snapshot block
- Price-weighted: `G = govUnits × govPrice / 1e18`

**2. LP Positions (L)**
- Sum of all staked LP tokens across all pools
- Price-weighted: `L = Σ(lpUnits × lpPrice / 1e18)`
- Aggregated cross-chain (Ethereum, Arbitrum, Optimism, etc.)

**3. Boost Mechanism**
- Boosted amount: `boosted = min(L, boostCap × G / 10000)`
- Incentivizes BTR holders to also provide liquidity

- Vault Shares (V, forward-looking): sVault component to be added when Vault stake pool ships.

### 2.2. Linear Voting Power

Base formula (before damping):

```
`V_linear` = (`L` × `lpBaseMultiplier` / 10000)
           + (`boosted` × `lpBoostedMultiplier` / 10000)
           + (`G` × `govMultiplier` / 10000)
```

**Default multipliers**:
- `lpBaseMultiplier` is 5000 (0.5x: base LP contribution)
- `lpBoostedMultiplier` is 5000 (0.5x: additional boost for paired BTR + LP)
- `govMultiplier` is 10000 (1x: pure BTR governance weight)
- `boostCap` is 50000 (5x: maximum LP boost relative to BTR holdings)

### 2.3. Quadratic Damping

Raw voting power is then suppressed using a **quadratic damping function** to reduce influence of ultra-large holders:

```
`V_effective` = `V_linear` × (1 - `scale` × `V_linear`^`exponent`)
```

**Damping parameters**:
- `scale` is approximately 0.00548 (5.48×10⁻³)
- `exponent` is approximately 0.265 (concave curve, 0 < `exponent` < 1)

**What damping prevents**: Voting power concentration at extreme scales. Without damping, a single whale with 10× more capital than average would have 10× more influence. With quadratic damping, their influence grows sublinearly, preventing plutocratic capture.

**Effect**: Creates a concave suppression curve where:
- Small holders (< $100k): Minimal suppression (~1-2%)
- Medium holders ($1-10M): Moderate suppression (~10-30%)
- Large holders (> $100M): Severe suppression (approaches 50%+)

**Rationale**: Prevents governance capture by whale accumulation while still rewarding good-faith large positions.

### 2.4. Example Calculation

**User profile at snapshot block:**
- sBTR: 10,000 units (10,000 staked BTR)
- BTR price is $50
- LP positions: 500 units across pools
- LP weighted price is $100,000/500 = $200/unit

**Calculations:**

```
G = 10,000 × $50 / 1e18 = $500,000
L = 500 × $200 / 1e18 = $100,000
boostCap × G = 50000 × $500,000 / 10000 = $2,500,000
boosted = min($100,000, $2,500,000) = $100,000

V_linear = ($100,000 × 5000/10000)
         + ($100,000 × 5000/10000)
         + ($500,000 × 10000/10000)
         = $50,000 + $50,000 + $500,000
         = $600,000

V_effective = $600,000 × (1 - 0.00548 × $600,000^0.265)
            = $600,000 × (1 - 0.00548 × 18.97)
            = $600,000 × (1 - 0.104)
            = $600,000 × 0.896
            ≈ $537,600
```

## 3. Snapshot Process

### 3.1. Snapshot Creation

1. **Proposal submission** on Snapshot platform (or equivalent governance forum)
2. **Governance multisig** approves proposal format and voting window
3. **Block height selection**: Snapshot block = current block - 1 (or specified safe past block)
4. **Voting period**: Typically 3-7 days

### 3.2. Voting Power Snapshot

At the designated block:

1. **Query on-chain state**:
   - sBTR balances (Staking module)
   - LP token balances across all pools (one per pool, one per chain)
   - BTR and LP token prices (Oracle or price feed)
   - Vault share balances (sVault, forward-looking).

2. **Calculate voting power off-chain**:
   - Aggregate stakes across chains
   - Apply formula (linear + damping)
   - Publish snapshot report

3. **Publish voting power list**:
   - Public spreadsheet or Snapshot integration
   - Allows users to verify their voting power
   - Typically 6-24 hours before voting opens

### 3.3. Voting Phase

1. **Users vote on Snapshot**:
   - Select proposal option (For, Against, Abstain)
   - Sign message with their wallet (no gas cost)
   - Voting power automatically determined from snapshot

2. **Vote aggregation**:
   - Snapshot protocol collects signed votes
   - Calculates quorum (% turnout) and result (% in favor)

3. **Voting rules**:
   - **Quorum**: ≥ 10% of voting power participates (adjustable by governance)
   - **Threshold**: ≥ 50% approval to pass (simple majority for most votes)
   - **Supermajority**: ≥ 67% for critical changes (protocol upgrades, supply cap changes)

## 4. Post-Vote Execution

### 4.1. Governance Actions

**Administrative (multisig or timelock execution)**:
- Emission schedule parameter changes (within bounds)
- Emission routing adjustments (sLP % vs sBTR %)
- Treasury spending approvals
- Oracle configuration updates

**Smart contract execution**:
- Flash loan fee adjustments
- Risk parameter updates per asset
- Module deployments or upgrades
- Emergency pause activation (if needed)

### 4.2. Timelock & Safety

Most governance actions are subject to a **7-day timelock** before execution:

1. **Vote passes** on Snapshot
2. **Multisig (Council) initiates** timelock
3. **7-day delay period** (community can exit if disagree)
4. **Multisig executes** action on-chain

**Exceptions** (no timelock):
- Governance-delegated parameters (e.g., per-asset vega/lambda fees)
- Emergency pause (pauses all pool operations immediately)

## 5. Delegation & Proxy Voting

### 5.1. Vote Delegation

Users can delegate voting power via:

1. **Direct delegation** (on sBTR via Staking module):
   ```solidity
   staking.delegateVoting(delegate_address)
   ```
   - Delegates full voting power to another address
   - Snapshot recognizes delegation at voting block

2. **LP delegation**:
   - Delegate LP staking voting rights separately from BTR
   - Enables delegated LP incentive optimization

3. **Revocation**:
   - Call `delegateVoting(msg.sender)` to self-delegate and regain voting power
   - Effective immediately (voting power transfers at next snapshot)

### 5.2. Self-Delegation Requirement

Users holding sBTR must **explicitly self-delegate** to vote:

```solidity
staking.delegateVoting(msg.sender)  // Enable your own voting power
```

**Rationale**: Prevents accidental vote power delegation, ensures intentional participation.

## 6. Governance Participation

### 6.1. Proposal Submission

**Eligible proposers**:
- Any address with ≥ 0.1% of voting power (dynamic threshold)
- Or DAO multisig (Council) directly

**Proposal types**:
1. **Funding requests**: Treasury spending > 200k BTR
2. **Parameter changes**: Emission split, risk parameters, fee structures
3. **Protocol upgrades**: Smart contract deployments, module additions
4. **Emergency actions**: Pause pools, salvage stuck tokens

### 6.2. Discussion & Feedback

1. **Forum stage** (48-72 hours):
   - Proposal discussion on governance forum
   - Community feedback and questions
   - Proposals can be refined or withdrawn

2. **Voting stage** (3-7 days):
   - Formal Snapshot vote
   - Immutable once posted

### 6.3. Governance Incentives

Users participating in governance receive:
- **Voting rewards** (optional, from emissions treasury): 0.1-1% of proposal contested budget
- **sBTR staking APR**: 5% of total emissions (3.25M BTR over 10 years) distributed to sBTR holders pro-rata
- **Governance points** (for leaderboard / community recognition)

## 7. Multi-Chain Governance

### 7.1. Cross-Chain Voting Power

Voting power is **aggregated across all supported chains**:

```
V_total = V_ethereum + V_arbitrum + V_optimism + ...

for each chain:
  V_chain = V_linear_chain × damping_chain
```

**Supported chains** (voting power eligible):
- Ethereum Mainnet (primary)
- Arbitrum One
- Optimism
- Base
- (others as pool deployment expands)

### 7.2. Snapshot Bridge Integration

Bridge contracts relay user staking snapshots to primary chain for voting power calculation:

1. **Per-chain snapshot**: Queries sBTR and LP balances at block N
2. **Bridge commitment**: Merkle root of chain-specific balances → Ethereum
3. **Ethereum aggregation**: Combines all chain snapshots into unified voting power
4. **Snapshot oracle**: Makes aggregated power available to Snapshot platform

**Timing**:
- Snapshot initiated ~12 hours before voting opens
- Bridge proofs finalized with 1-hour safety buffer (allows for 12-block confirmations per chain)

## 10. Governance Safety

### 10.1. Security Measures

1. **Snapshot finality**: Voting power frozen at past block (prevents flash loans, oracle manipulation)
2. **Timelock buffer**: 7 days allows community exit if proposal turns adversarial
3. **Quorum requirement**: ≥ 10% participation prevents apathy governance
4. **Multisig oversight**: Council can veto obviously malicious proposals (during V1 transition)
5. **Parameter bounds**: Governance changes capped (e.g., emissions can't increase by >10% per vote)

### 10.2. Emergency Procedures

**Pause mechanism** (no timelock):
- DAO or owner can immediately pause all pools if:
  - Critical vulnerability discovered
  - Extreme market conditions (e.g., oracle failure)
  - Governance attack underway
- Pause can be lifted by governance vote (7-day timelock)

**Stuck-token salvage** (Treasury / Bridge):
- `salvage(token, to, amount)` -owner-only, single-step sweep of stranded ERC-20 / native
- Timelock comes from the multisig owner, not the on-chain function
